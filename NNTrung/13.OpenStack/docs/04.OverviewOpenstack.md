# Tìm hiểu về Openstack
## I. Tổng quan Openstack
### 1. Khái niệm
- **OpenStack** là một open source cloud computing plaform cung cấp **Infrastructure as a Service (IaaS)**, cho phép quản lý và điều phối các tài nguyên điện toán đám mây như máy ảo, lưu trữ và mạng trong trung tâm dữ liệu.
- OpenStack giúp bạn xây dựng "AWS của riêng mình" với toàn quyền kiểm soát, không phụ thuộc vào nhà cung cấp.

### 2. Kiến trúc OpenStack
OpenStack được thiết kế theo kiến trúc kiến trúc microservices, trong đó mỗi service đảm nhận một chức năng riêng như compute (Nova), networking (Neutron), storage (Cinder/Swift). Các service giao tiếp với nhau thông qua message queue như RabbitMQ và được xác thực qua Keystone.

Điểm mạnh là tính modular và khả năng tích hợp nhiều backend khác nhau (KVM, Ceph, OVS), giúp nó linh hoạt hơn so với cloud public nhưng cũng phức tạp hơn khi vận hành – đặc biệt ở networking.

#### 2.1 Lớp quản lý
- **Keystone**: Trung tâm xác thực (authentication & authorization) cho tất cả dịch vụ khác. Keystone là service xác thực và phân quyền trung tâm, quản lý user, project và role, đồng thời cấp token để các service khác verify request.
  - Keystone là cổng bảo vệ của toàn bộ OpenStack
- **Ceilometer**: Giám sát, thu thập metrics (telemetry) từ toàn hệ thống
- **Horizon**: Giao diện web (dashboard) để người dùng/admin tương tác.

- OpenStack không phải là 1 phần mềm duy nhất mà được xây dựng từ nhiều service nhỏ độc lập như:
  - NOVA (compute), Neutron (network), Cinder (storage), Glance (image),...
  - Mỗi Service làm một nhiệm vụ riêng, giống microservices.
- Tất cả dịch vụ, request đều phải xác thực qua Keystone (Identity Service)
- Các dịch vụ riêng lẻ giao tiếp với nhau thông qua các API công khai, trừ những trường hợp cần quyền quản trị đặc biệt.
- Bên trong, mỗi dịch vụ OpenStack được cấu thành từ nhiều tiến trình (process). Mỗi dịch vụ đều có ít nhất một tiến trình API, nhiệm vụ là nhận các yêu cầu API, xử lý sơ bộ và chuyển tiếp chúng đến các thành phần khác trong dịch vụ. Ngoại trừ dịch vụ Identity, phần công việc xử lý chính sẽ được thực hiện bởi các tiến trình riêng biệt khác.
- Để các tiến trình trong cùng một dịch vụ giao tiếp với nhau, hệ thống sử dụng một message broker theo chuẩn AMQP (ví dụ như RabbitMQ).
- Trạng thái của các dịch vụ được lưu trữ trong cơ sở dữ liệu. Khi triển khai OpenStack, bạn có thể lựa chọn nhiều giải pháp message broker và database khác nhau như RabbitMQ, MySQL, MariaDB hoặc SQLite.
- Người dùng có thể truy cập OpenStack thông qua giao diện web được cung cấp bởi dashboard Horizon, thông qua công cụ dòng lệnh (CLI), hoặc bằng cách gửi trực tiếp các yêu cầu API (ví dụ dùng curl hoặc plugin trình duyệt). Đối với ứng dụng, cũng có sẵn nhiều bộ SDK hỗ trợ.
- Cuối cùng, tất cả các phương thức truy cập này đều thực chất là gửi các yêu cầu REST API đến các dịch vụ OpenStack.
- Kiến trúc phổ biến nhất cho OpenStack Cloud (không phải duy nhất, phát triển tùy thuộc vào mục đích doanh nghiệp có thể mở rộng thêm)

**NOVA (Compute)**: Nova là service chịu trách nhiệm quản lý toàn bộ vòng đời của máy ảo (VM). Dịch vụ này cho phép người dùng tạo, cấu hình, khởi động, dừng và xóa các máy ảo trên hạ tầng vật lý. Nova đóng vai trò như lớp trừu tượng hóa giữa phần cứng vật lý và máy ảo, quản lý việc phân bổ tài nguyên CPU, RAM, disk cho các VM. Nó không trực tiếp chạy VM mà orchestrate hypervisor như KVM thông qua libvirt.
  - Thành phần bên trong: 
    - **nova-api**: là cổng giao tiếp chính, nhận tất cả yêu cầu từ người dùng thông qua Horizon (web interface) hoặc CLI commands. Component này xác thực người dùng với Keystone, sau đó điều phối các yêu cầu đến các thành phần khác trong Nova.
    - **nova-scheduler**: đóng vai trò như bộ não ra quyết định, chọn máy chủ vật lý phù hợp nhất để đặt máy ảo mới dựa trên các tiêu chí như tài nguyên sẵn có (RAM, CPU, disk), zone khả dụng, và các policy được cấu hình.
    - **nova-compute**: là worker chạy trên từng compute node vật lý, trực tiếp giao tiếp với hypervisor (KVM, Xen, VMware) để tạo và quản lý máy ảo. Component này thực hiện các lệnh start, stop, reboot, delete VM và cập nhật trạng thái VM.
    - **nova-conductor**: đóng vai trò trung gian bảo mật giữa nova-compute và database, giúp nova-compute không cần truy cập trực tiếp vào database, tăng cường bảo mật và khả năng mở rộng hệ thống.
    - **nova-console/nova-consoleauth**:cung cấp tính năng remote console (VNC/SPICE) cho phép người dùng xem và tương tác với màn hình máy ảo từ xa.
    - **nova-cert**: Quản lý SSL certificate cho console access.
    - **Guest agent**: Chạy bên trong VM, giao tiếp với nova-compute để inject thông tin (hostname, SSH keys, metadata)
- **NOVA**: chỉ là bộ điều phối, còn việc chạy VM là do hypervisor(kiến trúc loose coupling).
- AWS tương đương: EC2

**Neutron(Networking)**: NNeutron cung cấp dịch vụ networking-as-a-service, cho phép người dùng tạo và quản lý mạng ảo cho các máy ảo. Dịch vụ này cung cấp các tính năng như tạo network, subnet, router, cấp phát IP, cấu hình firewall, load balancer, và VPN. Neutron cho phép tạo các mạng isolated giữa các tenant khác nhau.
  - Thành phần:
    - **neutron-server**: API endpoint nhận tất cả yêu cầu về networking như tạo network, subnet, router, port, security group. Component này xác thực với Keystone và lưu metadata vào database.
    - **neutron-l3-agent**: cung cấp chức năng routing layer 3, tạo network namespaces cho routers, cấu hình NAT và floating IPs để máy ảo có thể truy cập Internet hoặc được truy cập từ bên ngoài.
    - **neutron-dhcp-agent**: chạy dịch vụ DHCP (thường dùng dnsmasq) để cấp phát địa chỉ IP động cho các máy ảo, cung cấp thông tin về gateway, DNS servers.
    - **Neutron L2 agent(OVS/Linux Bridge)**: (Open vSwitch hoặc Linux Bridge) chạy trên mỗi compute node và network node, chịu trách nhiệm tạo virtual switches, quản lý VLAN/VXLAN/GRE tunnels, và kết nối các tap devices của VM vào virtual switch.
    - **Neutron-metada-agent**: cung cấp metadata service cho máy ảo, cho phép VM lấy thông tin như hostname, SSH public keys, user data thông qua địa chỉ IP đặc biệt 169.254.169.254.
    - **Neutron 3rd party plugins** cho phép tích hợp với các SDN controllers như OpenDaylight, NSX, OVN để thay thế cơ chế networking mặc định.
- Neutron là phần phức tạp nhất vì nó phải ảo hóa toàn bộ network stack từ switching đến routing.
- AWS tương đương: VPC + ELB

**OpenStack Identity (Keystone)**: Keystone là dịch vụ identity trung tâm của OpenStack, cung cấp chức năng xác thực (authentication), phân quyền (authorization) và service discovery. Tất cả các dịch vụ khác trong OpenStack đều phải thông qua Keystone để xác thực người dùng trước khi thực hiện bất kỳ operation nào. Keystone quản lý users, projects (tenants), domains, roles và service catalog.
- Thành phần: 
  - **keystone-all**: là process chạy tất cả các services của Keystone bao gồm Identity API (v2/v3), Admin API, token persistence, và catalog service. Trong các deployment lớn, các service này có thể tách riêng.
  - Keystone hỗ trợ nhiều backend lưu trữ: SQL database (MySQL/PostgreSQL) để lưu users, projects, roles, services, endpoints; LDAP để tích hợp với enterprise directories như Active Directory; và các token backends như SQL, Memcached/Redis, hoặc Fernet tokens (stateless).
  - Chức năng chính bao gồm: xác thực người dùng qua username/password hoặc token, cấp và validate tokens, quản lý service catalog (danh sách endpoints của tất cả services), và thực hiện role-based access control (RBAC).

**OpenStack Image Service(Glance)**: Glance là dịch vụ quản lý disk images của máy ảo, cho phép người dùng upload, download, và quản lý các bootable images (như Ubuntu, CentOS, Windows). Images có thể ở nhiều định dạng khác nhau (raw, qcow2, vmdk, iso) và được lưu trữ trên nhiều backend storage khác nhau. Glance cung cấp metadata về images như OS type, architecture, minimum disk/RAM requirements.
  - **glance-api**: Nhận lệnh tải lên/ xóa Image
  - **glance-registry**: Lưu thông tin mô tả Image(metadata) và database
  - **glance-store**: Nơi thực sự lưu Image (có thể lưu trên file system, Swift hoặc Ceph)
- AWS tương đương: AMI

**Cinder(OpenStack Block Storage)**: Cung cấp các khối lưu trữ (Volume) gắn thêm vào máy ảo (giống như cắm USB hoặc ổ cứng ngoài), hỗ trợ snapshot, backup và replication.
  - **cinder-api**: là API endpoint cho volume operations như create, delete, attach, detach, extend volumes. Component này xác thực với Keystone và đẩy tasks vào queue.
  - **cinder-scheduler**: chọn volume backend (storage pool) phù hợp nhất dựa trên capacity, capabilities (thin provisioning, encryption, replication), và availability zone.
  - **cinder-volume**: chạy trên storage node hoặc controller node, giao tiếp trực tiếp với storage backends để tạo, xóa, extend volumes. Component này quản lý volume lifecycle và thực hiện các operations cụ thể trên storage backend.
  - **cinder-backup**: cung cấp chức năng backup volumes đến backup backends như Swift, NFS, hoặc S3, hỗ trợ cả full và incremental backups.
  - **Volume providers** là storage backend drivers (LVM, Ceph/RBD, NFS, SAN drivers cho NetApp, EMC, HP...) thực hiện các operations create/delete/extend/clone trên storage systems.

- Cinder tách storage khỏi compute giúp VM stateless hơn và dễ scale
- AWS tương đương: EBS

**Swift(OpenStack Object Storage)**: Swift là dịch vụ distributed object storage, cung cấp khả năng lưu trữ phi cấu trúc (unstructured data) với khả năng mở rộng vô hạn. Swift lưu trữ dữ liệu dưới dạng objects trong containers, phù hợp cho việc lưu trữ files, backups, images, videos. Dịch vụ này cung cấp high availability và durability thông qua data replication across multiple nodes.
  - **swift-proxy-server**: là API endpoint duy nhất cho clients, nhận requests PUT, GET, DELETE objects/containers/accounts. Component này authenticate với Keystone, route requests đến đúng storage nodes, và handle replication và consistency.
  - **swift-account-server** chạy trên mỗi storage node, quản lý account databases (SQLite), lưu thông tin về containers, metadata, và quotas.
  - **swift-container-server** chạy trên mỗi storage node, quản lý container databases, lưu object listings, metadata, và sync points.
  - **swift-object-server** chạy trên mỗi storage node, quản lý object files trên disk, lưu objects dưới dạng files với metadata, và handle replication và deletion.
  - Swift sử dụng consistent hashing (hash ring) để phân phối objects across storage nodes, với default 3 replicas cho mỗi object để đảm bảo durability.

**Horizon(Dashboard)**: Cung cấp giao diện đồ họa (Web UI) giúp người dùng quản lý toàn bộ hệ thống (tạo máy ảo, mạng, volume...) mà không cần gõ lệnh.
- Horizon chỉ là frontend → mọi thứ thực sự vẫn đi qua API
- Horizon sử dụng Django sessions hoặc cookies để lưu user tokens và session information, hỗ trợ AJAX cho real-time updates (như refresh VM status, live console), và tích hợp noVNC/SPICE cho VM console access.

#### 2.3 Các dịch vụ Mở rộng quan trọng 
**Heat**: Heat cung cấp dịch vụ orchestration, cho phép tự động hóa việc triển khai hạ tầng phức tạp thông qua templates (HOT - Heat Orchestration Template hoặc CloudFormation compatible). Heat cho phép define infrastructure as code, tự động tạo và cấu hình multiple resources (VMs, networks, volumes) theo đúng thứ tự và dependencies.
- Thành phần:
  - **heat-api**: là API endpoint nhận templates, validate templates, và tạo stacks.
  - **heat-api-cfn**: cung cấp AWS CloudFormation compatible API, cho phép migrate applications từ AWS sang OpenStack.
  - **heat-engine** process templates để tạo resources, thực hiện dependency resolution (tạo resources theo đúng thứ tự dependencies), và rollback nếu có lỗi xảy ra.

**Ceilometer/Gnocchi/Aodh**: Đây là hệ thống telemetry để thu thập metric, lưu trữ và trigger alert.
- Ceilometer: collect data
- Cnocchi: lưu metric
- Aodh: alert

**Ceilometer** cung cấp dịch vụ đo đạc từ xa cho OpenStack, thu thập metrics về resource usage (CPU, memory, disk, network) từ các services và machines. Dữ liệu này được sử dụng cho monitoring, billing, alerting, và auto-scaling. Ceilometer thu thập data thông qua polling agents và notification listeners, sau đó lưu trữ vào database hoặc time-series storage.

- **ceilometer-agent-compute** chạy trên mỗi compute node, thu thập metrics từ máy ảo như CPU time, memory usage, disk I/O, network traffic thông qua libvirt API và cgroups.
- **ceilometer-agent-central** chạy trên controller node, thu thập metrics từ các services như API call counts, resource usage (images, volumes, networks) thông qua polling các APIs (Glance, Cinder, Neutron).
- **ceilometer-agent-notification** lắng nghe notification queue từ các services (Nova, Cinder, Neutron), nhận events như instance.create, volume.attach, network.create và convert thành metrics.
- **ceilometer-collector** nhận data từ agents qua message queue, aggregate và buffer data, sau đó write vào Ceilometer database hoặc external storage systems (SQL, NoSQL, time-series databases).
- **ceilometer-api** cung cấp API để query metrics và alarms, tính toán statistics (min, max, avg, sum, count) với khả năng group by resource, project, user, metadata.
- **ceilometer-alarm-evaluator** đọc alarms từ database, query metrics để evaluate thresholds, và trigger alarm state changes (ok → alarm → insufficient data).
- **ceilometer-alarm-notifier** gửi notifications khi alarm trigger thông qua các channels như email, webhook, log, hoặc Heat (cho auto-scaling).

**OpenStack Bare Metal (Ironic)**: Ironic cung cấp dịch vụ bare metal provisioning, cho phép provision và quản lý physical servers (thay vì virtual machines) giống như cách Nova quản lý VMs. Ironic phù hợp cho các use cases yêu cầu hiệu năng cao, hardware-specific features, hoặc compliance requirements không thể chạy trên VMs.
- Thành phần:
  - **ironic-api**: là API cho bare metal operations như đăng ký nodes, provision, deprovision physical servers.
  - **ironic-conductor** quản lý node lifecycle, giao tiếp với hardware drivers để thực hiện power control, boot device configuration, và OS deployment.
  - **Drivers (IPMI, Redfish, iRMC, WSMAN)** cung cấp interface để điều khiển hardware: power on/off, set boot device (PXE, disk, CDROM), và deploy operating systems thông qua network boot (PXE) hoặc direct imaging.

**OpenStack Database Service(TROVE)**:  Trove cung cấp Database-as-a-Service (DBaaS), cho phép người dùng tự cung cấp/tạo mới, thiết lập(provision) và quản lý database instances (MySQL, PostgreSQL, MongoDB, Redis...) mà không cần tự cài đặt và cấu hình. Trove tự động hóa việc deploy, configure, backup, và scale databases.
  - **trove-api**: API cho database operations như create DB instances, users, databases.
  - **trove-taskmanager**: thực hiện async tasks như create, delete, resize instances, và orchestrate
  - **trove-conductor**: nhận heartbeat từ guest agents và update instance status.
  - **trove-guest-agent**: chạy bên trong database instance VM

**OpenStack Data Processing (SAHARA)**: Sahara cung cấp dịch vụ data processing, cho phép deploy và manage big data clusters (Hadoop, Spark) trên OpenStack. Sahara tự động hóa việc provision cluster nodes, install và configure Hadoop/Spark, và execute data processing jobs.
- Thành phần:
  - **Sahara all**: cung cấp API và engine cho data processing operations, deploy Hadoop/Spark clusers, manage data sources và job execution. Component này tự động provision VMs, install và configure Hadoop ecosystem components (HDFS, YARN, MapReduce, Spark)

**Magnum**: HMagnum cung cấp API để triển khai Kubernetes hoặc Docker Swarm trên OpenStack. 

**Octavia**: Octavia cung cấp Load Balancer-as-a-Service, hỗ trợ L4/L7 load balancing.

#### 2.4 Các thành phần chung
**Message Queue**(RabbitMQ hoặc ZeroMQ) là backbone communication của OpenStack, cung cấp asynchronous(bất đồng bộ) messaging giữa các services thông qua RPC (Request-reply) và pub-sub (fanout) patterns

**Databases**: (MySQL hoặc PostgreSQL) lưu metadata của tất cả services, mỗi service có database riêng, với replication và clustering cho high availability.
  - **Replication**: Sao chép đồng bộ dữ liệu, là cơ chế tự động copy dữ liệu từ 1 database chính -> nhiều database phụ. Chống mất dữ liệu. Nếu server chính hỏng, server phụ vẫn giữ bản copy mới nhất để khôi phục hoặc tiếp tục phục vụ.-> Chống mất mát
  - **Clustering**: Cụm máy chủ, là kỹ thuật gom nhiều máy chủ database lại thành một nhóm (cluster) hoạt động như một hệ thống thống nhất. Chống gián đoạn dịch vụ. Cluster có cơ chế giám sát lẫn nhau, nếu 1 node bị lỗi, các node còn lại tự động tiếp quản công việc (failover) mà không cần can thiệp thủ công. -> Chống downtime

**Hypervisors** (KVM/QEMU - default, XEN, VMware ESXI, Hyper- V) cung cấp virtualization layer, với libvirt làm abstraction layer để Nova giao tiếp với các hypervisors khác.

**LDAP**(Optional) thay thế SQL database cho Keystone users/groups, tích hợp với enterprise directories như Active Directory

## II. LDAP
### 1. Khái niệm

LDAP (Lightweight Directory Access Protocol) là một giao thức dùng để truy vấn và quản lý thông tin dạng “directory” (danh bạ) qua mạng. Nó thường được dùng trong hệ thống doanh nghiệp để xác thực người dùng (login) và lưu trữ thông tin tập trung như user, nhóm, quyền, thiết bị…

Hiểu đơn giản:

- Bạn có thể hình dung LDAP giống như một “cuốn danh bạ tổ chức” online:
- Mỗi người dùng có thông tin: username, password, email, phòng ban…
- Các ứng dụng (web, VPN, server…) sẽ hỏi LDAP:
  - “User này có tồn tại không? Password đúng không?”
  - → LDAP trả lời → quyết định cho login hay không.

### 2. Cách hoạt động LDAP
Thành phần chính:
- LDAP Server: nơi lưu dữ liệu (ví dụ: OpenLDAP)
- LDAP Client: app/service gửi request (web app, SSH, VPN…)
- Directory Information Tree (DIT): cấu trúc cây dữ liệu

Quy trình xác thực (authentication):
- Client gửi request login (username/password)
- LDAP client gửi request đến LDAP server
  - Khi cấu hình chuẩn nâng cao thì Keystone sẽ đóng vai trò như một LDAP Client
  - User login vào OpenStack
  - Keystone nhận request
  - Keystone query LDAP (ví dụ OpenLDAP)
  - LDAP trả kết quả
  - Keystone:
    - nếu OK → cấp token
    - nếu fail → từ chối
- LDAP server:
  - Tìm user trong cây dữ liệu
  - So sánh password
  - Trả kết quả:
   - Đúng → cho phép truy cập
   - Sai → từ chối

### 3. Cấu trúc dữ liệu LDAP
LDAP lưu dữ liệu theo dạng cây (tree):
- **DN (Distinguished Name)**: định danh đầy đủ
```bash
 uid=trungg,ou=users,dc=example,dc=com
```
- **Entry**: một object (user, group…)
- **Attribute**: thuộc tính

Ví dụ:
```bash
uid: trungg
mail: trungg@example.com
```
- **ObjectClass**: kiểu object (user, group…)

### 4. Các thao tác chính trong LDAP
LDAP không phải database SQL, nó dùng các operation riêng:
- **Bind** → đăng nhập vào LDAP
- **Search** → tìm kiếm user/data
- **Add** → thêm entry
- **Modify** → sửa
- **Delete** → xóa

### 5. LDAP dùng để làm gì?
Phổ biến nhất:
#### Xác thực tập trung (SSO kiểu đơn giản)
- Đăng nhập 1 lần → dùng nhiều hệ thống
- Dùng trong:
  - Linux login
  - SSH
  - Web app
#### Quản lý user trong doanh nghiệp
- Lưu toàn bộ nhân viên
- Phân quyền theo group
#### Tích hợp với hệ thống lớn
LDAP thường đi cùng:
- Active Directory (Microsoft dùng LDAP + Kerberos)
- FreeIPA
- OpenStack Keystone

Một số port LDAP: 389 → LDAP thường, 636 → LDAPS (LDAP qua SSL)

Dùng khi:
- Có nhiều service cần login chung
- Muốn quản lý user tập trung
- Hệ thống enterprise / lab lớn

Tóm lại: “Bộ não quản lý danh tính” trong hệ thống lớn
- Lưu user
- Xác thực
- Phân quyền
- Dùng chung cho nhiều service

So sánh với Database
- LDAP rất mạnh về authentication (xác thực) và quản lý danh tính
- LDAP (directory service)
- Lưu dữ liệu danh tính (identity data):
  - `user`
  - `group`
  - `quyền`
  - `thông tin tổ chức`
- Dữ liệu có đặc điểm:
  - ít thay đổi
  - đọc nhiều hơn ghi (read-heavy)
- Cấu trúc: cây (hierarchical)

- Database (MySQL, PostgreSQL…) dùng cho dữ liệu ứng dụng thay đổi liên tục
- Lưu dữ liệu nghiệp vụ (business data):
  - đơn hàng
  - log
  - bài viết
- Dữ liệu:
  - thay đổi liên tục
  - insert/update/delete nhiều
- Cấu trúc: bảng, document…
