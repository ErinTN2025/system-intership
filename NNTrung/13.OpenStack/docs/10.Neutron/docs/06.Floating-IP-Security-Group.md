# Floating IP & Security Group – Chi tiết và Lab

## 1. Floating IP – Khái niệm và Workflow

### 1.1 Floating IP là gì?

**Floating IP (FIP)** là một địa chỉ IP **public** (lấy từ external network) được map (DNAT/SNAT) đến một địa chỉ IP **private** của VM. Mục đích:

- Cho phép truy cập VM từ Internet trong khi VM vẫn giữ IP private.
- IP public có thể **dời** từ VM này sang VM khác (khi sự cố, blue-green deploy, v.v.).
- Quản lý IPv4 public scarce — chỉ cấp FIP cho VM nào thực sự cần.

### 1.2 Khác biệt với "IP cố định" (Fixed IP)

| | Fixed IP | Floating IP |
|---|---|---|
| Nguồn | Subnet của tenant network | External network |
| Bound vào | Port (1-1 cố định) | Router gateway → DNAT về fixed IP |
| Truy cập từ ngoài | Không (cần qua router NAT/FIP) | Có |
| Có thể đổi VM | Không | Có (associate/disassociate) |

### 1.3 Workflow Floating IP

```
                    Internet
                      │
                      │ (192.168.1.150  ← FIP)
                      ▼
              ┌─────────────────┐
              │ Router (qrouter │
              │  hoặc OVN LR)   │
              │ ─────────────── │
              │ DNAT:           │
              │  192.168.1.150  │
              │  → 192.168.10.5 │
              │ SNAT:           │
              │  192.168.10.5   │
              │  → 192.168.1.150│
              └────────┬────────┘
                       │
                  ┌────▼────┐
                  │   VM    │ 192.168.10.5  (fixed IP)
                  └─────────┘
```

**Bước-by-bước**:

1. **Allocate**: User request FIP từ external network → Neutron reserve 1 IP trong `allocation_pool`.
2. **Associate**: User gắn FIP vào VM (chính xác là gắn vào **port** của VM).
3. **Apply NAT**:
   - Legacy ML2/OVS: thêm iptables rule trong namespace `qrouter-<id>` (`-A POSTROUTING -s <fixed> -j SNAT --to <fip>`; `-A PREROUTING -d <fip> -j DNAT --to <fixed>`).
   - OVN: thêm vào NB `NAT` table → tự tạo flow rule trên OVS.
4. **Disassociate/Delete**: Gỡ FIP → xóa NAT rule. FIP có thể được gán cho VM khác hoặc release về pool.

### 1.4 SNAT vs DNAT vs Floating IP

| Loại NAT | Hướng | Khi nào | Implementation |
|---|---|---|---|
| **SNAT** (Source NAT) | VM → Internet | Mặc định khi router có external gateway | Router NAT source thành IP gateway |
| **DNAT** (Destination NAT) | Internet → VM | Khi assign FIP | Map FIP → fixed IP |
| **Floating IP** | Cả 2 chiều | Khi assign FIP | DNAT inbound + SNAT outbound dùng FIP làm source |

> **Lưu ý**: Khi VM có FIP, outbound traffic sẽ SNAT thành **FIP** (không phải gateway IP). Khi VM không có FIP, outbound SNAT thành **gateway IP** của router.

## 2. Floating IP Lab

```bash
# ==== Trên CONTROLLER ====
source /root/admin-openrc

# 1. Verify external network có và router đã set gateway
openstack network list --external
openstack router show R1 -c external_gateway_info

# 2. Allocate FIP (random IP từ pool)
FIP=$(openstack floating ip create public -f value -c floating_ip_address)
echo "FIP: $FIP"

# 3. Hoặc allocate IP cụ thể
openstack floating ip create --floating-ip-address 192.168.1.150 public

# 4. Liệt kê FIP đang có
openstack floating ip list

# 5. Associate vào VM
openstack server add floating ip vm-a1 $FIP

# 6. Verify trong VM (qua console)
openstack console url show vm-a1

# Trong console:
# ip addr   # Vẫn chỉ thấy fixed IP - FIP nằm trên ROUTER, không trên VM
# Đây là điểm gây nhầm lẫn phổ biến!

# 7. Verify NAT rule
# === Legacy ML2/OVS ===
ROUTER_ID=$(openstack router show R1 -f value -c id)
ssh network01 "sudo ip netns exec qrouter-$ROUTER_ID iptables -t nat -L -n -v"
# Sẽ thấy DNAT + SNAT cho FIP

# === OVN ===
sudo ovn-nbctl lr-nat-list neutron-$ROUTER_ID

# 8. Ping FIP từ ngoài
ping -c 3 $FIP

# 9. SSH bằng FIP
ssh -i ~/.ssh/id_rsa cirros@$FIP

# 10. Disassociate
openstack server remove floating ip vm-a1 $FIP

# 11. Release FIP
openstack floating ip delete $FIP
```

### 2.1 Floating IP "không cần" gắn vào VM

Một số use case (HA, K8s LoadBalancer) cần FIP gắn vào **port** không thuộc VM:

```bash
# Tạo port standalone (vd: cho VIP)
openstack port create --network net-a \
  --fixed-ip subnet=subnet-a,ip-address=192.168.10.99 vip-port

# Gắn FIP vào port đó
openstack floating ip create --port vip-port public
```

### 2.2 Port Forwarding (không cần dùng FIP cho mỗi VM)

OpenStack hỗ trợ **Floating IP Port Forwarding** — 1 FIP có thể forward nhiều port khác nhau đến nhiều VM/port nội bộ:

```bash
# Yêu cầu service_plugins có "port_forwarding"

# 1. Tạo FIP không gắn VM
FIP_ID=$(openstack floating ip create public -f value -c id)

# 2. Forward TCP/8080 (external) → vm-a1:22 (SSH)
openstack floating ip port forwarding create \
  --internal-ip-address 192.168.10.10 \
  --port 22 \
  --internal-port-range 22:22 \
  --external-port 8080 \
  --external-port-range 8080:8080 \
  --protocol tcp \
  $FIP_ID

# 3. Forward TCP/8081 → vm-a2:22
openstack floating ip port forwarding create \
  --internal-ip-address 192.168.10.11 \
  --port 22 \
  --external-port 8081 \
  --protocol tcp \
  $FIP_ID

# 4. Liệt kê
openstack floating ip port forwarding list $FIP_ID

# 5. SSH qua port forward
ssh -p 8080 cirros@<FIP>     # → vm-a1
ssh -p 8081 cirros@<FIP>     # → vm-a2
```

> Tiết kiệm IP public — 1 FIP phục vụ nhiều VM (như behind NAT của router gia đình).

## 3. Security Group – Stateful Firewall

### 3.1 Khái niệm

Security Group (SG) là **stateful firewall** ở Layer 3/4, áp dụng tại **port** (vNIC) của VM. Mỗi port có thể gắn với 1 hoặc nhiều SG.

**Stateful** nghĩa là:
- Nếu allow `ingress TCP 22`, response từ VM tự động được allow (không cần mở egress).
- Connection được track (`conntrack`) — chỉ rule chiều đi cần khai báo.

### 3.2 Default Security Group

Mỗi project tự động có 1 SG tên `default`. Nó:
- **Ingress**: chỉ cho phép traffic từ chính group `default` (giữa các VM trong project).
- **Egress**: cho phép tất cả.

Nếu VM không gán SG nào → tự động gán `default` → VM **không bị ping được từ ngoài** (vì ingress default chỉ allow intra-group).

### 3.3 Implementation

| Stack | Backend | Vị trí |
|---|---|---|
| ML2/OVS + iptables | iptables_hybrid | iptables rule trên compute node (qua linux bridge `qbr-`) |
| ML2/OVS + native | openvswitch firewall | OVS flow rule trên `br-int` |
| ML2/OVN | OVN ACL | OVN logical flow → OpenFlow trên `br-int` |

### 3.4 Cấu trúc SG Rule

| Trường | Giá trị | Ý nghĩa |
|---|---|---|
| `direction` | `ingress` / `egress` | Vào hoặc ra |
| `ether_type` | `IPv4` / `IPv6` | Loại IP |
| `protocol` | `tcp`, `udp`, `icmp`, `ah`, `esp`, ... | Layer 4 |
| `port_range_min`, `port_range_max` | Số port hoặc range | Port (chỉ với TCP/UDP) |
| `remote_ip_prefix` | CIDR | Allow từ dải IP nào |
| `remote_group_id` | SG khác | Allow từ VM nào có SG đó |
| `description` | text | Mô tả |

### 3.5 So sánh: `remote_ip_prefix` vs `remote_group`

```bash
# Cách 1: cho phép từ IP cố định
openstack security group rule create --proto tcp --dst-port 3306 \
  --remote-ip 192.168.10.0/24 db-sg

# Cách 2 (dynamic): cho phép từ VM có SG "web"
openstack security group rule create --proto tcp --dst-port 3306 \
  --remote-group web db-sg
```

→ **Cách 2 tốt hơn** vì khi thêm VM mới vào SG `web`, rule tự áp dụng. Không cần update IP whitelist.

### 3.6 Sample Security Group thường gặp

#### Web server (cho VM web)
```bash
openstack security group create web --description "Web server"
openstack security group rule create --proto icmp                          web
openstack security group rule create --proto tcp --dst-port 22 --remote-ip 10.0.0.0/8 web   # SSH chỉ từ internal
openstack security group rule create --proto tcp --dst-port 80            web
openstack security group rule create --proto tcp --dst-port 443           web
```

#### Database (chỉ từ web tier)
```bash
openstack security group create db
openstack security group rule create --proto tcp --dst-port 22 --remote-ip 10.0.0.0/8 db
openstack security group rule create --proto tcp --dst-port 3306 --remote-group web db
```

#### Bastion (jump host)
```bash
openstack security group create bastion
openstack security group rule create --proto icmp                                bastion
openstack security group rule create --proto tcp --dst-port 22 --remote-ip 0.0.0.0/0 bastion
```

#### Internal-only (chỉ talk nhau)
```bash
openstack security group create internal
openstack security group rule create --proto tcp --dst-port 1:65535 --remote-group internal internal
openstack security group rule create --proto udp --dst-port 1:65535 --remote-group internal internal
openstack security group rule create --proto icmp                  --remote-group internal internal
```

### 3.7 Apply SG cho VM hoặc Port

```bash
# Khi tạo VM
openstack server create vm-web --image cirros --flavor m1.tiny \
  --network net-a \
  --security-group web --security-group internal \
  vm-web

# Thêm/bỏ SG sau khi VM tạo
openstack server add security group vm-web db
openstack server remove security group vm-web db

# Apply trực tiếp lên port (cần thiết khi 1 VM có nhiều port khác SG)
openstack port set --security-group web <port-uuid>
openstack port unset --security-group web <port-uuid>
```

### 3.8 Disable Port Security (cho VNF / SR-IOV)

Đôi khi (VNF, load balancer dùng promiscuous mode) cần tắt hoàn toàn filter:

```bash
# Phải xóa hết SG trước
openstack port unset --security-group <sg> <port>

# Sau đó disable port security
openstack port set --disable-port-security <port>
```

> Cảnh báo: VM bị disable port security có thể spoof MAC/IP → security risk.

### 3.9 Allowed Address Pairs (cho HA VIP)

Mặc định Neutron block traffic không match MAC/IP của port → ngăn IP spoof. Nhưng với HA (VRRP, keepalived), VIP cần "trôi" giữa các VM → phải khai báo:

```bash
# Cho phép port phụ trách thêm VIP 192.168.10.99
openstack port set --allowed-address ip-address=192.168.10.99 <port>

# Cả MAC khác (vd cho HA cluster có virtual MAC)
openstack port set --allowed-address mac-address=fa:16:3e:99:99:99,ip-address=192.168.10.99 <port>
```

## 4. Lab Security Group End-to-End

```bash
# ==== Trên CONTROLLER ====
source /root/admin-openrc

# 1. Tạo SG
openstack security group create lab-web
openstack security group rule create --proto icmp                      lab-web
openstack security group rule create --proto tcp --dst-port 22         lab-web
openstack security group rule create --proto tcp --dst-port 80         lab-web

openstack security group create lab-db
openstack security group rule create --proto icmp                                lab-db
openstack security group rule create --proto tcp --dst-port 3306 --remote-group lab-web lab-db

# 2. Tạo 2 VM
openstack server create web-vm --image cirros --flavor m1.tiny \
  --network net-a --security-group lab-web --key-name mykey --wait

openstack server create db-vm  --image cirros --flavor m1.tiny \
  --network net-a --security-group lab-db --key-name mykey --wait

WEB_IP=$(openstack server show web-vm -f value -c addresses | awk -F'=' '{print $2}')
DB_IP=$(openstack server show db-vm  -f value -c addresses | awk -F'=' '{print $2}')

# 3. Cấp FIP cho web-vm
FIP_WEB=$(openstack floating ip create public -f value -c floating_ip_address)
openstack server add floating ip web-vm $FIP_WEB

# 4. SSH vào web-vm, test connectivity
ssh -i ~/.ssh/id_rsa cirros@$FIP_WEB <<EOF
# Ping db-vm (ICMP allowed cho cả 2 SG)
ping -c 2 $DB_IP

# Mở fake "DB" trên db-vm port 3306
# (Trên 1 terminal khác đầu tiên ssh vào db-vm và chạy: nc -l -p 3306 &)
nc -zv $DB_IP 3306
# Phải kết nối được vì web-vm thuộc SG lab-web → match remote-group

# Test port khác (5432) → fail
nc -zv $DB_IP 5432
EOF

# 5. Tách web-vm khỏi SG lab-web → test lại → fail
openstack server remove security group web-vm lab-web
openstack server add security group web-vm default

# SSH lại - giờ web-vm không còn ở SG lab-web, nên rule remote-group không match
ssh -i ~/.ssh/id_rsa cirros@$FIP_WEB nc -zv $DB_IP 3306
# Connection refused / timeout

# 6. Verify thực tế trên compute (legacy iptables_hybrid)
ssh compute01 sudo iptables -L -n | grep 3306

# 7. OVN: xem ACL
sudo ovn-nbctl acl-list <port-group-id>

# 8. Cleanup
openstack server remove floating ip web-vm $FIP_WEB
openstack floating ip delete $FIP_WEB
openstack server delete web-vm db-vm
openstack security group delete lab-web lab-db
```

## 5. Troubleshooting Floating IP & SG

### 5.1 FIP không ping được

| Check | Cách kiểm |
|---|---|
| FIP đã associate VM? | `openstack server show <vm>` → `addresses` phải có 2 IP |
| Router có external gateway? | `openstack router show <r> -c external_gateway_info` |
| Router interface đã thêm subnet? | `openstack port list --router <r>` |
| Subnet có gateway đúng? | `openstack subnet show <s>` |
| SG cho phép ICMP? | `openstack security group rule list <sg>` |
| Compute đến gateway thông không? | `ping` từ compute đến gateway |
| Legacy: namespace `qrouter` có FIP? | `ip netns exec qrouter-<id> ip addr` |
| OVN: NAT rule trong NB? | `ovn-nbctl lr-nat-list <lr>` |

### 5.2 SG không có tác dụng

```bash
# 1. Verify SG đã apply
openstack port show <port> -c security_group_ids

# 2. Verify rule
openstack security group rule list <sg>

# 3. Verify backend
# === Legacy iptables_hybrid ===
ssh compute01 sudo iptables -L | grep <port-id-short>

# === OVS native firewall ===
ssh compute01 sudo ovs-ofctl dump-flows br-int | grep <vm-mac>

# === OVN ===
sudo ovn-nbctl acl-list neutron_pg_drop
sudo ovn-nbctl find acl name=<sg-id>
```

### 5.3 Conntrack debug

```bash
# Xem connection đang track (legacy)
ssh compute01 sudo conntrack -L | grep <vm-ip>

# Flush conntrack nếu thay đổi rule không có hiệu lực ngay
ssh compute01 sudo conntrack -F
```

### 5.4 Port stuck DOWN sau gán SG

```bash
# Restart neutron-openvswitch-agent (legacy)
ssh compute01 sudo systemctl restart neutron-openvswitch-agent

# Hoặc force rebuild flow (OVN)
sudo ovn-appctl -t ovn-controller recompute
```

## 6. Best practices

### 6.1 Floating IP
- **Một VM → một FIP** là đủ trong hầu hết trường hợp.
- Dùng **Port Forwarding** thay vì nhiều FIP nếu chỉ cần expose vài port.
- Allocate FIP **trước** khi cần — tránh bị scheduler từ chối lúc cao điểm.
- **Theo dõi quota FIP** — dễ bị exhaust ở scale lớn.

### 6.2 Security Group
- **Đặt tên SG theo role** (`web`, `db`, `bastion`) chứ không theo loại traffic.
- **Dùng `remote-group` thay vì IP** khi có thể.
- **Mỗi VM gán ≥2 SG**: 1 SG role + 1 SG `internal` (cho intra-tier).
- **Không xóa default SG** — dùng nó cho intra-project communication.
- **Audit định kỳ**: `openstack security group rule list` xem rule mở `0.0.0.0/0` có còn cần không.
- **Không disable port security** trừ khi thực sự cần (VNF/SR-IOV).
- **Stateful nên không cần mở egress cho response** — chỉ mở ingress là đủ với TCP/UDP.
