# Kiến trúc OVN (Open Virtual Network)

## 1. OVN là gì?

**OVN** (Open Virtual Network) là dự án con của Open vSwitch, cung cấp **SDN stack hoàn chỉnh** thay thế mô hình ML2/OVS truyền thống.

**Tại sao chọn OVN thay ML2/OVS?**

| Tiêu chí | ML2/OVS (cũ) | ML2/OVN (mới) |
|---|---|---|
| L3 routing | `neutron-l3-agent` + namespace `qrouter` | OVN logical router (distributed) |
| DHCP | `neutron-dhcp-agent` + dnsmasq + namespace `qdhcp` | Native trong OVN flow |
| Metadata | `neutron-metadata-agent` + haproxy + namespace | `ovn-metadata-agent` đọc trực tiếp SB DB |
| Security Group | iptables (mỗi compute node) | OVN ACL (flow-based, nhanh hơn) |
| Đồng bộ | RPC qua RabbitMQ | OVSDB protocol (binary, nhanh hơn) |
| Single point of failure | Network node | Không (distributed) |
| Multi-chassis L2 binding | l2population | Native (faster) |

**Tóm lại**: OVN gom tất cả logic L2/L3/DHCP/Metadata/Firewall vào **OpenFlow rules** trên OVS, do **ovn-controller** trên mỗi node áp đặt.

## 2. Các thành phần của OVN

### 2.1 Sơ đồ kiến trúc

```
                ┌─────────────────────────────────────┐
                │     Neutron Server (Controller)     │
                │     ML2 mechanism driver: OVN       │
                └──────────────┬──────────────────────┘
                               │  OVSDB protocol (port 6641)
                               ▼
                ┌─────────────────────────────────────┐
                │   ovn-northd  (NB DB → SB DB)       │
                │   ┌─────────────┐ ┌──────────────┐  │
                │   │ Northbound  │→│  Southbound  │  │
                │   │   DB        │ │     DB       │  │
                │   │ (logical)   │ │  (physical)  │  │
                │   └─────────────┘ └──────────────┘  │
                └─────────────────────────────────────┘
                               │  OVSDB protocol (port 6642)
            ┌──────────────────┼──────────────────┐
            ▼                  ▼                  ▼
      ┌──────────┐       ┌──────────┐       ┌──────────┐
      │ compute1 │       │ compute2 │       │ network1 │
      │          │       │          │       │ (gateway)│
      │ovn-      │       │ovn-      │       │ovn-      │
      │controller│       │controller│       │controller│
      │ ↓        │       │ ↓        │       │ ↓        │
      │OVS br-int│       │OVS br-int│       │OVS br-int│
      │ + flows  │       │ + flows  │       │ + flows  │
      └──────────┘       └──────────┘       └──────────┘
```

### 2.2 Các process

| Process | Chạy trên | Vai trò |
|---|---|---|
| **`ovn-northd`** | Controller (HA cluster) | Đọc Northbound DB → translate thành flow rules → ghi vào Southbound DB |
| **`ovsdb-server` NB** | Controller (HA) | Lưu trữ NB DB (logical topology mà Neutron ghi xuống) |
| **`ovsdb-server` SB** | Controller (HA) | Lưu trữ SB DB (binding + flow + chassis info) |
| **`ovn-controller`** | Mỗi compute/network node | Đọc SB DB → cấu hình OpenFlow lên OVS local |
| **`ovs-vswitchd`** | Mỗi node | OVS datapath thực sự xử lý packet |
| **`ovn-metadata-agent`** | Mỗi compute | Cung cấp metadata service (proxy về Nova) |

### 2.3 Northbound DB (NB)

NB lưu **logical topology** — những gì user/admin/Neutron tạo:

| NB Table | Tương ứng Neutron |
|---|---|
| `Logical_Switch` (`LS`) | Network |
| `Logical_Switch_Port` (`LSP`) | Port |
| `Logical_Router` (`LR`) | Router |
| `Logical_Router_Port` (`LRP`) | Router interface |
| `NAT` | Floating IP / SNAT |
| `ACL` | Security Group rule |
| `Address_Set` | Security Group (group of IPs) |
| `QoS` | QoS policy |
| `Load_Balancer` | Octavia LBaaS |
| `DHCP_Options` | DHCP setting |

### 2.4 Southbound DB (SB)

SB lưu **physical state** — chassis (compute node), port binding, computed flows:

| SB Table | Vai trò |
|---|---|
| `Chassis` | Mỗi compute/network node |
| `Port_Binding` | Logical port nào ở chassis nào |
| `Datapath_Binding` | Tunnel key của logical switch/router |
| `Logical_Flow` | Flow rule (sẽ được translate thành OpenFlow) |
| `MAC_Binding` | ARP/ND cache |
| `Encap` | Geneve/STT/VXLAN tunnel info |

## 3. Workflow tạo network/VM trong OVN

### 3.1 Flow khi user `openstack network create net-a`

```
1. User → Neutron API
2. Neutron mechanism driver OVN → ghi vào NB DB:
     ovn-nbctl ls-add neutron-<uuid>
3. ovn-northd phát hiện NB thay đổi → tính toán logical flow → ghi vào SB DB
4. Mỗi ovn-controller (trên compute) → subscribe SB DB → nhận updated flow
5. ovn-controller dịch logical flow thành OpenFlow rule → cài vào OVS br-int
```

Verify:
```bash
# Logical switch trong NB
sudo ovn-nbctl ls-list

# Datapath binding trong SB
sudo ovn-sbctl list datapath_binding

# Flow trên OVS
sudo ovs-ofctl dump-flows br-int | head -20
```

### 3.2 Flow khi user tạo VM gắn vào net-a

```
1. Nova gọi Neutron tạo port  → ghi NB:
     ovn-nbctl lsp-add neutron-<net-uuid> <port-uuid>
     ovn-nbctl lsp-set-addresses <port-uuid> "MAC IP"
     ovn-nbctl lsp-set-port-security <port-uuid> "MAC IP"
2. ovn-northd → SB Port_Binding (chassis = NULL, chưa bound)
3. Nova-compute tạo VM, gắn tap vào OVS br-int + set external_ids:iface-id=<port-uuid>
4. ovn-controller trên chassis đó phát hiện tap mới → bind port:
     UPDATE SB Port_Binding SET chassis=<this-chassis> WHERE uuid=<port-uuid>
5. ovn-northd recompute → đẩy flow xuống tất cả chassis (để các node khác biết port này ở đâu)
6. VM gửi DHCPDISCOVER → OVN native DHCP trả lời ngay trong flow (không qua dnsmasq)
7. VM có IP, send/receive traffic qua tunnel Geneve
```

### 3.3 Flow Floating IP

```
1. User: openstack floating ip create + add to VM
2. Neutron → NB NAT table:
     ovn-nbctl lr-nat-add <router> dnat_and_snat <fip> <vm-private-ip>
3. ovn-northd → SB logical_flow (NAT rule)
4. Khi packet đến FIP → ovn-controller match flow → DNAT về VM
```

## 4. Các lệnh debug OVN cốt lõi

### 4.1 `ovn-nbctl` — quản lý NB DB

```bash
# Hiển thị toàn bộ logical topology
sudo ovn-nbctl show

# Liệt kê
sudo ovn-nbctl ls-list                       # Logical switches
sudo ovn-nbctl lr-list                       # Logical routers
sudo ovn-nbctl lsp-list <ls>                 # Ports on a switch
sudo ovn-nbctl lrp-list <lr>                 # Ports on a router

# Chi tiết port
sudo ovn-nbctl lsp-get-addresses <port>
sudo ovn-nbctl lsp-get-port-security <port>
sudo ovn-nbctl lsp-get-type <port>

# ACL (Security Group rules)
sudo ovn-nbctl acl-list <port-group>
sudo ovn-nbctl list ACL

# NAT (Floating IP)
sudo ovn-nbctl lr-nat-list <router>

# DHCP options
sudo ovn-nbctl list DHCP_Options
```

### 4.2 `ovn-sbctl` — quản lý SB DB

```bash
# Hiển thị binding
sudo ovn-sbctl show

# Liệt kê chassis (compute/network nodes)
sudo ovn-sbctl list Chassis

# Port binding
sudo ovn-sbctl list Port_Binding
sudo ovn-sbctl find Port_Binding logical_port=<port-uuid>

# Flow đã compute
sudo ovn-sbctl lflow-list                    # Tất cả logical flow
sudo ovn-sbctl lflow-list <datapath>         # Filter theo datapath

# MAC bindings (ARP cache)
sudo ovn-sbctl list MAC_Binding
```

### 4.3 `ovn-trace` — Debug packet flow (CỰC MẠNH)

`ovn-trace` mô phỏng đường đi của 1 packet qua logical pipeline → cho biết flow nào match, drop ở đâu, NAT thế nào.

```bash
# Cú pháp:
sudo ovn-trace [--detailed] <datapath> '<microflow-expression>'

# Ví dụ: trace ICMP từ VM (192.168.10.10) đến VM khác (192.168.10.20)
sudo ovn-trace --detailed neutron-<uuid-of-net-a> \
  'inport=="<vm1-port-uuid>" \
   && eth.src==fa:16:3e:11:11:11 \
   && eth.dst==fa:16:3e:22:22:22 \
   && ip4.src==192.168.10.10 \
   && ip4.dst==192.168.10.20 \
   && ip.ttl==64 \
   && icmp4.type==8 && icmp4.code==0'

# Trace cross-network qua router
sudo ovn-trace --detailed neutron-<uuid-of-net-a> \
  'inport=="<vm-a1-port>" \
   && eth.src==<vm-a1-mac> \
   && eth.dst==<router-mac> \
   && ip4.src==192.168.10.10 \
   && ip4.dst==192.168.20.10 \
   && ip.ttl==64 && icmp4.type==8'
```

**Output** sẽ cho biết:
- Packet đi qua những logical flow nào
- Action: forward, drop, NAT, conntrack
- Có hit ACL nào không

### 4.4 `ovs-ofctl` — Inspect OpenFlow rules trên local OVS

```bash
# Liệt kê bridges
sudo ovs-vsctl list-br

# Dump tất cả flow
sudo ovs-ofctl dump-flows br-int

# Filter
sudo ovs-ofctl dump-flows br-int 'table=0'
sudo ovs-ofctl dump-flows br-int 'tcp,tp_dst=22'

# Group table (cho ECMP / hash)
sudo ovs-ofctl dump-groups br-int

# Port stats
sudo ovs-ofctl dump-ports br-int
```

### 4.5 `ovn-appctl` — Runtime control

```bash
# Trên ovn-controller
sudo ovn-appctl -t ovn-controller list-commands
sudo ovn-appctl -t ovn-controller debug/status
sudo ovn-appctl -t ovn-controller recompute       # Force recompute flows

# Trên ovn-northd
sudo ovn-appctl -t ovn-northd nb-cond-set         # Conditional monitoring
```

## 5. Mô hình HA của OVN

### 5.1 OVN DB cluster (RAFT)

OVN DB (NB + SB) hỗ trợ **RAFT clustering** từ 2.11+ → highly available không cần Pacemaker.

**Cài đặt 3-node RAFT cluster** (ví dụ controller01, controller02, controller03):

```bash
# Trên controller01 (init)
sudo ovn-ctl --db-nb-addr=10.10.34.161 --db-nb-cluster-local-addr=10.10.34.161 \
             --db-sb-addr=10.10.34.161 --db-sb-cluster-local-addr=10.10.34.161 \
             --db-nb-create-insecure-remote=yes \
             --db-sb-create-insecure-remote=yes \
             start_nb_ovsdb start_sb_ovsdb

# Trên controller02, controller03 (join)
sudo ovn-ctl --db-nb-addr=10.10.34.162 --db-nb-cluster-local-addr=10.10.34.162 \
             --db-nb-cluster-remote-addr=10.10.34.161 \
             --db-sb-addr=10.10.34.162 --db-sb-cluster-local-addr=10.10.34.162 \
             --db-sb-cluster-remote-addr=10.10.34.161 \
             start_nb_ovsdb start_sb_ovsdb

# Verify cluster status
sudo ovn-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound
```

### 5.2 Gateway Chassis (cho external traffic)

Trong OVN, **gateway chassis** là node xử lý traffic ra/vào external network (centralized cho mỗi router). Có thể chạy multiple gateway cho HA.

```bash
# Đánh dấu node là gateway
sudo ovs-vsctl set open . external-ids:ovn-cms-options=enable-chassis-as-gw

# Verify
sudo ovn-sbctl list chassis | grep gateway

# Gán router vào gateway chassis (Neutron tự làm)
sudo ovn-nbctl lrp-set-gateway-chassis <router-port> <chassis-name> 100
```

### 5.3 Distributed Gateway Port (DGP)

OVN hỗ trợ **multi-gateway active-active** cho 1 router → phân tải SNAT/FIP qua nhiều node.

```bash
# Thêm gateway chassis với priority
sudo ovn-nbctl lrp-set-gateway-chassis lrp-<id> chassis1 100
sudo ovn-nbctl lrp-set-gateway-chassis lrp-<id> chassis2 90    # backup
sudo ovn-nbctl lrp-set-gateway-chassis lrp-<id> chassis3 80
```

## 6. So sánh chi tiết các luồng

### 6.1 Tạo VM → DHCP

**ML2/OVS legacy**:
```
VM tap → br-int → patch port br-ex → namespace qdhcp → dnsmasq → reply ngược lại
```
Cần `qdhcp-<net>` namespace + dnsmasq process trên network node.

**ML2/OVN**:
```
VM tap → br-int → OVS flow match DHCP → ovn-controller respond inline
```
Không cần namespace, không cần dnsmasq → faster, ít tài nguyên hơn.

### 6.2 Cross-network routing

**ML2/OVS legacy (centralized)**:
```
vm-a → tap → br-int → patch br-tun → tunnel → network-node br-tun → patch br-int → qrouter ns → ip forward → patch br-int → tunnel → compute2 → vm-b
```
Tất cả traffic L3 phải đi qua **network node** → bottleneck.

**ML2/OVS DVR**:
```
vm-a → tap → br-int → distributed router (mỗi compute) → tunnel → compute2 → vm-b
```
L3 phân tán xuống compute → giảm load network node.

**ML2/OVN**:
```
vm-a → tap → br-int → OVS flow (logical router pipeline) → tunnel → compute2 → vm-b
```
Tương tự DVR nhưng không cần agent rời, được tích hợp ngay trong OVS flow.

## 7. Tunable parameters

### 7.1 Encap

```bash
# Đổi tunnel type (mặc định geneve)
sudo ovs-vsctl set open . external-ids:ovn-encap-type=geneve   # hoặc vxlan, stt
```

### 7.2 OVN remote (failover)

```bash
# Multi-target cho HA RAFT cluster
sudo ovs-vsctl set open . external-ids:ovn-remote=\
"tcp:10.10.34.161:6642,tcp:10.10.34.162:6642,tcp:10.10.34.163:6642"
```

### 7.3 Monitoring all / specific

```bash
# Theo dõi đầy đủ NB (mặc định, tốn memory ở scale lớn)
sudo ovn-appctl -t ovn-controller debug/status

# Conditional monitoring (chỉ subscribe data liên quan đến chassis này)
sudo ovs-vsctl set open . external-ids:ovn-monitor-all=false
```

### 7.4 Inactivity probe

```bash
# Tránh disconnect do idle (mặc định 5s, có thể cần tăng)
sudo ovn-nbctl set NB_Global . options:northd_probe_interval=180000
```

## 8. Khi nào không dùng OVN?

OVN tốt cho hầu hết deployment, nhưng cân nhắc dùng ML2/OVS khi:

- Cần SR-IOV (OVN hỗ trợ nhưng hạn chế, OVS có driver chuyên).
- Cần hardware offload trên switch vật lý (HW VTEP) — OVN hỗ trợ nhưng setup phức tạp.
- Có driver vendor cũ chỉ support ML2/OVS mechanism.
- Deployment hiện tại đang chạy ML2/OVS và chưa muốn migrate.
