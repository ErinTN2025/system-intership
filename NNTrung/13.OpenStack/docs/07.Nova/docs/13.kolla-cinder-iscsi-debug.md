# Debug VM ERROR khi boot-from-volume — Cinder LVM + iSCSI trên Kolla-Ansible 2025.1

Tài liệu này ghi lại toàn bộ quá trình chẩn đoán một lỗi tạo instance boot từ
volume bị `ERROR` trên OpenStack (Kolla-Ansible, release 2025.1, backend Cinder
LVM). Cấu trúc theo dạng: **Triệu chứng → Cách điều tra → Nguyên nhân → Cách sửa**,
đi tuần tự theo từng lớp lỗi được bóc ra.

Bối cảnh hệ thống:
- Kolla-Ansible 2025.1, base distro Ubuntu Noble, các service trong container Docker.
- 1 control node (`control01`) + 2 compute (`compute01`, `compute02`).
- Cinder backend: LVM (`volume_group = cinder-volumes`), backend name `lvm-1`.
- `kolla_internal_vip_address = 192.168.70.122`, **tắt** HAProxy và keepalived.
- Tạo VM bằng: `openstack server create --flavor m1.tiny --volume lab-vol --network <id> --key-name lab-key --security-group <id> vm-test`

---

## Nguyên tắc điều tra chung (áp dụng xuyên suốt)

1. **Lệnh `server create` trả về bảng đẹp KHÔNG có nghĩa là thành công.** Đó chỉ là
   xác nhận "đã nhận yêu cầu". Build diễn ra bất đồng bộ; lỗi hiện ra sau ở
   `openstack server list` (cột Status = ERROR).

2. **Luôn lấy lý do lỗi thật từ trường `fault`:**
   ```bash
   openstack server show <id> -c fault
   ```

3. **Lần theo Request-ID xuyên các service.** Trường `fault` thường chỉ nói lớp
   ngoài cùng. Lấy request-id rồi grep trong log của service bị nghi:
   ```bash
   sudo grep -r "req-xxxxxxxx-...." /var/log/kolla/<service>/
   ```

4. **Bóc lỗi theo tầng.** Nova → Cinder API → Cinder Volume → target driver
   (tgtd/LIO) → kernel → compute initiator → xác thực. Mỗi lớp sửa xong thường
   lòi ra lớp tiếp theo. Việc *đổi thông báo lỗi* (ví dụ exit code đổi từ 24 sang
   20, hay message đổi) là dấu hiệu TIẾN BỘ, không phải tệ đi.

5. **Đọc kỹ exit code của `iscsiadm`** — mỗi số là một loại lỗi khác nhau (xem phần cuối).

## LỚP 1 — VM ERROR, `server create` không báo gì

**Triệu chứng:**
- `openstack server create ...` chạy ra bảng đầy đủ, có cả `adminPass`, status `BUILD`.
- Ngay sau đó `openstack server list` → Status = **ERROR**.

**Cách điều tra:**
```bash
openstack server show <instance-id> -c fault
```

**Phát hiện:** trường `fault` báo:
```
Build of instance <id> aborted: Unable to update the attachment. (HTTP 500)
... cinderclient.exceptions.ClientException: Unable to update the attachment. (HTTP 500)
```

**Nguyên nhân:** Nova gọi Cinder để attach volume (`attachment_update`), Cinder
trả về HTTP 500. Vì attach thất bại nên Nova abort cả quá trình build. **Lỗi gốc
nằm ở Cinder, không phải Nova.** Nova chỉ là nạn nhân.

**Hành động tiếp theo:** chuyển sang điều tra Cinder, lấy request-id từ fault.

## LỚP 2 — Cinder 500: target driver `tgtadm` (tgtd) không tồn tại

**Cách điều tra:**
```bash
# Kiểm tra volume vẫn ổn?
openstack volume show lab-vol -c status -c attachments -c "os-vol-host-attr:host"
# -> status: available, host: control01@lvm-1#lvm-1  (volume OK, không phải lỗi data)

# Kiểm tra service Cinder
openstack volume service list
# -> cinder-volume control01@lvm-1 : up/enabled (service sống)

# Lần theo request-id trong log Cinder
sudo grep -r "req-xxxx" /var/log/kolla/cinder/
sudo tail -n 200 /var/log/kolla/cinder/cinder-volume.log
```
> Lưu ý: cần `sudo` để đọc `/var/log/kolla/cinder/` (Permission denied nếu không).

**Phát hiện** (trong cinder-volume.log):
```
Command: tgtadm --lld iscsi --op show --mode target
Exit code: 107
Stderr: 'tgtadm: failed to send request hdr to tgt daemon, Transport endpoint is not connected'
```
Backtrace: `cinder/volume/targets/tgt.py` → `create_iscsi_target` → `tgtadm_show()`.

**Nguyên nhân:** Backend dùng target driver `tgtadm` (tgt). Khi attach volume,
Cinder phải tạo iSCSI target qua `tgtadm`, nhưng **daemon `tgtd` không chạy /
không tồn tại** ("Transport endpoint is not connected").

**Xác nhận tgtd hoàn toàn không có:**
```bash
sudo docker ps -a | grep tgtd          # -> trống, không có container tgtd
lsmod | grep -E "tgt|scsi_target"      # -> trống
sudo systemctl status tgt              # -> Unit tgt.service could not be found
```
→ Không container, không module, không service. tgtd không hề được deploy.

**Cách sửa — chuyển target driver sang LIO (`lioadm`):**
LIO là target driver mặc định và được khuyến nghị cho LVM trên Kolla (tgtd là cũ,
đang bị loại bỏ). LIO chạy ở kernel host (qua `targetcli`/`rtslib`), không cần
container riêng.

Trong Kolla, KHÔNG sửa trực tiếp file trong container (sẽ bị ghi đè). Cách đúng:
sửa qua `globals.yml` hoặc file override rồi reconfigure.

## LỚP 3 — Cấu hình `target_helper` bị Kolla ghi đè lại

**Triệu chứng:** Sửa `target_helper = lioadm` trong file override
(`/etc/kolla/config/cinder/cinder-volume.conf`) rồi reconfigure, nhưng mỗi lần
reconfigure Kolla **tự sinh lại** block backend với `target_helper = tgtadm`:
```ini
[lvm-1]
volume_group = cinder-volumes
volume_driver = cinder.volume.drivers.lvm.LVMVolumeDriver
volume_backend_name = lvm-1
target_helper = tgtadm        # <- bị Kolla đặt lại
target_protocol = iscsi
```

**Cách điều tra — luôn kiểm tra config THẬT bên trong container:**
```bash
sudo docker exec cinder_volume grep -E "target_helper|target_protocol|enabled_backends|\[lvm" /etc/cinder/cinder.conf
# -> target_helper = tgtadm   (xác nhận override chưa ăn)
```

**Nguyên nhân:** Khi `enable_cinder_backend_lvm: yes`, Kolla tự render block
`[lvm-1]` từ template, với `target_helper` lấy từ một biến trong `globals.yml`.
File override người dùng bị giá trị Kolla sinh ra đè lên.

**Cách sửa — đặt biến trong `globals.yml` (không sửa file override):**
```yaml
# /etc/kolla/globals.yml
cinder_target_helper: "lioadm"
```
Đây là biến Kolla dùng để render `target_helper`, nên đặt ở đây thì reconfigure bao
nhiêu lần cũng giữ `lioadm`.

> Cách xác minh đúng tên biến cho version đang dùng (tên biến có thể khác giữa các
> bản): tìm trong template role cinder của kolla-ansible:
> ```bash
> find / -path "*/roles/cinder/templates/cinder-volume.conf.j2" 2>/dev/null
> grep -rn "target_helper" <đường-dẫn-template>
> ```

**Reconfigure rồi xác nhận:**
```bash
kolla-ansible -i ~/multinode reconfigure --tags cinder
sudo docker exec cinder_volume grep target_helper /etc/cinder/cinder.conf
# -> target_helper = lioadm   (và không bị trả về tgtadm nữa)
```

> **Câu hỏi phụ — có cần xóa volume cũ không?** KHÔNG. Volume `lab-vol` vẫn
> `available`, dữ liệu nguyên vẹn. Lỗi nằm ở bước *export volume ra iSCSI*, không
> phải ở data. Chỉ cần xóa **VM lỗi** (`openstack server delete vm-test`) rồi tạo lại.
> Nếu tạo lại vẫn lỗi attach, kiểm tra attachment "mồ côi":
> ```bash
> openstack volume attachment list --volume lab-vol
> openstack volume attachment delete <attachment-id>
> ```

## LỚP 4 — Sau khi dùng LIO: compute không tìm thấy device

**Triệu chứng:** fault của VM đổi thành:
```
Exceeded maximum number of retries. Exhausted all hosts available for retrying build failures
```
Đây là message chung của nova-conductor, KHÔNG nói lý do thật. Lý do thật ở
`nova-compute.log` trên các compute node.

**Cách điều tra — đào log compute theo instance-id:**
```bash
sudo grep -i "<instance-id>" /var/log/kolla/nova/nova-compute.log | tail -n 40
```

**Phát hiện:**
```
os_brick.exception.VolumeDeviceNotFound: Volume device not found at .
```
Compute nhận được connection-info iSCSI, thử login để tìm block device, nhưng không
thấy device nào.

**Manh mối quan trọng:**
```bash
sudo docker ps -a | grep iscsid    # -> trống trên cả compute01 và compute02
```
→ Không có initiator iSCSI nào để login vào target.

## LỚP 5 — Container `iscsid` chưa được deploy / chưa lên trên compute

**Triệu chứng:** Đã đặt `enable_iscsid: "yes"` trong globals.yml, nhưng
`docker ps -a | grep iscsid` trên compute trả về trống → chưa có initiator iSCSI
để login vào target.

**Cách điều tra:**
```bash
sudo docker ps -a | grep iscsid    # -> trống ở thời điểm kiểm tra ban đầu
sudo grep -nA15 '^\[compute\]' ~/multinode   # đối chiếu compute có khai đúng group
# [compute]
# compute01 ansible_user=kolla ansible_become=true
# compute02 ansible_user=kolla ansible_become=true
```

**Nguyên nhân:** Trong Kolla 2025.1, **`iscsid` LÀ một container độc lập**
(`quay.io/openstack.kolla/iscsid:2025.1-ubuntu-noble`), chạy trên các compute node
để làm initiator iSCSI. Ở thời điểm kiểm tra ban đầu nó **chưa được deploy / chưa
kịp lên**, nên `docker ps` thấy trống — KHÔNG phải vì nó không tồn tại.

**Cách sửa — deploy để container iscsid lên trên compute:**
```bash
# deploy đầy đủ cho compute (không dùng tag iscsid lẻ)
kolla-ansible deploy -i ~/multinode --limit compute 2>&1 | tail -n 60
# nhìn PLAY RECAP: compute01/02 phải có changed > 0
```

Sau khi deploy, xác nhận container đã lên trên **cả compute01 và compute02**:
```bash
sudo docker ps | grep iscsid
# bf6d316c1d0c  quay.io/openstack.kolla/iscsid:2025.1-ubuntu-noble  ...  Up  iscsid
```

> Lưu ý: container iscsid có thể cần một lúc mới thực sự `Up` sau khi deploy. Nếu
> kiểm tra ngay sau lệnh thấy trống, đợi rồi check lại — nó sẽ tự lên và bền vững
> (tự khởi động lại khi reboot), khác với việc bật daemon thủ công trong nova_compute.

> **Workaround tạm thời** (chỉ khi container iscsid chưa lên và cần chữa cháy ngay):
> bật daemon iscsid trực tiếp trong nova_compute. Cách này MẤT khi container restart,
> chỉ dùng để test nhanh, không bền vững:
> ```bash
> sudo docker exec nova_compute iscsid
> sudo docker exec nova_compute pgrep -a iscsid
> ```
> Khi đã có container iscsid đúng nghĩa thì KHÔNG cần workaround này.

## LỚP 6 — Xác minh đường iSCSI từng mắt xích

Trước khi đi tiếp, cần phân định lỗi nằm ở đầu nào của đường iSCSI: phía **target
(control01)** hay phía **initiator (compute)**. Dùng test tách bạch.

### 6a. Kernel modules trên control01 (phía target) — đầy đủ:
```bash
lsmod | grep -E "target_core_mod|iscsi_target_mod"
# iscsi_target_mod, target_core_mod đều có
sudo modprobe target_core_mod iscsi_target_mod   # nạp cho chắc
```

### 6b. Container cinder_volume có truy cập configfs host không — CÓ:
```bash
sudo docker exec cinder_volume ls /sys/kernel/config/target/
# -> core  dbroot  iscsi  version   (thấy được configfs host)
sudo docker inspect cinder_volume --format '{{.HostConfig.Privileged}}'   # -> true
```
→ LIO trong container và host dùng chung; KHÔNG bị cô lập configfs.

### 6c. LIO chỉ mở portal 3260 KHI có volume được export (KIẾN THỨC QUAN TRỌNG):
```bash
sudo ss -tlnp | grep 3260      # khi rảnh -> TRỐNG (bình thường!)
sudo targetcli ls              # khi rảnh -> Targets: 0 (bình thường!)
```
→ Target/portal chỉ tồn tại trong ~10 giây lúc attach. Kiểm tra bằng tay lúc rảnh
sẽ luôn thấy trống — KHÔNG có nghĩa LIO hỏng. Phải canh đúng "10 giây vàng".

### 6d. Theo dõi target sống/chết theo thời gian thực:
```bash
# Terminal 1 (control01):
sudo watch -n0.5 'ss -tlnp | grep 3260; echo ---; targetcli ls /iscsi 2>/dev/null | head'
# Terminal 2: openstack server create ...
```
**Kết quả quan sát:** lúc tạo VM, cổng 3260 **CÓ** mở (LISTEN) trong vài giây →
**phía Cinder/LIO hoạt động tốt.**

### 6e. Log cinder-volume cho thấy Cinder KHÔNG lỗi:
```bash
sudo awk '/15:03:59/,/15:04:20/' /var/log/kolla/cinder/cinder-volume.log
```
```
Creating iscsi_target for volume: volume-...
attachment_update completed successfully.        <- Cinder THÀNH CÔNG
Terminate volume connection completed successfully.   <- Nova chủ động yêu cầu ngắt
Removing iscsi_target: ...                        <- Cinder dọn theo yêu cầu Nova
```
**Diễn giải:** Cinder tạo target OK. Compute login THẤT BẠI → Nova gọi ngược
"Terminate volume connection" → Cinder gỡ target. Vòng lặp create/remove là do
Nova retry. **=> Lỗi nằm ở phía COMPUTE initiator, không phải Cinder.**

## LỚP 7 — Mạng tới portal 3260 OK, nhưng login fail

**Cách điều tra — test kết nối ĐÚNG LÚC target đang sống:**
```bash
# Trên compute01, chạy ngay khi watch ở control01 thấy 3260 LISTEN:
nc -zv 192.168.70.122 3260
# -> Connection to 192.168.70.122 3260 port [tcp/iscsi-target] succeeded!
```
→ **Mạng OK.** Loại bỏ giả thuyết routing/firewall/VIP.

> Lưu ý: nếu `nc` báo `Connection refused` thì thường là test SAI THỜI ĐIỂM (target
> đã bị gỡ), chứ không phải mạng hỏng. `refused` ≠ `timeout`: refused = không ai
> listen (target đã chết); timeout = bị chặn đường mạng/firewall.

**Log compute lúc này:**
```
Failed to login iSCSI target iqn.2010-10.org.openstack:volume-... on portal
192.168.70.122:3260 (exit code 24)
iscsiadm: No active sessions.
```
→ TCP tới được (nc succeeded) nhưng phiên iSCSI không thiết lập được.

**Kiểm tra initiator daemon đã sẵn sàng chưa:**
```bash
# container iscsid trên compute phải Up (xem Lớp 5)
sudo docker ps | grep iscsid
# binary và session trong nova_compute
sudo docker exec nova_compute which iscsiadm iscsid  # -> binary CÓ tồn tại
sudo docker exec nova_compute iscsiadm -m session    # -> No active sessions (chạy được)
```

> **Bẫy đánh lạc hướng:** `sudo docker exec nova_compute cat /etc/iscsi/initiatorname.iscsi`
> báo `Permission denied`. Đây KHÔNG phải lỗi quyền file — `chmod` trên host vô ích
> vì path trong container khác với host (security context, mount riêng). Bỏ qua,
> không liên quan tới login iSCSI. Quyền chuẩn của file này là `600`, đừng để 755.

Nếu container iscsid đã `Up` (theo Lớp 5) thì initiator đã sẵn sàng. Làm trên **cả
compute01 và compute02** (VM có thể được schedule lên node bất kỳ). Sau đó tạo lại
VM để kiểm tra login iSCSI có qua không.

## LỚP 8 — Login bị từ chối: lỗi xác thực (CHAP / ACL) — lớp cuối

**Triệu chứng:** iscsid đã chạy, portal mở, mạng OK, nhưng login vẫn fail.
Exit code dao động:
- `exit code 24` = **iSCSI login failed due to authorization failure** (CHAP/ACL sai)
- `exit code 20` = could not connect to target / target không phản hồi đúng lúc
  (thường do timing/target reset)
- `exit code 21` = no objects found (đi kèm: discovery "No targets found")

**Cách điều tra — soi ACL/CHAP trên target lúc nó sống** (vì target chỉ tồn tại
~10s, dùng vòng lặp tự ghi thay vì canh tay):
```bash
# Terminal 1 (control01): ghi snapshot liên tục ra file
for i in $(seq 1 40); do
  echo "=== $i $(date +%T) ===" >> /tmp/lio.txt
  sudo targetcli ls /iscsi 2>/dev/null >> /tmp/lio.txt
  sleep 0.5
done
# Terminal 2: openstack server create ... (tạo VM trong lúc Terminal 1 chạy)

# Xem ACL/CHAP/IQN bắt được
grep -iE "acls|iqn|chap|auth|mapped|InitiatorName" /tmp/lio.txt | sort -u

# Kiểm tra CHAP trong cinder config
sudo docker exec cinder_volume grep -iE "chap|auth" /etc/cinder/cinder.conf
```

**Nguyên nhân (theo loại):**
- **CHAP authentication:** Cinder LIO mặc định có thể bật CHAP — target yêu cầu
  initiator xác thực bằng user/password CHAP do Cinder cấp qua `connection_info`.
  Nếu iscsid trên compute không áp được credential (thiếu cấu hình
  `node.session.auth` trong `/etc/iscsi/iscsid.conf` của nova_compute) → login bị
  từ chối, exit 24.
- **ACL initiator IQN:** LIO chỉ cho phép đúng initiator IQN mà Cinder ghi vào ACL.
  Nếu IQN compute gửi (`iqn.2016-04.com.open-iscsi:<...>` trong
  `/etc/iscsi/initiatorname.iscsi`) không khớp ACL trên target → bị chặn.

**Cách sửa — tắt CHAP cho backend LVM/LIO (chấp nhận được trong môi trường lab):**
File override `/etc/kolla/config/cinder/cinder-volume.conf`:
```ini
[lvm-1]
target_helper = lioadm
target_protocol = iscsi
use_chap_auth = false
```
> Tên biến có thể khác giữa các bản (`use_chap_auth`, `lvm_iscsi_auth`...). Xác minh:
> ```bash
> sudo docker exec cinder_volume grep -iE "chap_auth|use_chap" /etc/cinder/cinder.conf
> ```

Reconfigure và áp dụng:
```bash
kolla-ansible -i ~/multinode reconfigure --tags cinder
# hoặc nhanh (tạm thời): sudo docker restart cinder_volume
```

**Nếu tắt CHAP vẫn exit 24** → vấn đề ở ACL/IQN. Đối chiếu IQN trong `acls` của
target (từ `/tmp/lio.txt`) với IQN compute
(`cat /etc/iscsi/initiatorname.iscsi` trên compute). Nếu lệch → xử ở tầng ACL/
initiatorname.

## Cách né hoàn toàn iSCSI (nếu chỉ cần VM chạy / không bắt buộc boot-from-volume)

Để xác nhận cụm khỏe mạnh và có VM ACTIVE ngay, boot từ image (disk ephemeral,
không đụng Cinder/iSCSI):
```bash
openstack image list
openstack server create --flavor m1.tiny --image cirros \
  --network <net-id> --key-name lab-key --security-group <sg-id> vm-img
openstack server list    # -> ACTIVE
```
VM image lên ACTIVE chứng minh: Nova, Neutron, scheduler, libvirt, networking đều
OK → lỗi được cô lập tuyệt đối vào đường iSCSI (Cinder ↔ compute).

Biến thể vẫn dùng Cinder nhưng không boot-from-volume: tạo VM từ image rồi attach
volume làm disk phụ:
```bash
openstack server add volume vm-img lab-vol
openstack server show vm-img -c volumes_attached
```
(Vẫn đi qua đường iSCSI nên có thể vướng cùng lỗi CHAP/ACL, nhưng đáng thử.)

## Bảng tra cứu nhanh exit code `iscsiadm`

| Exit code | Ý nghĩa | Hướng nghi ngờ |
|-----------|---------|----------------|
| 8  | connection timed out | mạng/portal không tới được, firewall |
| 20 | could not connect to target | target chưa sẵn sàng/reset, timing |
| 21 | no objects found / no targets | discovery rỗng, target chưa tạo đúng lúc |
| 24 | login failed due to authorization failure | **CHAP / ACL sai** |

## Bảng tra cứu nhanh đường đi của lỗi (mental model)

```
openstack server list -> ERROR
  └─ server show -c fault                → "Unable to update attachment (500)"  [Nova là nạn nhân]
       └─ cinder-volume.log (theo req-id) → ProcessExecutionError tgtadm exit 107 [tgtd không có]
            └─ đổi target_helper = lioadm (qua globals.yml: cinder_target_helper)
                 └─ nova-compute.log       → VolumeDeviceNotFound                [compute thiếu initiator]
                      └─ deploy container iscsid lên compute (Kolla 2025.1 vẫn có container iscsid riêng)
                           └─ nc 3260 succeeded (mạng OK), nhưng login exit 24   [xác thực]
                                └─ tắt CHAP (use_chap_auth=false) / sửa ACL IQN  [lớp cuối]
```

## Các lệnh điều tra cốt lõi (gom lại)

```bash
# 1. Lý do lỗi của instance
openstack server show <id> -c fault

# 2. Trạng thái volume & service Cinder
openstack volume show lab-vol -c status -c attachments -c "os-vol-host-attr:host"
openstack volume service list

# 3. Lần theo request-id trong log (cần sudo)
sudo grep -r "req-xxxx" /var/log/kolla/cinder/
sudo tail -n 200 /var/log/kolla/cinder/cinder-volume.log

# 4. Lý do thật trên compute
sudo grep -i "<instance-id>" /var/log/kolla/nova/nova-compute.log | tail -n 40

# 5. Config THẬT bên trong container
sudo docker exec cinder_volume grep -E "target_helper|enabled_backends|\[lvm" /etc/cinder/cinder.conf

# 6. Trạng thái container
sudo docker ps -a | grep -E "cinder|iscsid|tgtd"

# 7. LIO target (canh đúng lúc target sống ~10s)
sudo targetcli ls
sudo ss -tlnp | grep 3260

# 8. Kernel modules
lsmod | grep -E "target_core_mod|iscsi_target_mod|iscsi_tcp|configfs"

# 9. Initiator trong nova_compute
sudo docker exec nova_compute pgrep -a iscsid
sudo docker exec nova_compute iscsiadm -m session

# 10. Test mạng tới portal (đúng lúc target sống)
nc -zv 192.168.70.122 3260
```

## Bài học rút ra

- `server create` ra bảng đẹp ≠ thành công; luôn xem `server list` rồi `fault`.
- Trường `fault` chỉ nói lớp ngoài; lần request-id xuống log từng service.
- Sửa config Cinder trong Kolla phải qua `globals.yml`/override + reconfigure, vì
  Kolla ghi đè file trong container mỗi lần reconfigure.
- LVM backend nên dùng `lioadm` (LIO), không phải `tgtadm` (tgtd cũ, dễ thiếu daemon).
- LIO chỉ mở portal 3260 trong ~10s lúc attach — kiểm tra phải canh đúng thời điểm.
- Ở Kolla 2025.1, `iscsid` vẫn là một container độc lập chạy trên compute; nếu
  `docker ps` thấy trống thì là do nó chưa được deploy/chưa lên, cần deploy lại
  (`kolla-ansible deploy --limit compute`) chứ không phải nó không tồn tại.
- `Connection refused` ≠ `timeout`: refused = target đã chết/chưa listen; timeout = mạng chặn.
- exit code 24 = lỗi xác thực (CHAP/ACL) — lớp sâu nhất của đường iSCSI.
- Đổi thông báo lỗi qua từng bước = đang tiến bộ, không phải tệ đi.
- Permission denied khi `docker exec cat` file trong container thường là bẫy đánh
  lạc hướng (path/context khác host), không phải nguyên nhân thật.
