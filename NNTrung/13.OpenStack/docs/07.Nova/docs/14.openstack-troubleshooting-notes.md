## 1. Boot from Volume — cột Image hiện "N/A"

**Triệu chứng:** `openstack server list` hiện `N/A (booted from volume)` ở cột Image.

**Đây KHÔNG phải lỗi.** Đó là hành vi bình thường:
- VM boot thẳng từ image (Glance) → cột Image hiện tên image.
- VM boot từ volume (Cinder) → image gốc nằm trong volume, không gắn `image_id` vào instance → cột Image hiện `N/A`.

**Nếu muốn VM hiển thị tên image:** boot thẳng từ image, không dùng `--boot-from-volume` / `--block-device`:

```bash
openstack server create \
  --image cirros \
  --flavor m1.tiny \
  --network internal-net \
  vm-test
```

## 2. Truy ngược image gốc của một volume

**Mục đích:** Biết một volume (hoặc VM boot-from-volume) được tạo từ image nào.

```bash
# Xem volume nào attach vào VM
openstack server show <VM> -c volumes_attached -c name

# Xem chi tiết volume — chú ý field volume_image_metadata
openstack volume show <VOLUME_ID>
```

**Đọc kết quả:**
- `bootable = True` + `status = in-use` → volume khởi động được, đang gắn vào VM (bình thường).
- `volume_image_metadata` → chứa `image_name`, `image_id`, `disk_format`... cho biết image gốc.
- `os-vol-host-attr:host` → backend Cinder (vd `control01@lvm-1#lvm-1` = LVM backend).
- `attachments` → server_id, device (vd `/dev/vda`), host.

## 3. Hiểu trạng thái instance

**3 trạng thái cần phân biệt** (lệnh xem):

```bash
openstack server show <VM> \
  -c status \
  -c "OS-EXT-STS:vm_state" \
  -c "OS-EXT-STS:task_state" \
  -c OS-EXT-SRV-ATTR:host
```

| Field | Ý nghĩa |
|-------|---------|
| `status` | Trạng thái tổng hợp hiển thị (ACTIVE / SHUTOFF / ERROR / VERIFY_RESIZE...) |
| `vm_state` | Trạng thái nội bộ của VM (active / stopped / error...) |
| `task_state` | Tiến trình đang chạy. **`None` = không có gì đang chạy / không treo** |
| `host` | Compute node đang chạy VM |

**Quy tắc vàng:**
- `task_state = None` → VM đứng yên ổn định, an toàn để thao tác.
- `task_state` khác None (vd `resize_migrating`, `migrating`) → đang có tiến trình, **Nova khóa các action thường** (start/stop bị chặn). Đợi nó xong, đừng can thiệp.

## 4. Lỗi confirmResize / migration confirm

**Triệu chứng:**
```
ConflictException: 409: Cannot 'confirmResize' instance ... while it is in vm_state stopped
```

**Nguyên nhân:** Chạy `openstack server migration confirm` khi **không còn migration nào để confirm** (migrate đã hoàn tất/đã tự dọn). Nova vẫn cố chạy action `confirmResize` nhưng VM không ở `VERIFY_RESIZE` → ném lỗi 409.

**Cách phân biệt và xử lý:**

```bash
openstack server show <VM> -c status -c "OS-EXT-STS:task_state"
```

- `status = VERIFY_RESIZE` → cần confirm hoặc revert thật:
  ```bash
  openstack server resize confirm <VM>   # giữ kết quả mới
  openstack server resize revert <VM>    # quay về cũ
  ```
- `status = SHUTOFF/ACTIVE` và `task_state = None` → **không cần confirm gì nữa**, migrate đã xong. Chỉ cần:
  ```bash
  openstack server start <VM>
  ```

**Bài học:** Đừng gõ lại `migration confirm` khi đã xong — nó sẽ luôn báo 409 đó.

## 5. Security group trùng tên "default"

**Triệu chứng:** Có nhiều security group cùng tên `default` → lệnh dùng tên bị nhập nhằng.

**Lý do:** Mỗi **project** có một SG `default` riêng. Tên trùng nhưng khác Project.

**Cách xử lý — luôn dùng ID, không dùng tên:**

```bash
# Liệt kê, phân biệt bằng cột Project
openstack security group list

# Xác định chính xác VM đang dùng SG nào
openstack server show <VM> -c security_groups
openstack port list --server <VM> -c id -c "Security Groups"
```

Chọn SG có **Project khớp với project của VM** (tenant_id của VM, thấy trong volume show / server show). Rồi thêm rule bằng ID:

```bash
openstack security group rule create --proto icmp <SG_ID>
openstack security group rule create --proto tcp --dst-port 22 <SG_ID>
```

**Mẹo:** Khi tên bị trùng, mọi lệnh openstack đều nhận ID thay tên — ID luôn duy nhất.

## 6. Truy cập VM khi mạng hỏng

**Bối cảnh:** SSH không vào được, ping packet loss 100%.

### Phân biệt 2 đường vào

| | Đường đi | Phụ thuộc mạng VM? |
|---|---------|---------------------|
| **SSH** | Qua IP VM → security group → network | CÓ. Mạng hỏng = SSH chết. |
| **Console VNC** | Qua hypervisor (nova-compute), nối thẳng màn hình ảo | KHÔNG. Như cắm màn hình trực tiếp. |

→ **Mạng hỏng thì dùng VNC**, không dùng SSH.

### Cách dùng Console VNC (ĐÚNG)

```bash
openstack console url show <VM>
```

**QUAN TRỌNG:**
- URL này phải **mở bằng TRÌNH DUYỆT (browser)**, KHÔNG phải `curl` trong terminal.
- `curl` chỉ tải về mã HTML/JavaScript — JS phải chạy trong browser mới vẽ ra màn hình console.
- Vào được màn hình → click vào đó → gõ login **trực tiếp** (cirros / `gocubsgo`). Không qua SSH, không qua mạng VM.
- **Token VNC hết hạn nhanh (vài phút)** — lấy token mới ngay trước khi mở.

### Nếu browser không tới được IP control node (vd 192.168.70.122)

Tạo SSH tunnel từ **laptop** (máy có browser):

```bash
ssh -L 6080:192.168.70.122:6080 kolla@<IP_control_node>
```

Rồi mở trên browser laptop:
```
http://localhost:6080/vnc_lite.html?path=%3Ftoken%3D<TOKEN_MỚI>
```

### Chẩn đoán mạng từ trong VM (qua console)

```bash
ip addr            # VM nhận đúng IP chưa?
ip route           # route mặc định trỏ đâu?
ping -c 4 10.10.10.1   # ping gateway
```

- Ping gateway **được** → mạng nội bộ ổn, lỗi ở đường ngoài vào (floating IP / SG / external net).
- Ping gateway **100% loss** → lỗi tầng network/SG, sửa từ gốc.

**Cảnh báo cấu hình lạ:** VM có **2 IP lệch subnet** trên cùng một network (vd `10.0.2.200` + `10.10.10.97` cùng `internal-net`) là bất thường → dễ gây rối routing → packet loss. Kiểm tra:

```bash
openstack port list --server <VM>
openstack subnet list --network <NET>
```

Đừng detach port vội khi chưa xem output — gỡ nhầm port mang IP chính càng rối.

## 7. Live Migration — quy trình lab chuẩn

**Điều kiện bắt buộc:**
- VM phải `ACTIVE`.
- Phải có **ít nhất 2 compute node** (cần host đích để chuyển).

```bash
# Kiểm tra số compute node
openstack compute service list --service nova-compute
openstack hypervisor list
```

### Quy trình lab (quan sát mất gói khi migrate)

**Bước 1 — Vào VM qua console**, chạy ping liên tục sang VM khác hoặc gateway:
```bash
ping 10.10.10.67     # ping sang VM còn lại
# hoặc
ping 10.10.10.1      # ping gateway
```

**Bước 2 — Mở terminal khác trên control node, chạy migrate:**
```bash
openstack server show <VM> -c OS-EXT-SRV-ATTR:host   # xem host hiện tại
openstack server migrate --live-migration --wait <VM>
```

**Bước 3 — Theo dõi:**
```bash
openstack server migration list   # kỳ vọng status = completed
openstack server show <VM> -c status -c OS-EXT-SRV-ATTR:host
```

**Kết quả tốt:** ping chỉ rớt 0–1 gói, VM giữ `ACTIVE` suốt, `host` đổi sang node khác.

**Lưu ý:** Mỗi lần migrate fail tạo thêm record trong `migration list`. Record fail cũ không ảnh hưởng lần migrate mới — không cần dọn.

## 8. Lỗi Live Migrate: Postcopy / Userfaultfd not available

**Triệu chứng** (trong `/var/log/kolla/nova/nova-compute.log`):
```
Live Migration failure: internal error: unable to execute QEMU command
'migrate-set-capabilities': Postcopy is not supported:
Userfaultfd not available: Operation not permitted
```

`migration list` → `status = failed`, VM quay về node nguồn.

**Nguyên nhân:** Nova bật chế độ **post-copy** (`live_migration_permit_post_copy = true`). Post-copy cần syscall kernel **userfaultfd**, nhưng QEMU chạy trong Docker container (Kolla) bị **seccomp chặn** syscall này → "Operation not permitted".

### Cách tìm log lỗi

Log nằm trên TỪNG compute node tại `/var/log/kolla/nova/`:

```bash
# SSH vào từng compute, grep log
ssh compute01
sudo grep -i "live.migration\|migrat\|error" /var/log/kolla/nova/nova-compute.log | tail -50

ssh compute02   # node đích — lỗi thường ở đây
sudo grep -i "live.migration\|migrat\|error" /var/log/kolla/nova/nova-compute.log | tail -50
```

> Lưu ý phân biệt log cũ (ngày khác) với log của lần migrate hiện tại — đừng nhầm lỗi AMQP/MessagingTimeout cũ với lỗi migrate.

### Hướng 1 — Tắt post-copy (KHUYẾN NGHỊ cho lab)

Tạo override config trên node deploy:

```bash
sudo mkdir -p /etc/kolla/config/nova
sudo tee /etc/kolla/config/nova/nova-compute.conf > /dev/null <<'EOF'
[libvirt]
live_migration_permit_post_copy = false
live_migration_permit_auto_converge = true
EOF
```

> `auto_converge = true` thay thế post-copy: bóp CPU của VM để migrate hội tụ, không cần userfaultfd.

Reconfigure nova:
```bash
cd <thư_mục_kolla>   # nơi có globals.yml
kolla-ansible -i <inventory> reconfigure -t nova
```

**Cách nhanh kiểu lab** (sửa thẳng container, mất khi redeploy) — làm trên **CẢ HAI** compute:
```bash
sudo docker exec -it nova_compute bash
# sửa /etc/nova/nova.conf, mục [libvirt]:
#   live_migration_permit_post_copy = false
exit
sudo docker restart nova_compute
```

### Hướng 2 — Cho phép userfaultfd (production, giữ post-copy)

```bash
# trên cả 2 compute host
echo 'vm.unprivileged_userfaultfd = 1' | sudo tee /etc/sysctl.d/60-userfaultfd.conf
sudo sysctl --system
```
(Còn cần nới seccomp profile của Docker — phức tạp, lab nên dùng Hướng 1.)

**Sau khi sửa:** restart `nova_compute` trên cả 2 node rồi migrate lại.

## 9. VM kẹt status ERROR sau migrate

**Triệu chứng:** Sau live migrate, `status = ERROR`, `vm_state = error` nhưng `host` đã đổi sang node đích.

**ĐỪNG HOẢNG.** `ERROR` thường chỉ là **cờ Nova đánh dấu nhầm**, không có nghĩa VM hỏng vật lý. Volume trên Cinder vẫn an toàn.

### Bước 1 — Xem lý do lỗi

```bash
openstack server show <VM> -c fault -c status -c "OS-EXT-STS:vm_state" -c "OS-EXT-STS:task_state"
```

### Bước 2 — Kiểm tra domain libvirt thật sự còn sống không (trên node đích)

```bash
sudo docker exec nova_libvirt virsh list --all
```

Tìm `instance-0000xxxx`:
- State = `running` → **VM vẫn sống nguyên**, chỉ là Nova ghi nhầm trạng thái. (Kịch bản phổ biến nhất)
- State = `shut off` / không thấy → cần khởi động lại.

### Bước 3 — Reset cờ trạng thái

**Nếu domain `running`:**
```bash
openstack server set --state active <VM>
openstack server show <VM> -c status -c "OS-EXT-STS:vm_state" -c OS-EXT-SRV-ATTR:host
```
→ Kỳ vọng `ACTIVE` / `active` / host = node đích. **Migrate thành công!**

**Nếu domain không chạy:**
```bash
nova reset-state --active <INSTANCE_ID>
openstack server stop <VM>
openstack server start <VM>
```

### Bước 4 — Xác nhận VM chạy thật

```bash
openstack console url show <VM>   # mở browser, login kiểm tra
```

> **Lưu ý:** `task_state = None` + domain `running` = an toàn tuyệt đối để `set --state active`. VM và volume không mất.

## 10. Bộ lệnh chẩn đoán nhanh

```bash
# === TRẠNG THÁI VM ===
openstack server list
openstack server show <VM> -c status -c "OS-EXT-STS:vm_state" \
  -c "OS-EXT-STS:task_state" -c OS-EXT-SRV-ATTR:host -c fault
openstack server show <VM> -c addresses -c security_groups -c volumes_attached

# === VOLUME ===
openstack volume show <VOL_ID>          # xem volume_image_metadata, attachments

# === NETWORK / SECURITY GROUP ===
openstack security group list           # phân biệt bằng Project
openstack security group rule list <SG_ID>
openstack port list --server <VM>
openstack subnet list --network <NET>
openstack floating ip list

# === CONSOLE (vào VM khi mạng hỏng) ===
openstack console url show <VM>         # MỞ BẰNG BROWSER, không curl

# === COMPUTE / HYPERVISOR ===
openstack compute service list --service nova-compute
openstack hypervisor list

# === MIGRATION ===
openstack server migrate --live-migration --wait <VM>
openstack server migration list
openstack server event list <VM>
openstack server event show <VM> <REQUEST_ID>

# === RESET TRẠNG THÁI (khi ERROR) ===
openstack server set --state active <VM>
nova reset-state --active <INSTANCE_ID>

# === TRÊN COMPUTE NODE (Kolla) ===
sudo docker exec nova_libvirt virsh list --all
sudo grep -i "migrat\|error" /var/log/kolla/nova/nova-compute.log | tail -50
sudo docker restart nova_compute

# === RECONFIGURE KOLLA ===
# Override config: /etc/kolla/config/nova/nova-compute.conf
kolla-ansible -i <inventory> reconfigure -t nova
```

## Ghi nhớ cốt lõi

1. **`N/A booted from volume` không phải lỗi** — chỉ là VM boot từ Cinder.
2. **`task_state = None` = an toàn thao tác**; khác None = Nova đang khóa, đợi.
3. **Lỗi 409 confirmResize** = không còn gì để confirm, chỉ cần `server start`.
4. **Trùng tên SG** = dùng ID, phân biệt bằng Project.
5. **Mạng hỏng** = vào bằng Console VNC (browser), không SSH, không curl.
6. **Live migrate cần ≥ 2 compute node.**
7. **Postcopy/Userfaultfd lỗi** = tắt post-copy + bật auto_converge (lab).
8. **Status ERROR sau migrate** = kiểm tra `virsh list`; nếu domain `running` thì chỉ cần `set --state active`. VM/volume không mất.
