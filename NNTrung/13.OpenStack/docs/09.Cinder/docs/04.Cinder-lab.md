# Lab thực hành Cinder

## Mục tiêu

1. Tạo và quản lý volume cơ bản (create / attach / format / detach).
2. Mở rộng (extend) volume khi hết dung lượng.
3. Snapshot + restore (rollback nhanh).
4. Backup volume sang Swift/file-backend + restore vào volume mới.
5. Clone volume.
6. Boot from Volume (BfV) – tạo VM với root disk là Cinder volume.
7. Volume Type + QoS – tạo SSD tier có giới hạn IOPS.
8. Encryption – tạo volume mã hóa LUKS.
9. Multi-attach – gắn 1 volume vào nhiều VM.
10. Transfer volume giữa 2 project.

## Mô hình lab

```
controller (10.10.34.161)
   ├── cinder-api, cinder-scheduler
   ├── nova-api, neutron-server, glance-api, keystone
   └── MariaDB, RabbitMQ, etcd

storage01  (10.10.34.171)  ── cinder-volume + cinder-backup (LVM/Ceph)
compute01  (10.10.34.162)  ── nova-compute
```

> Tất cả lệnh chạy trên **controller** trừ khi có ghi `[storage01]` hoặc `[compute01]`.

## Phần 0 – Chuẩn bị môi trường

```bash
# ==== Trên CONTROLLER ====
source /root/admin-openrc

# 1. Kiểm tra Cinder service UP
openstack volume service list

# 2. Kiểm tra backend pool còn dung lượng
openstack volume backend pool list --long

# 3. Tạo image + flavor + network nếu chưa có (xem lab Nova nếu cần)
openstack image list | grep -i cirros
openstack flavor list | grep m1.tiny
openstack network show private >/dev/null 2>&1

# 4. Tạo 1 VM để các bài lab attach vào
openstack server create vm-lab \
  --image cirros \
  --flavor m1.tiny \
  --network private \
  --wait

# 5. Lấy IP của VM
VM_IP=$(openstack server show vm-lab -f value -c addresses | awk -F'=' '{print $2}')
echo "VM IP: $VM_IP"
```

## Phần 1 – Tạo, Attach, Format Volume

```bash
# ==== Trên CONTROLLER ====
# 1. Tạo volume rỗng 5GB
openstack volume create --size 5 --description "Lab basic volume" vol-basic

# 2. Đợi đến status = available
watch -n 2 "openstack volume show vol-basic -f value -c status"
# Nhấn Ctrl+C khi thấy "available"

# 3. Attach vào VM
openstack server add volume vm-lab vol-basic

# 4. Verify status = in-use
openstack volume show vol-basic -f value -c status -c attachments

# 5. SSH vào VM (qua console hoặc floating IP)
# Cirros: user cirros / pass gocubsgo (cirros >= 0.6) hoặc cubswin:)
ssh cirros@$VM_IP

# === Bên trong VM ===
sudo fdisk -l | grep -i "Disk /dev/vd"
# Phải thấy /dev/vdb (volume mới gắn)

# Format ext4
sudo mkfs.ext4 /dev/vdb

# Mount + ghi data test
sudo mkdir /mnt/cinder
sudo mount /dev/vdb /mnt/cinder
echo "Hello Cinder $(date)" | sudo tee /mnt/cinder/test.txt
sudo umount /mnt/cinder
exit
# === Thoát VM ===

# 6. Detach
openstack server remove volume vm-lab vol-basic
openstack volume show vol-basic -f value -c status
# Phải là "available"
```

## Phần 2 – Extend Volume (mở rộng)

```bash
# ==== Trên CONTROLLER ====
# 1. Volume phải ở status available HOẶC in-use (Cinder hỗ trợ online extend từ Pike)
openstack volume show vol-basic -f value -c size

# 2. Extend từ 5GB → 10GB
openstack volume set --size 10 vol-basic

# 3. Verify
openstack volume show vol-basic -f value -c size

# 4. Nếu volume đang attached, vào VM mở rộng filesystem
openstack server add volume vm-lab vol-basic
ssh cirros@$VM_IP

# === Bên trong VM ===
lsblk                              # /dev/vdb hiện size mới
sudo mount /dev/vdb /mnt/cinder
sudo resize2fs /dev/vdb            # ext4 online resize
df -h /mnt/cinder                  # Phải thấy size mới
sudo umount /mnt/cinder
exit

# 5. Detach lại
openstack server remove volume vm-lab vol-basic
```

## Phần 3 – Snapshot + Rollback

```bash
# ==== Trên CONTROLLER ====
# 1. Tạo snapshot ban đầu (volume available)
openstack volume snapshot create --volume vol-basic snap-v1

# 2. Đợi đến status = available
watch -n 2 "openstack volume snapshot show snap-v1 -f value -c status"

# 3. Mô phỏng "thay đổi data" trong volume
openstack server add volume vm-lab vol-basic
ssh cirros@$VM_IP <<'EOF'
sudo mount /dev/vdb /mnt
echo "MODIFIED DATA $(date)" | sudo tee /mnt/test.txt
sudo umount /mnt
EOF
openstack server remove volume vm-lab vol-basic

# 4. Tạo volume mới từ snapshot (rollback)
openstack volume create --size 10 --snapshot snap-v1 vol-restored

# 5. Attach vol-restored và verify data ban đầu vẫn còn
openstack server add volume vm-lab vol-restored
ssh cirros@$VM_IP <<'EOF'
sudo mount /dev/vdb /mnt
cat /mnt/test.txt
sudo umount /mnt
EOF
# Phải thấy "Hello Cinder ..." chứ không phải "MODIFIED DATA"

openstack server remove volume vm-lab vol-restored
```

**Snapshot khi volume đang in-use (force)**:

```bash
openstack server add volume vm-lab vol-basic
openstack volume snapshot create --volume vol-basic --force snap-live
openstack volume snapshot list
openstack server remove volume vm-lab vol-basic
```

## Phần 4 – Backup + Restore

> Yêu cầu `cinder-backup` đang chạy. Backend backup thường là Swift, Ceph, hoặc NFS.

```bash
# ==== Trên CONTROLLER ====
# 1. Kiểm tra cinder-backup
openstack volume service list --service cinder-backup

# 2. Volume phải ở status available
openstack volume show vol-basic -f value -c status

# 3. Tạo full backup
openstack volume backup create --name backup-v1 vol-basic

# 4. Theo dõi
watch -n 5 "openstack volume backup show backup-v1 -f value -c status -c size"
# creating → available

# 5. Tạo incremental backup (nhanh hơn, chỉ lưu phần thay đổi)
openstack volume backup create --name backup-v2 --incremental vol-basic

# 6. Liệt kê tất cả backup
openstack volume backup list

# 7. Restore vào volume MỚI
openstack volume backup restore backup-v1 vol-from-backup

# 8. Verify
openstack volume show vol-from-backup
openstack server add volume vm-lab vol-from-backup
ssh cirros@$VM_IP "sudo mount /dev/vdb /mnt && cat /mnt/test.txt && sudo umount /mnt"
openstack server remove volume vm-lab vol-from-backup
```

**Backup volume đang in-use**:

```bash
openstack server add volume vm-lab vol-basic
openstack volume backup create --name backup-live --force vol-basic
openstack volume backup list
openstack server remove volume vm-lab vol-basic
```

## Phần 6 – Boot from Volume (BfV)

```bash
# ==== Trên CONTROLLER ====
# 1. Tạo volume bootable từ image
openstack volume create --size 5 --image cirros --bootable vol-boot

# 2. Đợi available
watch -n 3 "openstack volume show vol-boot -f value -c status -c bootable"

# 3. Tạo VM dùng vol-boot làm root disk
openstack server create vm-bfv \
  --flavor m1.tiny \
  --network private \
  --volume vol-boot \
  --wait

# 4. Verify: VM ACTIVE, root disk = vol-boot
openstack server show vm-bfv -f value -c status -c volumes_attached

# 5. Test xóa VM nhưng giữ volume
openstack server delete vm-bfv --wait
openstack volume list | grep vol-boot
# Volume vẫn còn → data an toàn

# 6. Tạo lại VM từ vol-boot đó
openstack server create vm-bfv-v2 \
  --flavor m1.tiny \
  --network private \
  --volume vol-boot \
  --wait
```

## Phần 7 – Volume Type + QoS (giới hạn IOPS)

```bash
# ==== Trên CONTROLLER ====
# 1. Tạo Volume Type "ssd"
openstack volume type create ssd
openstack volume type set ssd --property volume_backend_name=LVM_iSCSI
# (Thay LVM_iSCSI bằng tên backend thực tế: openstack volume backend pool list)

# 2. Tạo QoS giới hạn 500 IOPS, 50 MB/s
openstack volume qos create gold-qos --consumer front-end \
  --property total_iops_sec=500 \
  --property total_bytes_sec=52428800

# 3. Gắn QoS vào type
openstack volume qos associate gold-qos ssd

# 4. Tạo volume với type ssd
openstack volume create --size 5 --type ssd vol-ssd

# 5. Attach + benchmark
openstack server add volume vm-lab vol-ssd
ssh cirros@$VM_IP <<'EOF'
# Cirros không có fio, dùng dd cho đơn giản
sudo dd if=/dev/zero of=/dev/vdb bs=1M count=100 oflag=direct
# Speed phải bị giới hạn ở ~50 MB/s
EOF
openstack server remove volume vm-lab vol-ssd

# 6. So sánh với volume thường (không có QoS) - tốc độ cao hơn
openstack volume create --size 5 vol-no-qos
openstack server add volume vm-lab vol-no-qos
ssh cirros@$VM_IP "sudo dd if=/dev/zero of=/dev/vdb bs=1M count=100 oflag=direct"
openstack server remove volume vm-lab vol-no-qos
```

## Phần 8 – Encryption (LUKS)

```bash
# ==== Trên CONTROLLER ====
# 1. Tạo volume type có encryption
openstack volume type create encrypted-type

openstack volume type set encrypted-type \
  --encryption-provider luks \
  --encryption-cipher aes-xts-plain64 \
  --encryption-key-size 256 \
  --encryption-control-location front-end

# 2. Verify
openstack volume type show encrypted-type

# 3. Tạo volume mã hóa
openstack volume create --size 2 --type encrypted-type vol-secret

# 4. Attach + verify từ trong VM
openstack server add volume vm-lab vol-secret
ssh cirros@$VM_IP <<'EOF'
# Volume sẽ thấy là /dev/vdb đã được decrypt sẵn bởi Nova (qos os-brick + LUKS)
sudo fdisk -l /dev/vdb
sudo mkfs.ext4 /dev/vdb
sudo mount /dev/vdb /mnt
echo "secret data" | sudo tee /mnt/secret.txt
sudo umount /mnt
EOF
openstack server remove volume vm-lab vol-secret

# Note: Trên host vật lý, raw device hiển thị là dữ liệu đã mã hóa (random bytes)
```

## Phần 9 – Multi-attach (gắn 1 volume vào nhiều VM)

> Yêu cầu backend hỗ trợ multiattach (Ceph RBD, NFS, một số SAN). LVM mặc định KHÔNG hỗ trợ.

```bash
# ==== Trên CONTROLLER ====
# 1. Tạo type hỗ trợ multiattach
openstack volume type create multiattach-type
openstack volume type set multiattach-type --property multiattach="<is> True"
openstack volume type set multiattach-type --property volume_backend_name=CEPH_RBD

# 2. Tạo volume
openstack volume create --size 5 --type multiattach-type shared-vol

# 3. Tạo VM thứ 2
openstack server create vm-lab-2 --image cirros --flavor m1.tiny --network private --wait

# 4. Attach vào 2 VM
openstack server add volume vm-lab shared-vol
openstack server add volume vm-lab-2 shared-vol

# 5. Verify
openstack volume show shared-vol -c attachments

# 6. Cảnh báo: KHÔNG dùng ext4/xfs trên multi-attach (sẽ corrupt data!)
# Phải dùng cluster filesystem: OCFS2, GFS2, hoặc app-level coordination

# 7. Cleanup
openstack server remove volume vm-lab shared-vol
openstack server remove volume vm-lab-2 shared-vol
openstack volume delete shared-vol
openstack server delete vm-lab-2
```

## Phần 10 – Transfer Volume giữa 2 project

```bash
# ==== Bước 1: Owner cũ tạo request ====
source /root/admin-openrc           # Đang ở project admin
openstack volume create --size 2 vol-transfer

# Tạo transfer request
openstack volume transfer request create vol-transfer
# Output:
#   id          : xxxxxxxx-xxxx-...
#   auth_key    : abcd1234...

# Ghi lại ID + auth_key
TRANSFER_ID=<id-từ-output>
AUTH_KEY=<auth_key-từ-output>

# ==== Bước 2: Owner mới accept (project khác) ====
source /root/demo-openrc            # User trong project khác
openstack volume transfer request accept --auth-key $AUTH_KEY $TRANSFER_ID

# ==== Bước 3: Verify ====
openstack volume list
# vol-transfer giờ thuộc project demo

# Cleanup
openstack volume delete vol-transfer
```

## Phần 11 – Cleanup toàn bộ lab

```bash
# ==== Trên CONTROLLER ====
source /root/admin-openrc

# Detach mọi volume khỏi VM
for vol in $(openstack volume list -f value -c ID -c Status | grep in-use | awk '{print $1}'); do
  vm=$(openstack volume show $vol -f json -c attachments | jq -r '.attachments[0].server_id')
  openstack server remove volume $vm $vol || true
done

# Xóa VM
openstack server delete vm-lab vm-bfv vm-bfv-v2 --wait 2>/dev/null

# Xóa snapshot
for s in $(openstack volume snapshot list -f value -c ID); do
  openstack volume snapshot delete $s
done

# Xóa backup
for b in $(openstack volume backup list -f value -c ID); do
  openstack volume backup delete $b
done

# Xóa volume
for v in $(openstack volume list -f value -c ID); do
  openstack volume delete --force $v
done

# Xóa QoS + Type
openstack volume qos disassociate gold-qos --volume-type ssd 2>/dev/null
openstack volume qos delete gold-qos 2>/dev/null
openstack volume type delete ssd encrypted-type multiattach-type 2>/dev/null
```

## Phần 12 – Troubleshooting nhanh

| Triệu chứng | Nguyên nhân thường gặp | Cách fix |
|---|---|---|
| Volume stuck ở `creating` | Scheduler không tìm thấy backend phù hợp / driver lỗi | `tail /var/log/cinder/scheduler.log`, `openstack volume backend pool list` |
| Volume stuck ở `error` | Backend hết dung lượng / driver fail | `openstack volume show <v> -c fault`, `cinder-manage volume update_host` |
| Attach fail: `device-busy` | Tap còn dangling từ lần trước | Restart `nova-compute` trên host VM |
| Multi-attach fail | Type chưa set `multiattach="<is> True"` hoặc backend không support | Kiểm tra `volume type show` |
| Backup stuck | Backup backend (Swift/Ceph) unreachable | `tail /var/log/cinder/backup.log` |
| Snapshot fail "in-use" | Quên `--force` khi snapshot volume đang attached | Thêm `--force` |
| Extend không thấy trong VM | Cần `resize2fs` (ext4) hoặc `xfs_growfs` (xfs) | Vào VM chạy `lsblk` rồi resize FS |
| `NoValidHost` khi tạo | Backend full hoặc disable | `openstack volume service list`, check capacity |

**Reset volume bị stuck**:
```bash
# Admin only — chỉ dùng khi chắc chắn volume không còn task gì đang chạy
openstack volume set --state available <volume>

# Hoặc với cinder client:
cinder reset-state --state available <volume>
cinder reset-state --attach-status detached <volume>
```

**Log quan trọng**:
```bash
# Controller
tail -f /var/log/cinder/api.log
tail -f /var/log/cinder/scheduler.log

# Storage node
tail -f /var/log/cinder/volume.log
tail -f /var/log/cinder/backup.log

# Trace 1 request-id
grep "req-xxxx" /var/log/cinder/*.log
```
