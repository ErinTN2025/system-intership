# File log Cinder & Troubleshooting

## 1. Vị trí file log

Cinder dùng thư viện **oslo.log** (như Nova) để quản lý logging.

- **Non-container** (cài truyền thống):
  `/var/log/cinder/`

- **Containerized** (Kolla, TripleO, Helm):
  `/var/log/containers/cinder/`

## 2. Danh sách file log chính

| File log | Service tương ứng | Chạy trên node | Nội dung chính |
|---|---|---|---|
| **cinder-api.log** | cinder-api | Controller | Tất cả REST API request (create/delete volume, attach, snapshot…), authentication, quota check, policy. **Trace request-id đầu tiên ở đây.** |
| **cinder-scheduler.log** | cinder-scheduler | Controller | Quyết định backend nào tạo volume, filter/weigher, lý do loại backend. **Rất hữu ích khi volume bị `error` ngay khi tạo.** |
| **cinder-volume.log** | cinder-volume | Storage node | Toàn bộ vòng đời volume: create/delete/extend/snapshot/clone, driver-specific errors (LVM/Ceph/iSCSI). **File quan trọng nhất khi debug volume.** |
| **cinder-backup.log** | cinder-backup | Storage / dedicated node | Backup/restore, kết nối Swift/Ceph repo. |
| **cinder-manage.log** | cinder-manage (CLI) | Bất kỳ node admin | Output của lệnh `cinder-manage db sync`, `service list`, etc. |

### 2.1 Log của tools liên quan (thường cần xem chung)

| File | Mục đích |
|---|---|
| `/var/log/nova/nova-compute.log` | Lỗi attach/detach phía compute (os-brick, iSCSI initiator) |
| `/var/log/libvirt/libvirtd.log` | Lỗi gắn device vào VM (XML, SELinux) |
| `/var/log/syslog` hoặc `/var/log/messages` | Lỗi iSCSI initiator, multipath |
| `/var/log/ceph/ceph-*.log` | Lỗi Ceph cluster (nếu dùng RBD backend) |
| `/var/log/targetcli/` | iSCSI target config (LVM backend) |

## 3. Cấu hình logging

Trong `/etc/cinder/cinder.conf` phần `[DEFAULT]`:

```ini
[DEFAULT]
log_dir = /var/log/cinder
# log_file = cinder.log              # Không khuyến khích — gộp 1 file
debug = False                         # True chỉ khi debug
use_json = False                      # True nếu dùng ELK/Loki
use_journal = False                   # True để gửi vào journald
log_rotation_type = size
max_logfile_size_mb = 100
max_logfile_count = 10

default_log_levels = cinder=INFO,oslo=WARNING,sqlalchemy=WARNING,suds=INFO,eventlet.wsgi.server=WARNING

logging_context_format_string = %(asctime)s.%(msecs)03d %(process)d %(levelname)s %(name)s [%(request_id)s %(user_identity)s] %(instance)s%(message)s
logging_default_format_string = %(asctime)s.%(msecs)03d %(process)d %(levelname)s %(name)s [-] %(instance)s%(message)s
```

**Bật debug tạm thời** (không cần restart toàn bộ):

```bash
# Trên node bạn muốn debug
sudo crudini --set /etc/cinder/cinder.conf DEFAULT debug True

# Restart service tương ứng (chỉ service đó)
sudo systemctl restart openstack-cinder-volume     # Trên storage node
# Hoặc
sudo systemctl restart openstack-cinder-api openstack-cinder-scheduler   # Controller

# Sau khi debug xong, tắt lại
sudo crudini --set /etc/cinder/cinder.conf DEFAULT debug False
sudo systemctl restart openstack-cinder-volume
```

## 4. Cách trace 1 request từ đầu đến cuối

Mỗi API call có một **request-id** dạng `req-xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`.

### 4.1 Lấy request-id

```bash
# Cách 1: Bật debug ở client
openstack --debug volume create --size 1 test-vol 2>&1 | grep -i "x-openstack-request-id"

# Cách 2: Xem trong log API
tail -f /var/log/cinder/cinder-api.log | grep "POST /v3"
```

### 4.2 Trace toàn bộ flow

```bash
# Trên Controller
REQ_ID="req-abc12345-1234-5678-90ab-cdef12345678"

grep "$REQ_ID" /var/log/cinder/cinder-api.log
grep "$REQ_ID" /var/log/cinder/cinder-scheduler.log

# Trên Storage node
ssh storage01 "grep '$REQ_ID' /var/log/cinder/cinder-volume.log"

# Tất cả 1 lệnh với pdsh (nếu có)
pdsh -w storage[01-03] "grep '$REQ_ID' /var/log/cinder/*.log"
```

### 4.3 Trace 1 volume cụ thể

```bash
VOL_ID="<volume-uuid>"

grep "$VOL_ID" /var/log/cinder/*.log
ssh storage01 "grep '$VOL_ID' /var/log/cinder/cinder-volume.log"
ssh compute01 "grep '$VOL_ID' /var/log/nova/nova-compute.log"
```

## 5. Các pattern log quan trọng

### 5.1 ERROR + Traceback

```bash
# Tìm tất cả ERROR trong 24h
sudo find /var/log/cinder -name "*.log" -mtime -1 -exec grep -l "ERROR" {} \;

# Lấy ERROR + 10 dòng phía sau (traceback thường ngay sau)
grep -A 10 "ERROR" /var/log/cinder/cinder-volume.log | head -50
```

### 5.2 Warning về capacity

```bash
grep -i "Insufficient\|capacity\|free_capacity" /var/log/cinder/cinder-scheduler.log
```

### 5.3 Driver-specific (Ceph)

```bash
grep -i "rbd\|ceph\|rados" /var/log/cinder/cinder-volume.log
# Lỗi điển hình: "ConnectionShutdown", "No such file or directory", "rados_connect"
```

### 5.4 Driver-specific (LVM)

```bash
grep -i "lvcreate\|lvremove\|tgtadm\|lioadm" /var/log/cinder/cinder-volume.log
# Lỗi điển hình: "Insufficient free extents", "iSCSI target not found"
```

## 6. Các lỗi thường gặp và cách fix

### 6.1 Volume stuck ở `creating`

**Nguyên nhân**:
- Backend down → scheduler không assign được.
- Driver bị treo (Ceph timeout, LVM lock).

**Debug**:
```bash
# 1. Status volume
openstack volume show <vol> -f value -c status -c fault

# 2. Check scheduler log
sudo tail -n 200 /var/log/cinder/cinder-scheduler.log | grep <vol-id>

# 3. Check volume service
openstack volume service list

# 4. Check capacity
openstack volume backend pool list --long
```

**Fix**:
```bash
# Reset state (sau khi đã giải quyết root cause)
openstack volume set --state error <vol>
openstack volume delete <vol>

# Hoặc admin reset về available nếu đã verify volume tốt
cinder reset-state --state available <vol>
```

### 6.2 Volume stuck ở `error_deleting`

**Nguyên nhân**: Volume vẫn còn snapshot/backup phụ thuộc.

**Fix**:
```bash
# Xem dependency
openstack volume snapshot list --volume <vol>
openstack volume backup list --volume <vol>

# Xóa snapshot/backup trước
openstack volume snapshot delete <snap-id>

# Reset state + force delete
openstack volume set --state available <vol>
openstack volume delete --force <vol>
```

### 6.3 Attach fail: `device-busy` / `attached to wrong host`

**Nguyên nhân**:
- Tap/iSCSI session cũ còn dangling.
- Cinder DB nghĩ volume đang attached nhưng thực tế không.

**Debug**:
```bash
# 1. Xem attachment trong Cinder
openstack volume show <vol> -c attachments

# 2. Trên compute node — verify iSCSI session
sudo iscsiadm -m session
sudo multipath -ll

# 3. Trong VM XML
sudo virsh dumpxml <instance-name> | grep -A 5 "<disk"

# 4. Log
tail -100 /var/log/nova/nova-compute.log | grep -i "attach\|os-brick"
```

**Fix**:
```bash
# Reset attach status
cinder reset-state --attach-status detached <vol>

# Restart nova-compute trên compute node có vấn đề
ssh compute01 sudo systemctl restart openstack-nova-compute

# Cleanup iSCSI session cũ (nếu cần)
ssh compute01 sudo iscsiadm -m node --logout
```

### 6.4 Backup stuck ở `creating`

**Nguyên nhân**:
- Backup backend (Swift/Ceph) unreachable.
- `cinder-backup` service không chạy.

**Debug**:
```bash
openstack volume service list --service cinder-backup
tail -100 /var/log/cinder/cinder-backup.log

# Check Swift (nếu dùng Swift làm backup target)
openstack object store account show

# Check Ceph
ceph -s
rados lspools | grep backup
```

**Fix**:
```bash
# Reset
openstack volume backup set --state error <backup>
openstack volume backup delete <backup>

# Restart service
sudo systemctl restart openstack-cinder-backup
```

### 6.5 `NoValidHost` khi tạo volume

**Nguyên nhân**: Scheduler không tìm thấy backend pass tất cả filter.

**Debug**:
```bash
# 1. Log scheduler (rất chi tiết)
sudo tail -100 /var/log/cinder/cinder-scheduler.log

# 2. Check pool capacity
openstack volume backend pool list --long

# 3. Check service UP
openstack volume service list

# 4. Verify type spec vs backend
openstack volume type show <type>     # Xem volume_backend_name
openstack volume backend pool list    # So sánh
```

**Fix**:
- Sai `volume_backend_name`: chỉnh trong `volume type set` hoặc `cinder.conf`.
- Hết dung lượng: extend VG (LVM) hoặc thêm OSD (Ceph).
- AZ mismatch: dùng `--availability-zone` đúng.

### 6.6 Volume tạo OK nhưng VM không thấy disk

**Nguyên nhân**:
- iSCSI initiator không login.
- libvirt secret thiếu (Ceph).
- SELinux/AppArmor block.

**Debug**:
```bash
# Compute node
sudo virsh list
sudo virsh dumpxml <instance> | grep -A 10 disk

# Cinder side
openstack volume show <vol> -c attachments
ssh compute01 "sudo iscsiadm -m session"        # LVM/iSCSI
ssh compute01 "sudo virsh secret-list"          # Ceph

# Log
tail -100 /var/log/nova/nova-compute.log
tail -100 /var/log/libvirt/libvirtd.log
sudo dmesg | tail -50                            # Kernel-level errors
```

### 6.7 Volume `error_extending`

**Nguyên nhân**: Driver không extend được (backend hết dung lượng / không support online extend).

**Fix**:
```bash
# Reset
openstack volume set --state available <vol>

# Verify backend capacity
openstack volume backend pool list --long

# Thử extend lại với size nhỏ hơn
openstack volume set --size <size> <vol>
```

## 7. Bật trace toàn bộ một thao tác (debug full)

Khi cần debug sâu, bật debug trên TẤT CẢ service Cinder + Nova:

```bash
# ==== Controller ====
sudo crudini --set /etc/cinder/cinder.conf DEFAULT debug True
sudo systemctl restart openstack-cinder-api openstack-cinder-scheduler

# ==== Storage node ====
sudo crudini --set /etc/cinder/cinder.conf DEFAULT debug True
sudo systemctl restart openstack-cinder-volume openstack-cinder-backup

# ==== Compute node (nếu debug attach) ====
sudo crudini --set /etc/nova/nova.conf DEFAULT debug True
sudo systemctl restart openstack-nova-compute

# Reproduce lỗi
openstack --debug volume create --size 1 debug-vol

# Lấy request-id và grep
REQ=$(openstack volume show debug-vol -f json | jq -r '.request_id // empty')
grep "req-" /var/log/cinder/*.log | head -5

# QUAN TRỌNG: Tắt debug sau khi xong (log debug rất nặng)
sudo crudini --set /etc/cinder/cinder.conf DEFAULT debug False
sudo systemctl restart openstack-cinder-api openstack-cinder-scheduler openstack-cinder-volume
```

## 8. Giám sát Production

### 8.1 Log aggregation

Khuyến nghị dùng:
- **Loki + Grafana**: nhẹ, dễ deploy
- **ELK** (Elasticsearch + Logstash + Kibana): mạnh nhưng nặng
- **Fluentd / Fluent Bit**: collector tốt cho containerized

Cấu hình bật JSON log để dễ parse:
```ini
[DEFAULT]
use_json = True
```

### 8.2 Alert quan trọng

| Event | Severity | Action |
|---|---|---|
| `cinder-volume` service DOWN | Critical | Page oncall ngay |
| Backend free capacity < 10% | Warning | Mở rộng backend |
| ERROR / Traceback xuất hiện trong log | Warning | Investigate trong 1h |
| Volume stuck `creating/deleting` > 30 min | Warning | Manual intervention |
| Backup fail rate > 5% trong 24h | Warning | Check backup repo |

### 8.3 Metrics nên monitor

- Số volume theo trạng thái (available / in-use / error)
- Free capacity từng backend
- IOPS / throughput từng backend
- API response time (p50, p95, p99)
- Backup success rate
- Số attach/detach pending

Lấy qua:
```bash
# Volume status
openstack volume list --all-projects -f value -c Status | sort | uniq -c

# Backend capacity
openstack volume backend pool list --long -f json | \
  jq '.[] | {name:.Name, total:.Capabilities.total_capacity_gb, free:.Capabilities.free_capacity_gb}'

# Hoặc qua Cinder API:
curl -H "X-Auth-Token: $TOKEN" http://controller:8776/v3/<project>/scheduler-stats/get_pools?detail=True
```

## 9. Quick reference – Lệnh debug hay dùng

```bash
# Top 10 ERROR gần nhất trên storage node
sudo grep ERROR /var/log/cinder/cinder-volume.log | tail -10

# Đếm ERROR theo type trong 1 ngày
sudo grep ERROR /var/log/cinder/cinder-volume.log | \
  awk -F'ERROR' '{print $2}' | awk '{print $2}' | sort | uniq -c | sort -rn | head

# Trace 1 volume
VOL=xxxx-yyyy
grep $VOL /var/log/cinder/*.log

# Trace 1 request
REQ=req-abc
grep $REQ /var/log/cinder/*.log

# Tail tất cả log Cinder cùng lúc
sudo tail -F /var/log/cinder/*.log

# Hoặc dùng multitail
sudo multitail /var/log/cinder/cinder-api.log /var/log/cinder/cinder-volume.log

# Filter chỉ WARNING + ERROR
sudo tail -F /var/log/cinder/cinder-volume.log | grep -E "WARNING|ERROR"
```
