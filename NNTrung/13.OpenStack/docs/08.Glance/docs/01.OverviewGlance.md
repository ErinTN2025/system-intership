# OpenStack Glance - Image Service

## 1. Khái niệm

**Glance** là dịch vụ quản lý **disk image** (ảnh đĩa máy ảo) trong OpenStack. Nó đóng vai trò là trung tâm lưu trữ và cung cấp các "bản thiết kế" để các dịch vụ khác (chủ yếu là **Nova**) lấy ra và khởi tạo máy ảo.

> **Analogy:** Glance giống như một **thư viện sách** — nó không đọc sách thay bạn, nhưng nó lưu trữ, quản lý catalog và cho mượn đúng cuốn sách (image) khi cần.

**Vai trò chính:**
- Nhận và xử lý API request liên quan đến disk image và metadata
- Lưu trữ image trên nhiều loại backend khác nhau (local, Ceph, S3, Swift...)
- Quản lý metadata của image (tên, định dạng, trạng thái, checksum...)
- Cung cấp snapshot của máy ảo đang chạy để lưu trữ hoặc nhân bản

---

## 2. Advantages

| Lợi ích | Giải thích |
|---------|-----------|
| **Tính nhất quán** | Mọi VM khởi tạo từ cùng một image đều có cấu hình phần mềm giống hệt nhau |
| **Tốc độ** | Tích hợp với Ceph (Copy-on-Write) giúp khởi tạo VM gần như tức thì, không cần copy toàn bộ dữ liệu |
| **Linh hoạt backend** | Hỗ trợ lưu trữ trên File System, Ceph, Swift, S3 — dễ dàng thay đổi tùy hạ tầng |
| **Phân quyền** | Kiểm soát ai có thể xem hoặc dùng image (Public / Private / Shared) |
| **Độc lập, scalable** | Là dịch vụ tách biệt, có thể scale riêng khi số lượng image tăng lớn |

## 3. Kiến trúc

Glance theo mô hình **client-server**, cung cấp **REST API** để người dùng và các dịch vụ OpenStack khác gửi yêu cầu.

### 3.1 Các thành phần

| Thành phần | Vai trò | Ghi chú |
|-----------|---------|---------|
| **Client** | Bất kỳ ứng dụng nào gọi đến Glance API (Nova, CLI, Horizon...) | |
| **REST API** | Điểm tiếp nhận mọi request từ bên ngoài | Xác thực qua Keystone |
| **Glance-api** | Tiếp nhận, xác thực và điều phối các thao tác lưu trữ/truy xuất image | Thành phần chính, luôn chạy |
| **Domain Controller** | Middleware xử lý logic nghiệp vụ: authorization, policy, notification | |
| **DAL** (Database Abstraction Layer) | Chuẩn hóa giao tiếp giữa Glance và database | |
| **Database** | Lưu **metadata** của image: ID, tên, định dạng, trạng thái, checksum, location | MariaDB / PostgreSQL |
| **Glance Store** | Lớp giao tiếp giữa Glance và storage backend | Cung cấp interface thống nhất |
| **Storage Backend** | Nơi lưu **file nhị phân thực tế** của image (vài trăm MB đến hàng chục GB) | File System, Ceph, Swift, S3, HTTP |

> **Lưu ý:** `glance-registry` đã bị **deprecated** và loại bỏ hoàn toàn từ phiên bản Queens trở đi. Không còn tồn tại ở các phiên bản hiện tại.

### 3.2 Tại sao tách Database và Storage Backend

Glance tách biệt metadata (Database) và file thực tế (Backend) vì 3 lý do:

- **Hiệu năng:** Database SQL được tối ưu cho tìm kiếm nhanh. Nhét hàng TB dữ liệu ảnh vào DB sẽ cực kỳ chậm và dễ corrupt.
- **Linh hoạt:** Metadata lưu ở MariaDB, file ảnh lưu ở Ceph hoặc S3 — tùy hạ tầng, không bị lock-in.
- **Truy vấn nhẹ:** Lệnh `openstack image list` chỉ cần đọc DB (rất nhanh), không cần quét toàn bộ backend lưu trữ.

### 3.3 Workflow thực tế — Nova tạo VM từ Image

![Glance Workflow](../images/Screenshot_2.png)

```
User/Nova          Glance-api         Keystone        Database        Backend
    |                   |                 |               |               |
    |-- Request image ->|                 |               |               |
    |                   |-- Verify token->|               |               |
    |                   |<-- Token OK ----|               |               |
    |                   |-- Query metadata--------------->|               |
    |                   |<-- Location (path/URL) ---------|               |
    |                   |-- Fetch file data --------------------------------->|
    |                   |<-- Binary data ----------------------------------|
    |<-- Image data ----|                 |               |               |
    |                   |                 |               |               |
```

**Các bước:**
1. **Nova** gửi request đến `glance-api` để lấy image cụ thể
2. `glance-api` xác thực token của người dùng qua **Keystone**
3. Glance truy vấn **Database** để lấy thông tin `location` (địa chỉ file trong backend)
4. Glance lấy file nhị phân từ **Storage Backend**
5. Dữ liệu được trả về Nova → Nova dùng để boot VM trên Compute Node

## 4. Định dạng Image

### 4.1 Disk Format

Định dạng của file ảnh đĩa — quyết định cấu trúc dữ liệu bên trong file.

| Format | Mô tả | Khi nào dùng |
|--------|-------|-------------|
| `raw` | Định dạng thô, không nén | Hiệu năng cao nhất, dùng với Ceph |
| `qcow2` | QEMU Copy On Write v2, hỗ trợ nén và snapshot | Phổ biến nhất với KVM/QEMU |
| `vhd` / `vhdx` | Microsoft Hyper-V format | Khi migrate từ môi trường Hyper-V |
| `vmdk` | VMware format | Khi migrate từ môi trường VMware |
| `iso` | Disk image dạng đĩa quang | Cài đặt OS từ đầu |

### 4.2 Vòng đời Image (Status Lifecycle)

Từ khi tạo đến khi sẵn sàng, image trải qua các trạng thái sau:

```
[Create request]
       |
       v
   QUEUED ──(upload bắt đầu)──> SAVING ──(upload xong)──> ACTIVE
       |                           |
       |                     (lỗi upload)
       |                           |
       |                           v
       |                        KILLED
       |
   ACTIVE ──(deactivate)──> DEACTIVATED
       |
   ACTIVE ──(xóa)──> PENDING_DELETE ──> DELETED
```

| Trạng thái | Ý nghĩa |
|-----------|---------|
| `queued` | Image đã đăng ký metadata, chưa có dữ liệu upload |
| `saving` | Đang trong quá trình upload dữ liệu lên backend |
| `active` | Sẵn sàng sử dụng |
| `killed` | Upload thất bại, không thể dùng được |
| `deactivated` | Bị tạm vô hiệu hóa bởi admin (không thể download) |
| `pending_delete` | Đã xóa nhưng chờ scrubber dọn dẹp file thực tế |
| `deleted` | Đã xóa hoàn toàn |

### 4.3 Container Format

Mô tả cách **đóng gói** (wrapper) bên ngoài của image — không liên quan đến nội dung bên trong.

| Format | Mô tả |
|--------|-------|
| `bare` | Không có wrapper — **dùng hầu hết mọi trường hợp** |
| `ovf` | Open Virtualization Format |
| `ova` | OVA tar archive |
| `aki` | Amazon Kernel Image |
| `ari` | Amazon Ramdisk Image |
| `ami` | Amazon Machine Image |
| `docker` | Docker tar archive của container filesystem |
| `compressed` | Image đã được nén |

> **Thực tế:** Container format hiện không còn quan trọng. Glance không parse nó nhiều nữa — chủ yếu dùng để ghi metadata. **Mặc định luôn chọn `bare`**, kể cả với qcow2, raw, hay iso.
