# Volume Types, Extra Specs và QoS

## 1. Volume Type là gì?

**Volume Type** là một "template" định nghĩa **đặc tính** của volume khi user tạo. Nó tương đương khái niệm **Flavor** bên Nova nhưng dành cho storage.

Volume Type cho phép admin:
- **Chia tier**: SSD (nhanh, đắt) vs HDD (chậm, rẻ).
- **Định tuyến backend**: gửi volume đến đúng backend (LVM / Ceph / NetApp).
- **Áp QoS**: giới hạn IOPS, bandwidth.
- **Bật encryption**: tự động mã hóa LUKS.
- **Bật multi-attach**: cho phép gắn vào nhiều VM.
- **Cô lập tenant**: chỉ một số project được dùng type nhất định.

**Khi user tạo volume**:
```bash
# Không chỉ định type → dùng default_volume_type trong cinder.conf
openstack volume create --size 10 my-vol

# Chỉ định type cụ thể
openstack volume create --size 10 --type ssd my-vol
```

## 2. Extra Specs – Các thuộc tính của Volume Type

Extra Specs là **key-value pairs** gắn vào Volume Type. Cinder Scheduler dùng chúng để chọn backend phù hợp.

### 2.1 Các key thường dùng

| Extra Spec Key | Ý nghĩa | Ví dụ |
|---|---|---|
| `volume_backend_name` | **Quan trọng nhất** — gắn type vào backend cụ thể | `LVM_iSCSI`, `CEPH_RBD` |
| `multiattach` | Cho phép volume gắn vào nhiều VM | `"<is> True"` |
| `replication_enabled` | Bật replication cho disaster recovery | `"<is> True"` |
| `RESKEY:availability_zones` | Giới hạn volume chỉ tạo trong AZ chỉ định | `nova,az-2` |
| `thin_provisioning_support` | Yêu cầu backend hỗ trợ thin provisioning | `"<is> True"` |
| `compression_support` | Yêu cầu nén dữ liệu | `"<is> True"` |
| `qos_specs` | Liên kết QoS (set qua `qos associate`) | — |
| `hw:scsi_model` | Loại SCSI controller cho VM | `virtio-scsi` |

### 2.2 Cú pháp toán tử (Operators)

Khi spec là một điều kiện so sánh thay vì exact match, dùng operator:

| Operator | Ý nghĩa | Ví dụ |
|---|---|---|
| `=` (hoặc không có) | Bằng (exact match) | `volume_backend_name=ceph` |
| `<is>` | So sánh boolean | `multiattach="<is> True"` |
| `<in>` | Substring match | `architecture="<in> x86"` |
| `<or>` | Một trong các giá trị | `disk_type="<or> ssd <or> nvme"` |
| `==`, `!=`, `>=`, `<=`, `<`, `>` | So sánh số | `iops=">= 1000"` |
| `s==`, `s!=`, `s<`, `s>` | So sánh chuỗi | `tier="s== gold"` |

**Ví dụ**:
```bash
# Type chỉ chấp nhận backend hỗ trợ multiattach
openstack volume type set ssd --property multiattach="<is> True"

# Type chấp nhận cả SSD và NVMe
openstack volume type set fast --property disk_type="<or> ssd <or> nvme"
```

### 2.3 Quản lý Volume Type qua CLI

```bash
# Tạo type (public - mọi project đều dùng được)
openstack volume type create ssd
openstack volume type create --public ssd

# Tạo type private (chỉ admin/specific project)
openstack volume type create --private vip-only

# Cho project private dùng type
openstack volume type set --project <project-id> vip-only

# Liệt kê (--long để xem extra specs)
openstack volume type list --long

# Set extra spec
openstack volume type set ssd --property volume_backend_name=CEPH_SSD

# Set nhiều spec một lúc
openstack volume type set ssd \
  --property volume_backend_name=CEPH_SSD \
  --property multiattach="<is> True"

# Unset spec
openstack volume type unset ssd --property multiattach

# Xem chi tiết
openstack volume type show ssd

# Xóa
openstack volume type delete ssd
```

## 3. QoS Specs – Giới hạn IOPS & Bandwidth

QoS (Quality of Service) cho phép giới hạn **IOPS** (operations/sec) và **throughput** (bytes/sec) trên volume.

### 3.1 Các thuộc tính QoS

| Property | Ý nghĩa | Đơn vị |
|---|---|---|
| `total_iops_sec` | Tổng IOPS (đọc + ghi) | ops/s |
| `read_iops_sec` | IOPS đọc | ops/s |
| `write_iops_sec` | IOPS ghi | ops/s |
| `total_bytes_sec` | Tổng throughput | bytes/s |
| `read_bytes_sec` | Throughput đọc | bytes/s |
| `write_bytes_sec` | Throughput ghi | bytes/s |
| `total_iops_sec_max` / `total_bytes_sec_max` | Burst limit | — |
| `size_iops_sec` | IOPS tỷ lệ theo GB (vd 10 = 10 IOPS/GB) | ops/s/GB |

### 3.2 Consumer

QoS có 3 chế độ enforce:

| Consumer | Áp ở đâu | Khi nào dùng |
|---|---|---|
| `front-end` | Tại hypervisor (Nova/libvirt) | Phổ biến nhất, hoạt động với mọi backend |
| `back-end` | Tại storage backend (driver-specific) | Khi backend hỗ trợ native QoS (NetApp, SolidFire…) |
| `both` | Cả 2 | Bảo vệ kép |

### 3.3 Quản lý QoS qua CLI

```bash
# Tạo QoS
openstack volume qos create gold-qos --consumer front-end \
  --property total_iops_sec=2000 \
  --property total_bytes_sec=104857600 \
  --property read_iops_sec=1500 \
  --property write_iops_sec=500

# Gắn QoS vào Volume Type
openstack volume qos associate gold-qos ssd

# Verify
openstack volume qos show gold-qos
openstack volume type show ssd        # Phải thấy qos_specs_id

# Đổi property
openstack volume qos set gold-qos --property total_iops_sec=5000

# Xóa property
openstack volume qos unset gold-qos --property read_iops_sec

# Bỏ liên kết
openstack volume qos disassociate gold-qos --volume-type ssd

# Bỏ tất cả liên kết
openstack volume qos disassociate-all gold-qos

# Xóa QoS
openstack volume qos delete gold-qos
```

### 3.4 Bảng tier mẫu

| Tier | IOPS | Bandwidth | Latency | Backend |
|---|---|---|---|---|
| **bronze** | 500 | 50 MB/s | Best-effort | HDD pool |
| **silver** | 2,000 | 200 MB/s | Best-effort | SAS pool |
| **gold** | 10,000 | 500 MB/s | < 5ms | SSD pool |
| **platinum** | 50,000 | 2 GB/s | < 1ms | NVMe pool |

**Tạo nhanh 4 tier**:

```bash
# Bronze
openstack volume type create bronze
openstack volume type set bronze --property volume_backend_name=HDD_POOL
openstack volume qos create bronze-qos --consumer front-end \
  --property total_iops_sec=500 --property total_bytes_sec=52428800
openstack volume qos associate bronze-qos bronze

# Silver
openstack volume type create silver
openstack volume type set silver --property volume_backend_name=SAS_POOL
openstack volume qos create silver-qos --consumer front-end \
  --property total_iops_sec=2000 --property total_bytes_sec=209715200
openstack volume qos associate silver-qos silver

# Gold
openstack volume type create gold
openstack volume type set gold --property volume_backend_name=SSD_POOL
openstack volume qos create gold-qos --consumer front-end \
  --property total_iops_sec=10000 --property total_bytes_sec=524288000
openstack volume qos associate gold-qos gold

# Platinum
openstack volume type create platinum
openstack volume type set platinum --property volume_backend_name=NVME_POOL
openstack volume qos create platinum-qos --consumer front-end \
  --property total_iops_sec=50000 --property total_bytes_sec=2147483648
openstack volume qos associate platinum-qos platinum
```

## 4. Encryption – Mã hóa Volume (LUKS)

Cinder hỗ trợ mã hóa volume bằng **LUKS** (Linux Unified Key Setup). Key được lưu ở Key Manager (Barbican) hoặc fixed_key (chỉ lab).

### 4.1 Cấu hình Key Manager

**Lab/DevStack** (không an toàn):
```ini
# cinder.conf + nova.conf trên compute
[key_manager]
backend = cinder.keymgr.conf_key_mgr.ConfKeyManager
fixed_key = <64-hex-chars>
```

**Production** (dùng Barbican):
```ini
[key_manager]
backend = barbican
auth_endpoint = http://controller:5000/v3
```

### 4.2 Tạo Encrypted Volume Type

```bash
# Tạo type
openstack volume type create encrypted-type

# Bật encryption với LUKS
openstack volume type set encrypted-type \
  --encryption-provider luks \
  --encryption-cipher aes-xts-plain64 \
  --encryption-key-size 256 \
  --encryption-control-location front-end

# Verify
openstack volume type show encrypted-type
```

### 4.3 Các tham số encryption

| Tham số | Giá trị phổ biến | Ý nghĩa |
|---|---|---|
| `--encryption-provider` | `luks` (Linux) hoặc `plain` | Cơ chế mã hóa |
| `--encryption-cipher` | `aes-xts-plain64` (khuyến nghị) | Thuật toán |
| `--encryption-key-size` | 128, 256, 512 | Độ dài key (bits) |
| `--encryption-control-location` | `front-end` hoặc `back-end` | Nơi enforce: hypervisor hay backend |

### 4.4 Workflow encrypted volume

1. User: `openstack volume create --type encrypted-type ...`
2. Cinder request Barbican → tạo encryption key, lưu key UUID
3. Volume tạo bình thường, metadata chứa key UUID
4. Khi attach: Nova gọi Barbican → lấy key → libvirt mở LUKS → VM thấy `/dev/vdb` đã decrypt
5. Trên host: raw data trên backend là encrypted (random bytes)

### 4.5 Lưu ý

- **Không thể đổi key** sau khi tạo volume (phải tạo lại).
- **Không clone được** từ encrypted sang non-encrypted (và ngược lại).
- **Performance overhead** ~5–15% tùy CPU (modern CPU có AES-NI sẽ nhanh).
- **Boot from encrypted volume**: hỗ trợ từ Train, nhưng cần test kỹ với cloud-init.

## 5. Multi-attach

Multi-attach cho phép **1 volume gắn vào nhiều VM** cùng lúc.

### 5.1 Khi nào dùng

- **Cluster filesystem** (OCFS2, GFS2): các node trong cluster cùng truy cập shared storage.
- **DB cluster** (Oracle RAC, MS SQL Cluster).
- **HA application** với shared state.

### 5.2 Cảnh báo nguy hiểm

⚠️ **TUYỆT ĐỐI KHÔNG dùng filesystem thường (ext4/xfs)** trên multi-attach volume — sẽ corrupt data ngay lập tức.

Phải dùng:
- **Cluster FS**: OCFS2, GFS2 (cần cluster manager như Pacemaker).
- **Hoặc app-level**: ứng dụng tự coordinate (vd: chỉ 1 node write, các node khác readonly).

### 5.3 Backend hỗ trợ

| Backend | Multi-attach | Ghi chú |
|---|---|---|
| Ceph RBD | ✅ | Hỗ trợ tốt |
| NFS | ✅ | Cần backend export đúng |
| LVM iSCSI | ❌ | Không hỗ trợ |
| NetApp | ✅ | Một số model |
| Pure Storage | ✅ | — |

Xem driver matrix: https://docs.openstack.org/cinder/latest/reference/support-matrix.html

### 5.4 Cấu hình

```bash
# Tạo type
openstack volume type create multiattach-type
openstack volume type set multiattach-type --property multiattach="<is> True"
openstack volume type set multiattach-type --property volume_backend_name=CEPH_RBD

# Tạo volume
openstack volume create --size 10 --type multiattach-type shared-vol

# Attach vào 2 VM (cùng lúc)
openstack server add volume vm1 shared-vol
openstack server add volume vm2 shared-vol

# Verify
openstack volume show shared-vol -c attachments
```

## 6. Best practices

### 6.1 Đặt tên Volume Type

- Đặt tên theo **purpose**, không theo backend: `gold`, `silver` (tốt hơn `ceph-ssd-pool-1`).
- Nếu có nhiều tier × nhiều AZ: `gold-az1`, `gold-az2`.

### 6.2 Default Volume Type

```ini
# cinder.conf
[DEFAULT]
default_volume_type = silver
```

Luôn set `default_volume_type` — nếu không, volume create không có `--type` sẽ fail trên multi-backend.

### 6.3 Quota theo Volume Type

```bash
# Đặt quota riêng cho type "gold" trong project
openstack quota set --volume-type gold --volumes 5 <project>
openstack quota set --volume-type gold --gigabytes 100 <project>
```

### 6.4 Migration giữa các Type (Retype)

```bash
# Retype volume sang type khác (có thể trigger migration)
openstack volume set --type ssd --retype-policy on-demand <volume>

# Policy:
# - never:      Chỉ retype nếu cùng backend (đổi metadata)
# - on-demand:  Migrate nếu cần (chậm hơn nhưng đảm bảo)
```

### 6.5 Audit

```bash
# Xem volume nào dùng type nào
openstack volume list --long -c ID -c Name -c "Volume Type"

# Đếm
openstack volume list --long -f json | jq '[.[] | .Type] | group_by(.) | map({type:.[0], count:length})'
```

## 7. Tóm tắt

| Concept | Vai trò |
|---|---|
| **Volume Type** | "Template" để user chọn khi tạo volume |
| **Extra Specs** | Key-value gắn vào type, scheduler dùng để match backend |
| **`volume_backend_name`** | Spec quan trọng nhất — gắn type vào backend cụ thể |
| **QoS Specs** | Giới hạn IOPS / bandwidth, gắn vào type qua `qos associate` |
| **Encryption** | Mã hóa LUKS, key lưu ở Barbican (production) |
| **Multi-attach** | 1 volume → nhiều VM, cần cluster FS |
| **Retype** | Đổi type sau khi tạo, có thể trigger migration |
