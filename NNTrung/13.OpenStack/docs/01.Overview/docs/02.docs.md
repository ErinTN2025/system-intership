# Phân tích một số thuật ngữ phổ biến trong OpenStack
### 1. Image(VMImage)
- Là khuôn mẫu(template) để tạo ra máy ảo
  - **Image chứa**:
    - Hệ điều hành - OS( Ubuntu, Rocky,...)
    - Phần mềm tự cài hoặc cài sẵn (Nginx, PHP, Docker,...)
    - Cấu hình cơ bản
    - Ví dụ: **AMI**( AWS), **Custom Image**(OpenStack, GCP), **Golden Image**( trong doanh nghiệp)

### 2. Instance 
- Là máy ảo đang chạy
  - Đặc điểm
    - Có RAM, CPU, IP, DISK
    - Có thể Start/ Stop. SSH vào và install Plugin
### 3. Snapshot
- Bản chụp tại 1 thời điểm của Instance hay sao lưu trạng thái của Instance
  - Đặc điểm:
    - Dùng để Backup
    - Rollback khi lỗi
    - Clone nhanh Instance
    - Có thể chụp RootDisk, DataDisk, Cả hệ thống(tùy Cloud)
### 4. Container
- Là môi trường chạy ứng dụng nhẹ
  - Đặc điểm:
    - Chạy trên HostOS
    - Dùng kernel chung
    - Cô lập ở mức Process
    - Khởi động rất nhanh (ms)
- Container không phải là máy ảo, nó chỉ là môi trường chạy trong máy ảo để run product
- **Container Image**: Là khuôn mẫu (or Template) để tạo Container
  - **Đặc điểm**
    - Chứa **App, thư viện, Runtime**


### 5. Metadata Service endpoint

- Địa chỉ `169.254.169.254` là một địa chỉ IP cực kỳ đặc biệt và quen thuộc nếu bạn làm việc trong môi trường Điện toán đám mây (Cloud Computing) như AWS, Azure, Google Cloud, hay OpenStack.

Về bản chất, đây là Metadata Service (IMDS) endpoint — một "hộp thư" cục bộ giúp các máy ảo (VM/Instance) tự truy vấn thông tin của chính nó mà không cần biết cấu hình mạng phức tạp bên ngoài.

#### 5.1 Bản chất của dải IP `169.254.x.x`
Trước hết, dải IP `169.254.0.0/16` được định nghĩa là Link-Local Address (địa chỉ liên kết cục bộ).

- Bình thường, nếu một máy tính không nhận được IP từ DHCP server, hệ điều hành sẽ tự cấp một IP ngẫu nhiên trong dải này (gọi là APIPA trên Windows) để các máy trong cùng mạng nội bộ có thể thấy nhau.
- Các gói tin gửi đến dải IP này không bao giờ được định tuyến (route) ra ngoài Internet.

Tận dụng đặc tính "không lọt ra ngoài" này, các nền tảng đám mây đã giữ lại một IP cố định duy nhất là `169.254.169.254` làm Dịch vụ cung cấp dữ liệu cấu hình (Metadata Service).

#### 5.2 Vai trò của `169.254.169.254` trong Cloud
Khi một máy ảo vừa được khởi tạo từ một Image trắng (Ubuntu, Rocky Linux, Windows Server...), nó hoàn toàn "mất trí nhớ": Nó không biết tên của nó là gì, nằm trong vùng mạng nào, hay cần chạy script gì lúc khởi động.

Lúc này, các công cụ như cloud-init cài sẵn trong máy ảo sẽ gửi một request HTTP (thường là lệnh curl) đến địa chỉ 169.254.169.254 để lấy thông tin.

Các thông tin chính mà máy ảo có thể lấy được:
- **User-data**: Các đoạn script (Shell script, Ansible, Cloud-config) do người dùng truyền vào lúc tạo máy ảo để tự động cài đặt phần mềm, cấu hình ban đầu (gọi là Bootstrapping).
- **Networking**: IP public, IP private, MAC address, Subnet ID.
- **Identity/Security**: Các Token bảo mật, IAM Role/Credentials tạm thời để máy ảo có quyền truy cập hợp lệ vào các tài nguyên khác (như Storage, Database) mà không cần lưu cứng mật khẩu (Hardcoded password) trong mã nguồn.
- **System info**: Instance ID, Instance Type, Availability Zone, Hostname.

```bash
[ VM ]
       │
       │ (1) HTTP Request đến 169.254.169.254:80
       ▼
[ iptables / Quy tắc mạng trong Router Namespace hoặc DHCP Namespace ]
       │
       │ (2) Chặn & Redirect (Dùng metadata_proxy_shared_secret)
       ▼
[ haproxy (Metadata Proxy) ]
       │
       │ (3) Chuyển tiếp qua Unix Domain Socket
       ▼
[ neutron-metadata-agent ]
       │
       │ (4) Gửi HTTP Request kèm các Header bổ sung (Instance IP, Network ID...)
       ▼
[ nova-api-metadata ]
       │
       │ (5) Truy vấn thông tin
       ▼
[ Nova DB (Cơ sở dữ liệu) ]
```

Từ VM đến `iptables`: Khi bạn chạy lệnh `curl 169.254.169.254`, gói tin sẽ đi ra cổng mạng của VM. Tại Namespace quản lý mạng của mạng đó (có thể là DHCP Namespace hoặc Router Namespace tùy thuộc vào cấu hình isolated_metadata của Neutron), một rule `iptables` sẽ chặn (intercept) cổng 80 của IP này và ép nó chuyển hướng (Redirect) về một cổng nội bộ (thường là 8775 hoặc cổng của proxy).

Qua `haproxy` (Metadata Proxy):
Neutron không hứng trực tiếp traffic HTTP từ VM. Thay vào đó, nó sinh ra các tiến trình `haproxy` siêu nhỏ nằm ngay trong các Network Namespace. `haproxy` này có nhiệm vụ tiếp nhận request từ VM, sau đó đóng gói thêm một số thông tin định danh rồi đẩy qua một cổng giao tiếp nội bộ gọi là Unix Domain Socket để tới Agent.

`neutron-metadata-agent` làm nhiệm vụ gì?
Thành phần này cực kỳ quan trọng vì bản thân nova-api-metadata (ở bước sau) không hiểu được cấu hình mạng của Neutron. neutron-metadata-agent sẽ lấy IP nguồn của VM (từ request) và tra cứu với Neutron Server xem: "IP này thuộc về Instance ID nào? Thuộc Tenant nào?".
Sau khi đã "giải mã" được danh tính của VM, nó sẽ thêm các thông tin này vào HTTP Header (như X-Neutron-Network-Id, X-Forwarded-For) rồi mới gửi tiếp sang cho Nova.

Đến `nova-api-metadata` và `Nova DB`:
Lúc này nova-api-metadata nhận được một request đã "rõ ràng danh tính". Nó chỉ việc cầm Instance ID đó, vào Nova DB gõ cửa để lấy thông tin (Hostname, Keypair, User-data...) và trả ngược lại theo đúng đường cũ về cho VM.
