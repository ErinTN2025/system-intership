# Sự cố "Binding failed for port" trên OpenStack Kolla-Ansible — Ghi chép chẩn đoán & khắc phục

> Tài liệu này ghi lại toàn bộ quá trình truy vết một sự cố thực tế: từ triệu chứng ban
> đầu ("Binding failed for port") cho tới nguyên nhân gốc (RabbitMQ 4.x + quorum/stream
> queue chưa migrate đúng) và cách xử lý. Đọc theo thứ tự để hiểu cách lần ngược từ
> triệu chứng về gốc rễ.S

---

## 1. Triệu chứng ban đầu

Thông báo lỗi nhận được:

```
Binding failed for port <port-uuid>, please check neutron logs for more information.
```

Lỗi này thường xuất hiện khi:
- Tạo VM mới (nova không gắn được network cho instance).
- Tạo/đính port thủ công.
- Gắn floating IP.

**Quan trọng:** Đây hầu như luôn là *triệu chứng*, không phải *bệnh*. Nó chỉ nói "Neutron
không bind được port vào host" — còn *vì sao* thì phải đào log.

---

## 2. Bản chất lỗi "Binding failed for port"

Cơ chế bind port trong ML2:

1. Neutron-server (controller) nhận yêu cầu bind port cho một host.
2. ML2 duyệt qua các **mechanism driver** (openvswitch, ovn, linuxbridge...).
3. Driver chỉ bind thành công nếu trên host đó có **L2 agent đang "alive"** và segment
   network khớp với cấu hình (bridge_mappings, network type...).
4. Nếu không driver nào bind được → trả về **"Binding failed for port"**.

Các nguyên nhân gốc có thể dẫn tới đây:

| Nhóm nguyên nhân | Ví dụ cụ thể |
|---|---|
| Agent không "alive" | OVS/LinuxBridge agent chết, hoặc **không report được state** lên controller |
| Sai cấu hình ML2 | `mechanism_drivers` không khớp agent thực tế |
| Thiếu mapping | `bridge_mappings` không có physnet mà network dùng |
| Network type chưa khai báo | tạo VLAN nhưng thiếu `network_vlan_ranges` |
| **Mất/chậm RPC (RabbitMQ)** | agent sống nhưng không report_state được → controller tưởng agent down |

> Trong sự cố này, thủ phạm cuối cùng nằm ở nhóm cuối: **RPC qua RabbitMQ bị nghẽn**,
> khiến agent không report_state được → controller coi agent là down → không có agent
> alive để bind → "Binding failed".

### Lệnh kiểm tra nhanh ban đầu

```bash
# Xem agent có alive không (cột Alive / State)
openstack network agent list

# Xem cấu hình ML2
docker exec -it neutron_server cat /etc/neutron/plugins/ml2/ml2_conf.ini
```

Nếu dùng **OVN** thay vì OVS, hướng chẩn đoán khác (kiểm tra `ovn-controller`,
`ovn_metadata_agent` thay cho OVS agent).

## 3. Đặc thù khi chạy trên Kolla-Ansible

Trên Kolla-Ansible, mọi service chạy trong **Docker container**, không phải systemd. Vì
vậy:

```bash

docker logs --tail 100 -f neutron_server
```

```bash
docker ps -a | grep -E "neutron|ovn"
```


## 4. Lần theo log: phát hiện MessagingTimeout

Khi xem log OVS agent, xuất hiện lỗi mấu chốt:

```
ERROR ... ovs_neutron_agent Failed reporting state!:
oslo_messaging.exceptions.MessagingTimeout:
Timed out waiting for a reply to message ID 8b8a6925607d466bb4fc676d74f54b4d
```

### Diễn giải

- OVS agent định kỳ gọi `report_state` (RPC call) lên message queue để báo "tôi còn sống".
- Nó **gửi được** message nhưng **không nhận lại reply** → timeout.
- Hệ quả dây chuyền:

```
OVS agent không report_state được
        ↓
neutron-server thấy agent này "down" (dù thực tế agent vẫn chạy)
        ↓
khi tạo VM/port, ML2 không tìm thấy agent "alive" trên host để bind
        ↓
trả về "Binding failed for port"
```

**Kết luận giai đoạn này:** bệnh không nằm ở Neutron, mà ở tầng **RPC / message queue
(RabbitMQ)**.

## 5. Kiểm tra RabbitMQ — loại trừ partition/alarm

```bash
docker exec -it rabbitmq rabbitmqctl cluster_status
```

Những thứ cần soi trong output:

| Mục | Ý nghĩa nếu bất thường |
|---|---|
| `Network Partitions` | Nếu **không rỗng** → cluster bị chia rẽ (split-brain), nguyên nhân kinh điển |
| `Alarms` | `memory` / `disk` alarm → queue bị block, mọi RPC treo |
| `Running Nodes` | Thiếu node → node RabbitMQ chết |
| `Disk Nodes` | Xác định cấu trúc cluster |

Trong sự cố này, kết quả là:
- Cluster **single-node** (`rabbit@control01`) — loại trừ partition giữa các node.
- **Alarms: (none)** — không bị memory/disk alarm.
- **Network Partitions: (none)** — không split-brain.
- RabbitMQ 4.0.9 trên Erlang 26.

→ RabbitMQ "trông khỏe" nhưng RPC vẫn timeout → phải đào sâu hơn vào **queue và
connection**.

### Lệnh phụ trợ

```bash
# Log RabbitMQ — tìm partition, alarm, memory, disk
tail -100 /var/log/kolla/rabbitmq/*.log

# Kiểm tra mạng từ compute node tới RabbitMQ (port AMQP 5672)
nc -zv <rabbitmq_vip_hoặc_ip> 5672
```

## 6. Phát hiện fanout queue tồn đọng hàng chục nghìn message

```bash
docker exec rabbitmq rabbitmqctl list_queues name messages consumers | grep -i fanout
```

Kết quả bất thường:

```
scheduler_fanout                              13082   5    ← nova scheduler, kẹt 13k
cinder-scheduler_fanout                       12962   1    ← cinder, kẹt ~13k
q-agent-notifier-l2population-update_fanout     305   2
neutron-vo-Port-1.10_fanout                     243   3
q-server-resource-versions_fanout              144    3
```

### Diễn giải

- **Fanout queue** trong OpenStack vốn là loại broadcast tạm thời — message lẽ ra phải
  được consume và biến mất gần như tức thì.
- Khi chúng dồn lên **hàng nghìn / chục nghìn** message → có gì đó khiến consumer
  **không tiêu thụ** dù vẫn đăng ký (cột `consumers` > 0).
- Điều này làm tắc nghẽn cơ chế reply RPC → chính là lý do `report_state` timeout.

### Phân biệt ready vs unacked (rất quan trọng)

```bash
docker exec rabbitmq rabbitmqctl list_queues \
  name messages messages_ready messages_unacknowledged consumers \
  | grep -iE "scheduler|Port-1.10"
```

| Tình huống | Ý nghĩa |
|---|---|
| `messages_ready` cao, `unacked` = 0 | Consumer **không lấy** message (consumer "ma" / sai routing / queue sai loại) |
| `messages_unacknowledged` cao | Consumer **đã nhận nhưng không ack** (đang treo khi xử lý, hoặc message lỗi/poison) |

> Nếu `unacked` cao → **đừng vội purge**, vì purge xong vẫn tái diễn; phải đào vào
> consumer (nova-scheduler / cinder-scheduler / neutron-server).

### Kiểm tra connection còn sống không

```bash
docker exec rabbitmq rabbitmqctl list_connections name peer_host state timeout
```

Trong sự cố này, tất cả connection đều `running` (timeout 60s) → connection không chết.
Nhưng số lượng connection **rất nhiều** (>150 cho 1 node), gợi ý khả năng connection
leak hoặc service từng reconnect nhiều lần.

## 7. Vì sao restart RabbitMQ không dọn được queue

Thử restart RabbitMQ với kỳ vọng "queue tạm sẽ bị xóa khi restart":

```bash
docker restart rabbitmq
```

**Kết quả:** `scheduler_fanout` vẫn đứng nguyên ~13086 message sau restart.

### Diễn giải

- Ở mô hình RabbitMQ **cũ (3.x)**, fanout queue là transient/non-durable → restart sẽ
  xóa sạch.
- Nhưng ở **RabbitMQ 4.x** (đang chạy 4.0.9), mặc định **mọi queue là durable** và là
  **quorum queue**; fanout có thể được thay bằng **stream queue**. Các loại này
  **persistent qua restart** → message **không** bị xóa khi restart RabbitMQ.

→ Đây là lý do "restart cho sạch" **không ăn thua**.

## 8. `purge_queue` trả về `not_supported` — manh mối quyết định

Thử purge thẳng các queue tồn đọng:

```bash
docker exec rabbitmq rabbitmqctl purge_queue scheduler_fanout
# Error: not_supported

docker exec rabbitmq rabbitmqctl purge_queue cinder-scheduler_fanout
# Error: not_supported
```

### Diễn giải — đây là chìa khóa của cả vụ

`purge_queue` trả `not_supported` khi queue đó là **quorum queue** hoặc **stream** —
hai loại này **không hỗ trợ purge** bằng lệnh thông thường.

Kết hợp với `cluster_status` ở bước 5 (có `Flag: quorum_queue, state: enabled`), ta xác
định: **các fanout/transient queue đã bị tạo dưới dạng quorum/stream queue** — vốn
**không phù hợp** với bản chất broadcast tạm thời của chúng, và đây là kết quả của việc
nâng cấp lên RabbitMQ 4.x mà **chưa migrate đúng quy trình**.

## 9. Nguyên nhân gốc

### Tóm tắt

> Stack đã chạy **RabbitMQ 4.0.9**. Trong các bản Kolla gần đây (Epoxy), mặc định mọi
> queue là **quorum queue**, và fanout queue được thay bằng **stream queue**
> (điều khiển bởi `om_enable_rabbitmq_stream_fanout`, mặc định bật). Tuy nhiên, khi nâng
> RabbitMQ lên 4.0, **bắt buộc** phải migrate mọi queue sang loại durable **trước khi
> nâng cấp**, kèm theo việc **reset state RabbitMQ** và **tắt toàn bộ OpenStack service**
> để chúng không tạo lại queue cũ. Quy trình reset này đã **bị bỏ sót**, để lại các queue
> ở trạng thái lai/hỏng → message kẹt → RPC timeout → "Binding failed for port".

### Các biến cấu hình liên quan (trong `/etc/kolla/globals.yml`)

```yaml
om_enable_queue_manager: true
om_enable_rabbitmq_quorum_queues: true
om_enable_rabbitmq_transient_quorum_queue: true   # quorum cho cả transient/fanout
om_enable_rabbitmq_stream_fanout: true            # dùng stream thay fanout (mặc định bật ở Epoxy)
```

- `om_enable_rabbitmq_quorum_queues`: bật quorum queue cho các service queue.
- `om_enable_rabbitmq_transient_quorum_queue`: mở rộng quorum cho cả transient/fanout —
  giúp service chịu lỗi broker tốt hơn.
- `om_enable_rabbitmq_stream_fanout`: dùng **stream** (hiệu năng cao hơn) thay cho fanout;
  bật mặc định khi cả hai biến trên đều true.

### Bối cảnh lịch sử (vì sao có thay đổi này)

- Classic mirrored queue bị **deprecated** và loại bỏ ở RabbitMQ 4.0.
- Quorum queue (dựa trên thuật toán đồng thuận Raft) thay thế: đáng tin cậy hơn, an toàn
  hơn, nhanh hơn mirrored classic queue.
- Stream queue thay thế fanout với hiệu năng tốt hơn cho broadcast.
- **Hệ quả:** mọi lần nâng cấp lớn RabbitMQ (đặc biệt 3.x → 4.x) đều cần migration +
  reset state có kiểm soát.

## 10. Cách khắc phục DỨT ĐIỂM (reset state đúng quy trình)

> ⚠️ Đây là thao tác **gián đoạn control plane vài phút**. **VM đang chạy KHÔNG bị tắt**,
> nhưng API/tạo mới sẽ ngừng trong lúc reset. Trên cluster single-node, RabbitMQ vốn đã
> là single point of failure nên downtime này không làm thay đổi mức HA của hệ thống.

**Thứ tự cực kỳ quan trọng** — làm sai sẽ khiến service tạo lại queue hỏng.

### Bước 1 — Khai báo cấu hình & sinh lại config (chưa restart container)

Đảm bảo `/etc/kolla/globals.yml` có 4 biến ở mục 9, rồi:

```bash
kolla-ansible -i <inventory> genconfig
```

### Bước 2 — DỪNG toàn bộ OpenStack service dùng RabbitMQ

(Để chúng không tạo lại queue trong lúc reset — làm trên **tất cả** node.)

```bash
# Trên controller
docker stop nova_scheduler nova_conductor nova_api
docker stop neutron_server neutron_l3_agent neutron_dhcp_agent neutron_metadata_agent
docker stop cinder_scheduler cinder_volume cinder_api cinder_backup
docker stop heat_engine heat_api   # nếu có

# Trên từng compute/network node
docker stop nova_compute
docker stop neutron_openvswitch_agent
```

### Bước 3 — Reset state RabbitMQ (xóa sạch queue/exchange cũ bị hỏng)

```bash
docker exec rabbitmq rabbitmqctl stop_app
docker exec rabbitmq rabbitmqctl reset
docker exec rabbitmq rabbitmqctl start_app
```

> Trên single-node, `reset` an toàn vì RabbitMQ chỉ chứa message tạm, không có dữ liệu
> nghiệp vụ.

### Bước 4 — Khởi động lại toàn bộ service (chúng sẽ tạo lại queue đúng loại)

```bash
# Cách thủ công
docker start nova_scheduler nova_conductor nova_api nova_compute
docker start neutron_server neutron_openvswitch_agent neutron_l3_agent \
             neutron_dhcp_agent neutron_metadata_agent
docker start cinder_scheduler cinder_volume cinder_api cinder_backup

# HOẶC gọn & chuẩn hơn — để Kolla lo
kolla-ansible -i <inventory> reconfigure
```

### Bước 5 — Kiểm tra

```bash
docker exec rabbitmq rabbitmqctl list_queues name messages consumers | grep -iE "scheduler|fanout"
openstack network agent list      # tất cả phải Alive = True / :-)
openstack compute service list    # nova service phải up
```

`messages` phải dao động về 0; agent về alive; tạo VM/port lại bình thường.

## 11. Cách khắc phục "PHẪU THUẬT" (nhanh, chỉ chữa triệu chứng)

Dùng khi đang là production, cần gấp, chưa thể đặt cửa sổ bảo trì cho cả stack. Chỉ động
vào đúng service liên quan.

```bash
# 1) Force-delete các queue kẹt (kể cả khi còn consumer)
docker exec rabbitmq rabbitmqctl delete_queue scheduler_fanout --if-empty=false
docker exec rabbitmq rabbitmqctl delete_queue cinder-scheduler_fanout --if-empty=false
docker exec rabbitmq rabbitmqctl delete_queue neutron-vo-Port-1.10_fanout --if-empty=false

# 2) Restart đúng service sở hữu queue vừa xóa (để nó tạo lại queue)
docker restart nova_scheduler      # cho scheduler_fanout
docker restart cinder_scheduler    # cho cinder-scheduler_fanout
docker restart neutron_server      # cho neutron-vo-Port-*
```

### ⚠️ Cảnh báo quan trọng

- **Không thể chỉ "xóa" mà không restart.** Khi xóa queue đang được consume, service giữ
  channel cũ trỏ vào queue đã biến mất → rơi vào lỗi → vẫn phải restart service đó để nó
  đăng ký lại. "Xóa" và "restart" luôn đi liền nhau.
- Cách này **chỉ chữa triệu chứng**. Nếu gốc rễ là queue sai loại (quorum/stream lai
  hỏng do chưa migrate), service sẽ **tạo lại đúng cái queue hỏng** → vài hôm sau lại kẹt
  → vòng lặp vô tận. Muốn hết bệnh phải làm mục 10.

## 12. Bẫy thường gặp: RabbitMQ bị TẮT (ECONNREFUSED)

Trong lúc thao tác (ví dụ vừa `restart rabbitmq` hoặc `stop_app`), log service đầy:

```
(320) CONNECTION_FORCED - broker forced connection closure with reason 'shutdown'
...
[Errno 111] ECONNREFUSED   ← lặp lại, retry với backoff tăng dần 1s,2s,4s,7s...
```

### Diễn giải

- `shutdown` rồi `ECONNREFUSED` liên tục ở port 5672 = **không có gì lắng nghe** ở port
  đó nữa → **RabbitMQ container đã dừng và chưa chạy lại** (hoặc đang khởi động lại).
- Các lỗi `ECONNREFUSED` này là oslo.messaging **tự động retry** reconnect (backoff tăng
  dần). Khi RabbitMQ lên lại, chúng tự hết.

### Xử lý ngay

```bash
docker ps -a | grep rabbitmq          # xem STATUS: Up / Exited / Restarting
docker start rabbitmq                 # nếu Exited
docker logs --tail 80 rabbitmq        # nếu start xong vẫn chết -> tìm lỗi Mnesia/disk/erlang cookie
docker exec rabbitmq rabbitmqctl status | grep -i uptime   # xác nhận đã sống
```

> Nguyên tắc: **đưa RabbitMQ sống lại trước**, mọi việc khác (queue kẹt, reset...) tính
> sau. Khi broker chưa chạy thì không lệnh `rabbitmqctl` nào chạy được.

### Ghi chú phụ

Một dòng log khác có thể xuất hiện đồng thời:

```
<uuid> has allocations against this compute host but is not found in the database.
```

Đây là log của **nova/placement** (allocation mồ côi — instance đã xóa nhưng allocation
còn sót trong placement). **Không liên quan trực tiếp** tới RabbitMQ; nó chỉ xác nhận log
đang đến từ `nova_compute`. Xử lý riêng sau bằng `nova-manage placement
heal_allocations` / kiểm tra allocation mồ côi nếu cần.

## 13. Phòng ngừa & bài học rút ra

### Bài học chẩn đoán

1. **"Binding failed" gần như không bao giờ là lỗi gốc.** Luôn đào xuống tầng dưới:
   agent → RPC → RabbitMQ → DB.
2. **MessagingTimeout = vấn đề RPC**, không phải vấn đề neutron logic.
3. **RabbitMQ "trông khỏe" (no alarm, no partition) vẫn có thể nghẽn** ở tầng
   queue/connection — phải soi `list_queues` và phân biệt ready/unacked.
4. **`not_supported` khi purge = queue là quorum/stream** → dấu hiệu RabbitMQ 4.x +
   migration chưa hoàn tất.
5. **Một thao tác sai (restart/stop RabbitMQ giữa chừng) có thể làm broker chết** và sinh
   ra lớp lỗi mới (ECONNREFUSED) chồng lên lỗi cũ → luôn theo dõi trạng thái container.

### Phòng ngừa

- **Trước mọi nâng cấp RabbitMQ lớn (3.x → 4.x):** đọc kỹ release notes Kolla, thực hiện
  đầy đủ quy trình migrate quorum/durable queue + reset state. Đừng bỏ bước.
- **Tinh chỉnh heartbeat** để service tự reconnect sạch sau sự cố broker, tránh để lại
  queue/consumer "ma". Đặt trong `/etc/kolla/config/global.conf` (hoặc per-service):

  ```ini
  [oslo_messaging_rabbit]
  heartbeat_timeout_threshold = 60
  heartbeat_in_pthread = true
  ```

  Rồi `kolla-ansible -i <inventory> reconfigure`.

- **Single-node RabbitMQ là single point of failure.** Cân nhắc cluster 3 node + HA cho
  production thật sự.
- **Giám sát chủ động** số message tồn đọng trong fanout queue (Prometheus rabbitmq
  exporter) để bắt sớm trước khi nó làm sập RPC.

## 14. Bảng tra lệnh nhanh (cheat sheet)

### Chẩn đoán

```bash
# Agent / service status
openstack network agent list
openstack compute service list

# Log (Kolla mount ra host)
tail -f /var/log/kolla/neutron/neutron-server.log
tail -f /var/log/kolla/neutron/neutron-openvswitch-agent.log
docker logs --tail 100 -f neutron_server

# RabbitMQ tổng quan
docker exec rabbitmq rabbitmqctl cluster_status
docker exec rabbitmq rabbitmqctl status | grep -i uptime
docker stats --no-stream rabbitmq

# Queue & connection
docker exec rabbitmq rabbitmqctl list_queues name messages messages_ready messages_unacknowledged consumers
docker exec rabbitmq rabbitmqctl list_queues name messages consumers | grep -i fanout
docker exec rabbitmq rabbitmqctl list_connections name peer_host state timeout
docker exec rabbitmq rabbitmqctl list_consumers | grep -i <queue>

# Mạng tới broker
nc -zv <rabbitmq_ip> 5672
```

### Xử lý nhanh (phẫu thuật)

```bash
docker exec rabbitmq rabbitmqctl delete_queue <queue> --if-empty=false
docker restart <service_sở_hữu_queue>
```

### Xử lý dứt điểm (reset state)

```bash
kolla-ansible -i <inventory> genconfig
# stop tất cả OpenStack service (nova/neutron/cinder...) trên mọi node
docker exec rabbitmq rabbitmqctl stop_app
docker exec rabbitmq rabbitmqctl reset
docker exec rabbitmq rabbitmqctl start_app
kolla-ansible -i <inventory> reconfigure
```

### Cứu RabbitMQ chết

```bash
docker ps -a | grep rabbitmq
docker start rabbitmq
docker logs --tail 80 rabbitmq
```

## Phụ lục — Chuỗi nhân quả tóm tắt (đọc 30 giây)

```
RabbitMQ nâng lên 4.x nhưng chưa migrate/reset đúng quy trình
        ↓
fanout/transient queue bị tạo dưới dạng quorum/stream queue (sai bản chất)
        ↓
message broadcast không được consume/dọn đúng cách → tồn đọng hàng chục nghìn
        ↓
cơ chế reply RPC bị nghẽn
        ↓
OVS agent gọi report_state → MessagingTimeout (không nhận được reply)
        ↓
neutron-server coi agent là "down" (dù agent vẫn chạy)
        ↓
ML2 không tìm thấy agent "alive" trên host để bind port
        ↓
"Binding failed for port"   ← triệu chứng người dùng nhìn thấy
```

**Sửa gốc:** reset state RabbitMQ + restart toàn bộ service để tạo lại queue đúng loại
(mục 10).
