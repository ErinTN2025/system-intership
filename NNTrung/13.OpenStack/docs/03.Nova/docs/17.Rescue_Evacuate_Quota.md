## 1. Chuẩn bị môi trường chung

Nạp credentials admin (trên deploy node hoặc controller):

```bash
source /etc/kolla/admin-openrc.sh
```

Kiểm tra các dịch vụ lõi sẵn sàng:

```bash
openstack service list
openstack compute service list      # nova-compute, nova-conductor, nova-scheduler
openstack network agent list

openstack hypervisor list           # cho phần evacuate
```

Chuẩn bị tài nguyên nền tảng:

```bash

openstack image list

# Flavor

# Network + subnet nội bộ

# Keypair

# Security group: cho phép SSH + ICMP
openstack security group create rescue-sg
openstack security group rule create --proto icmp rescue-sg
openstack security group rule create --proto tcp --dst-port 22 rescue-sg
```

## 2. Rescue Localdisk

### 2.1. Khái niệm

Rescue là chế độ "cứu hộ" một VM bị hỏng (không boot được, hỏng bootloader, sai `fstab`, lỡ xóa file hệ thống...). Nova tạm dừng VM, boot lại bằng một **rescue image** tạm thời, đồng thời gắn ổ gốc của VM như một disk phụ để bạn mount vào sửa lỗi.

**Localdisk** = instance boot từ **ephemeral disk** nằm local trên compute node (do Nova quản lý), không dùng Cinder volume. Đây là rescue **legacy mode** (mặc định), hỗ trợ đầy đủ và đơn giản nhất.

### 2.2. Tạo instance boot từ local disk

Điểm mấu chốt: **không** boot from volume → Nova tạo ephemeral disk local.

```bash
openstack server create \
  --image cirros \
  --flavor m1.small \
  --network demo-net \
  --key-name mykey \
  --security-group rescue-sg \
  vm-localdisk

# Xác nhận KHÔNG có volume gắn vào (volumes_attached rỗng)
openstack server show vm-localdisk -c volumes_attached -c status
```

### 2.3. Thực hiện rescue

```bash
# Cách 1: dùng chính image gốc làm rescue
openstack server rescue vm-localdisk

# Cách 2 (khuyến nghị): chỉ định rescue image riêng
openstack server rescue --image cirros vm-localdisk

# Theo dõi trạng thái -> RESCUE
openstack server show vm-localdisk -f value -c status
```

### 2.4. Vào môi trường rescue và sửa lỗi

```bash
openstack console url show vm-localdisk
```

Trong VM rescue (ổ gốc thường là `/dev/vdb`):

```bash
lsblk
sudo mount /dev/vdb1 /mnt
ls /mnt
# ... sửa fstab / bootloader / xóa file lỗi ...
sudo umount /mnt
```

### 2.5. Unrescue

```bash
openstack server unrescue vm-localdisk
openstack server show vm-localdisk -f value -c status   # -> ACTIVE
```

## 3. Rescue Volume (Boot-from-Volume)

### 3.1. Bối cảnh kỹ thuật — đọc kỹ trước khi làm

Rescue **legacy** (mặc định) **KHÔNG** hỗ trợ instance boot-from-volume. Để rescue được, Nova có một chế độ thứ hai gọi là **stable device rescue**, có từ Ussuri (21.0.0) và vẫn áp dụng cho 2025.1.

Có **hai điều kiện bắt buộc đồng thời**:

| Điều kiện | Giá trị | Ghi chú |
|---|---|---|
| API microversion | `2.87` trở lên | Để API cho phép rescue BFV instance |
| Property trên rescue image | `hw_rescue_device` và/hoặc `hw_rescue_bus` | Kích hoạt chế độ stable device |

- `hw_rescue_device`: loại thiết bị rescue — `disk`, `cdrom`, hoặc `floppy`.
- `hw_rescue_bus`: loại bus — `virtio`, `scsi`, `ide`, hoặc `usb`.

> **Cảnh báo thực tế (bug đã ghi nhận):** với một số guest (vd. Ubuntu Focal), `hw_rescue_bus=scsi` có thể gây lỗi **"No bootable device"** trên console, trong khi `usb` hoạt động ổn định. Lab này dùng `usb` làm lựa chọn an toàn.

> Nếu **thiếu** property khi rescue BFV instance → API call thất bại, instance vào **ERROR**. Đây là lỗi phổ biến nhất của bài lab này.

### 3.2. Tạo rescue image có property

```bash
# Tạo image rescue chuyên dụng
openstack image create \
  --disk-format qcow2 \
  --container-format bare \
  --file cirros-0.6.2-x86_64-disk.img \
  --property hw_rescue_device=disk \
  --property hw_rescue_bus=usb \
  rescue-image

openstack image show rescue-image -c properties
```

Nếu dùng image có sẵn, chỉ cần set property:

```bash
openstack image set --property hw_rescue_device=disk --property hw_rescue_bus=usb <image-id>
```

### 3.3. Tạo instance boot-from-volume

```bash
# Tạo bootable volume từ image
openstack volume create --image cirros --size 5 --bootable rootvol
openstack volume show rootvol -f value -c status   # chờ "available"

# Tạo instance boot từ volume
openstack server create \
  --volume rootvol \
  --flavor m1.small \
  --network demo-net \
  --key-name mykey \
  --security-group rescue-sg \
  vm-volume

openstack server show vm-volume -c volumes_attached   # phải có volume
```

### 3.4. Rescue instance volume-backed

```bash
openstack --os-compute-api-version 2.87 server rescue \
  --image rescue-image \
  vm-volume

openstack server show vm-volume -f value -c status   # -> RESCUE
```

### 3.5. Vào rescue, sửa lỗi, unrescue

Ở chế độ stable device, mọi thiết bị gốc được giữ nguyên thứ tự, rescue image gắn cuối cùng và đánh dấu bootable.

```bash
openstack console url show vm-volume
```

Trong VM:

```bash
lsblk
# Root volume gốc thường là /dev/sda (hoặc vda); rescue device là thiết bị usb cuối
sudo mount /dev/sda1 /mnt
# ... sửa lỗi ...
sudo umount /mnt
```

Quay lại:

```bash
openstack server unrescue vm-volume
openstack server show vm-volume -f value -c status   # -> ACTIVE
```

### 3.6. So sánh localdisk vs volume

| Tiêu chí | rescue-localdisk | rescue-volume |
|---|---|---|
| Backend root disk | Ephemeral (Nova, local compute) | Cinder volume |
| Chế độ rescue | Legacy (mặc định) | Stable device (bắt buộc) |
| Microversion | Mặc định OK | Cần `2.87+` |
| Property image | Không bắt buộc | **Bắt buộc** `hw_rescue_device`/`hw_rescue_bus` |
| Image rescue | Tùy chọn | Nên/cần chỉ định |

## 4. Evacuate

### 4.1. Khái niệm

**Evacuate** dùng khi một compute node **đã chết** (hỏng phần cứng, mất điện, kernel panic...). Admin "sơ tán" các instance từ node hỏng sang node khác, **giữ nguyên** danh tính (UUID), volume, network và trạng thái VM.

Bản chất kỹ thuật: evacuate gọi lại flow **rebuild** với cờ `recreate`. Nếu disk nằm trên **shared storage** (Ceph, NFS), dữ liệu được giữ nguyên; nếu **không** có shared storage, instance được **dựng lại từ image gốc** (dữ liệu ephemeral mất, nhưng volume Cinder vẫn còn).

> **Điều kiện sống còn:** node nguồn phải **THẬT SỰ down**. Nếu node vẫn chạy, evacuate sẽ **thất bại** (để tránh hai bản VM cùng ghi vào một disk → hỏng dữ liệu). Luôn xác minh node đã chết trước khi evacuate.

### 4.2. Kiểm tra trạng thái compute node

```bash
# Xem dịch vụ nova-compute — node hỏng sẽ ở State "down"
openstack compute service list --service nova-compute

# Liệt kê instance trên node hỏng
openstack server list --all-projects --host compute02 --long
```

Khi node down, `nova-compute` trên node đó hiện `State = down` (sau khoảng `service_down_time`, mặc định ~60s).

### 4.3. Evacuate một instance

Lưu ý: lệnh evacuate hiện nằm trong **nova client** (`python-novaclient`). OpenStackClient (`openstack`) chưa có lệnh evacuate tương đương đầy đủ, nên ta dùng `nova`.

```bash
# Để scheduler tự chọn host đích (khuyến nghị)
nova evacuate vm-on-dead-node

# Hoặc chỉ định host đích cụ thể
nova evacuate vm-on-dead-node compute03
```

Về tham số `--on-shared-storage`: từ compute API **2.14** trở đi **không cần** chỉ định cờ này nữa — Nova **tự phát hiện** node đích có shared storage hay không. (Với 2025.1, mặc định bạn không cần cờ này.)

Đặt mật khẩu mới cho VM (tùy chọn) — nếu không chỉ định, một mật khẩu sẽ được sinh tự động và in ra:

```bash
nova evacuate --password <new_pwd> vm-on-dead-node compute03
```

### 4.4. Evacuate toàn bộ instance trên một node (host-evacuate)

Khi node hỏng có nhiều VM, sơ tán hàng loạt:

```bash
# Sơ tán tất cả instance khỏi compute02, để scheduler chọn đích
nova host-evacuate compute02

# Hoặc dồn tất cả về một host đích
nova host-evacuate --target_host compute03 compute02
```

> Chỉ dùng `host-evacuate` khi node **không còn instance nào đang chạy** (node đã down hẳn). Không dùng khi host còn VM sống.

### 4.5. Theo dõi và xác minh

```bash
# Kiểm tra instance đã chuyển host
openstack server show vm-on-dead-node -c OS-EXT-SRV-ATTR:host -c status

# Xem tiến trình migration kiểu evacuation
openstack server migration list --server vm-on-dead-node
```

### 4.6. Sau khi node hỏng hồi phục

Khi node cũ online trở lại, Nova (trong `init_host`) **tự dọn dẹp** các instance "mồ côi" còn sót (xóa domain libvirt cũ và network của chúng) để tránh xung đột với bản đã evacuate. Tuy nhiên nên kiểm tra thủ công:

```bash
# Trên node vừa hồi phục — không nên còn domain của VM đã evacuate
sudo virsh list --all

# Nếu nova-compute không tự dọn, kiểm tra log
sudo tail -f /var/log/kolla/nova/nova-compute.log
```

>  **Cực kỳ quan trọng:** đừng để node hỏng bật `nova-compute` lên *trước khi* evacuate hoàn tất ở node đích. Việc bật sớm với cấu hình sai (vd. trỏ nhầm hypervisor) có thể khiến Nova tưởng instance đã được sơ tán và **xóa dữ liệu** (kể cả trên shared storage).

### 4.7. So sánh evacuate vs migrate vs live-migrate

| Thao tác | Node nguồn | Downtime | Tình huống dùng |
|---|---|---|---|
| `live-migrate` | Đang sống | Gần như không | Bảo trì có kế hoạch |
| `migrate` (cold) | Đang sống | Có (VM tắt rồi chuyển) | Đổi flavor/rebalance |
| `evacuate` | **Đã chết** | Có (dựng lại ở node mới) | Khôi phục sau sự cố node |

## 5. Quota — Unified Limits

### 5.1. Bối cảnh: legacy quota vs unified limits

Từ Nova **28.0.0 (2023.2 Bobcat)**, khuyến nghị chuyển sang **Keystone Unified Limits** thay cho legacy quota (lưu trong DB của Nova). Với **2025.1**, unified limits là hướng đi chính thức.

Mô hình unified limits có 3 bước thực thi:
1. **Lấy giới hạn** từ Keystone Unified Limits API.
2. **Đếm mức sử dụng** từ Placement API service.
3. **Thực thi** cục bộ bằng thư viện `oslo.limit`.

Thuật ngữ mới:
- **Registered limit** = giới hạn mặc định/toàn cục, áp cho mọi project.
- **Limit** = giới hạn riêng theo từng project (đè lên registered limit).

> Tên resource theo Placement dùng tiền tố `class:` — ví dụ: `class:VCPU`, `class:PCPU`, `class:MEMORY_MB`, `class:DISK_GB`, `class:VGPU`. Các resource khác như số server dùng tên trực tiếp: `servers`.

### 5.2. Kiểm tra hệ thống đang dùng quota driver nào

```bash
# Trên controller, xem cấu hình nova
grep -i driver /etc/kolla/nova-api/nova.conf | grep -i quota
```

Nếu thấy `driver = nova.quota.UnifiedLimitsDriver` → đang dùng unified limits. Nếu là `DbQuotaDriver` (hoặc trống) → vẫn legacy.

### 5.3. (Cấu hình) Bật Unified Limits trong Kolla-Ansible

> Phần này là cấu hình deploy. Nếu lab của bạn đã bật sẵn, bỏ qua sang 5.4.

Trên deploy node, dùng cơ chế **config override** của Kolla. Tạo/sửa file:

```bash
# /etc/kolla/config/nova/nova-api.conf  và  nova-conductor.conf
```

Nội dung override:

```ini
[quota]
driver = nova.quota.UnifiedLimitsDriver

[oslo_limit]
username = nova
user_domain_name = Default
auth_url = http://<keystone-internal-vip>:5000/v3
auth_type = password
password = <nova_service_password>
system_scope = all
endpoint_id = <NOVA_ENDPOINT_ID>
```

Lấy `endpoint_id` của Nova:

```bash
openstack endpoint list --service nova -f value -c ID
```

Cấp role `reader` phạm vi system cho service user `nova` (bắt buộc để `oslo.limit` đọc được limit từ Keystone):

```bash
openstack role add --user nova --user-domain Default --system all reader
```

Triển khai lại cấu hình:

```bash
kolla-ansible -i multinode reconfigure -t nova
```

### 5.4. Di chuyển limit cũ sang unified limits

Nova cung cấp công cụ `nova-manage` để copy limit legacy từ DB sang Keystone. Trong Kolla, chạy bên trong container `nova_conductor`:

```bash
# Migrate default (registered) limits + phát hiện resource còn thiếu limit
docker exec -it nova_conductor nova-manage limits migrate_to_unified_limits

# Migrate limit theo từng project
docker exec -it nova_conductor \
  nova-manage limits migrate_to_unified_limits --project-id <PROJECT_ID>
```

Lệnh sẽ in bảng cho biết resource nào **chưa** có limit trong Keystone — cần đặt thủ công ở bước 5.5.

### 5.5. Quản lý registered limit (mặc định toàn cục)

```bash
# Đặt giới hạn mặc định: tối đa 10 server cho mọi project
openstack registered limit create --service nova --default-limit 10 servers

# Giới hạn vCPU (theo resource class của Placement)
openstack registered limit create --service nova --default-limit 20 class:VCPU

# Giới hạn RAM (MB)
openstack registered limit create --service nova --default-limit 51200 class:MEMORY_MB

# Liệt kê
openstack registered limit list
```

> Yêu cầu token **system-scoped** để tạo/sửa registered limit (theo policy mặc định của Keystone).

### 5.6. Quản lý limit theo project (đè mặc định)

```bash
# Project "demo" được phép tối đa 5 server
openstack limit create --service nova --project demo --resource-limit 5 servers

# Project "demo" tối đa 8 vCPU
openstack limit create --service nova --project demo --resource-limit 8 class:VCPU

# Liệt kê limit của project
openstack limit list --service nova --project demo
```

### 5.7. Thứ tự áp dụng giới hạn (quan trọng)

Khi kiểm tra quota cho một resource trong một project, Nova xét theo thứ tự:

1. Có **limit riêng theo project** trong Keystone? → dùng giá trị đó.
2. Nếu không → dùng **registered (default) limit**.
3. Nếu cũng không → `oslo.limit` coi giới hạn là **0** → quota check **thất bại**, báo "quota exceeded".

>  Hệ quả: với unified limits, **bất kỳ resource nào chưa được đặt limit (cả project lẫn default) sẽ bị coi là 0** và chặn mọi request. Vì vậy sau khi bật unified limits, **phải** đặt registered limit cho mọi resource bạn muốn cho phép. (Từ 2025.1 có config option cho phép nới lỏng hành vi với resource chưa đặt limit — xem mục lỗi thường gặp.)

### 5.8. Xem mức sử dụng và giới hạn hiện tại

```bash
# Xem usage thực tế (đếm từ Placement) + giới hạn của project hiện tại
openstack limit list --service nova

# Với legacy quota (nếu hệ thống chưa chuyển), vẫn dùng:
openstack quota show <project>
openstack quota set --instances 20 --cores 40 --ram 51200 <project>
```

### 5.9. recheck_quota

Mặc định `quota.recheck_quota = True`: Nova thực hiện **lần kiểm tra quota thứ hai** sau khi đã cấp phát resource. Lần đầu kiểm tra *trước* khi cấp phát; lần hai đảm bảo không vượt quota khi có nhiều request song song (race condition). Có thể tắt để giảm tải nhưng sẽ giảm độ chặt chẽ.

## 6. Phụ lục: lỗi thường gặp & mẹo vận hành

### Rescue

- **Instance vào ERROR khi rescue volume:** quên `--os-compute-api-version 2.87` hoặc quên property trên image. Khắc phục:
  ```bash
  openstack server unrescue vm-volume
  # nếu kẹt trạng thái:
  openstack server set --state active vm-volume
  ```
  Sau đó kiểm tra lại property image và thêm microversion.

- **"No bootable device" trên console (volume-backed):** đổi `hw_rescue_bus` từ `scsi` sang `usb`, rồi unrescue/rescue lại.

- **Kiểm tra trait COMPUTE_RESCUE_BFV** (cụm nhiều backend lẫn lộn):
  ```bash
  openstack --os-placement-api-version 1.6 resource provider trait list <rp-uuid>
  ```
  Với Kolla mặc định (libvirt+KVM), trait này được report sẵn, không cần cấu hình thêm.

### Evacuate

- **Evacuate thất bại "host not down":** node nguồn chưa thực sự down. Xác minh `openstack compute service list` thấy `State = down`. Tuyệt đối không evacuate khi node còn sống.

- **Dữ liệu mất sau evacuate:** instance không nằm trên shared storage và không boot-from-volume → ephemeral disk được dựng lại từ image. Đây là hành vi đúng; muốn giữ data phải dùng Ceph/shared storage hoặc BFV.

- **VM "ma" trên node hồi phục:** nếu `nova-compute` không tự dọn, kiểm tra `virsh list --all` và log `nova-compute.log`.

### Quota / Unified Limits

- **Mọi request bị "quota exceeded" sau khi bật unified limits:** do resource chưa có registered limit (mặc định = 0). Đặt registered limit cho tất cả resource cần thiết (`servers`, `class:VCPU`, `class:MEMORY_MB`, `class:DISK_GB`...).

- **Nới lỏng resource chưa đặt limit (2025.1):** từ 2025.1 có config option trong `[quota]` cho phép chỉ enforce một số resource cụ thể, hoặc enforce tất cả *trừ* một số resource — giúp tránh việc bỏ sót một limit làm sập toàn bộ API. Tham khảo tài liệu `[quota]` của bản 2025.1 để biết tên option chính xác trước khi áp dụng.

- **Không tạo được registered limit:** thiếu token system-scoped. Dùng user có role `admin`/`reader` ở `--system all`.
