# PHẦN A — GIAI ĐOẠN 8: CINDER MULTI-BACKEND

## 0. CINDER LÀ GÌ + VÌ SAO MULTI-BACKEND (đọc 5 phút)

**Cinder = "tủ ổ cứng gắn ngoài" cho VM.** Ephemeral disk (Phase 6) sống chết cùng VM;
Cinder volume là block device **độc lập vòng đời** — xoá VM, volume vẫn còn, attach sang VM
khác được, snapshot/backup được.

**4 thành phần Cinder:**

| Tên | Vai trò | Chạy ở đâu | Port |
|---|---|---|---|
| cinder-api | REST API entrypoint | 3 controller | 8776 |
| cinder-scheduler | Chọn backend nào nhận volume mới | 3 controller | (RabbitMQ) |
| cinder-volume | Driver nói chuyện với storage thật | **cinder01** | (RabbitMQ) |
| cinder-backup | Backup volume ra object store | (không dựng — chưa có Swift) | — |

**Dòng đời 1 request `openstack volume create --type ceph`:**

```
client → cinder-api (8776, qua VIP) → cinder-scheduler (đọc extra_spec volume_backend_name
       của type 'ceph' → match backend [ceph] trên cinder01) → cinder-volume@ceph
       → RBDDriver tạo image trong pool volumes
```

**Multi-backend = 1 tiến trình cinder-volume chạy NHIỀU driver song song.** Mỗi backend là 1
section trong `cinder.conf`, khai báo qua `enabled_backends = lvm,nfs,ceph`. Mỗi backend
tự đăng ký thành 1 "host" riêng dạng `cinder01@lvm`, `cinder01@nfs`, `cinder01@ceph`.
Scheduler route bằng **volume type**: type có extra_spec `volume_backend_name=X` → chỉ
backend khai `volume_backend_name = X` mới được nhận.

**3 backend của lab này — và lý do tồn tại của từng cái:**

| Backend | Data nằm đâu | Attach kiểu gì | Dạy ta cái gì |
|---|---|---|---|
| **LVM** | LV trong VG `cinder-volumes` trên đĩa cinder01 | iSCSI (tgt export → compute open-iscsi login) | Pattern storage truyền thống: local disk + iSCSI target |
| **NFS** | File raw trong export NFS của cinder01 | Compute mount NFS, qemu mở file trực tiếp | Pattern file-based: volume = file, không phải block |
| **Ceph RBD** | RBD image trong pool `volumes` | libvirt nói RBD protocol thẳng tới OSD | Pattern phân tán: data HA, không đi qua cinder01 khi IO |

**Điểm kiến trúc PHẢI hiểu trước khi gõ — data path khác nhau hoàn toàn:**
- **LVM/NFS:** mọi byte IO của VM đi xuyên qua cinder01. cinder01 chết = volume LVM/NFS
  đứng IO ngay lập tức. Đây là SPOF **cố ý giữ lại** để thấy sự khác biệt.
- **Ceph:** cinder-volume chỉ làm control plane (tạo/xoá/snapshot RBD image). Khi VM đọc/ghi,
  libvirt trên compute node nói chuyện **trực tiếp** với 3 OSD — cinder01 chết, VM đang chạy
  với volume ceph **vẫn IO bình thường**, chỉ không tạo/xoá volume mới được. Sẽ test đúng
  điều này ở mục 9.6.

**Vì sao cinder-volume không chạy 3 bản trên 3 controller cho HA?** Vì cinder-volume giữ
state (mỗi volume thuộc về đúng 1 host `cinder01@backend`). Chạy active/active cần
coordination (etcd/tooz) hoặc active/passive qua Pacemaker — ngoài phạm vi lab. Chấp nhận:
**API/scheduler HA, volume service single node.** Trade-off ghi rõ để bảo vệ trước hội đồng.

**Tái sử dụng từ Phase 6 (KHÔNG tạo lại):**
- `client.cinder` + caps — đã đúng chuẩn cho cả Cinder lẫn Nova.
- Libvirt secret UUID trên 3 compute — `rbd_secret_uuid` trong cinder.conf **phải trùng**
  UUID này, vì khi attach, Nova đưa connection info cho libvirt, libvirt tra secret local.

## 1. CHUẨN BỊ CINDER01 — ĐĨA LVM + NFS SERVER

### 1.1. Verify đĩa từ deployer

```bash
# trên deployer
ssh cinder01 '
  hostname
  lsblk
  sudo blkid /dev/vdb 2>&1 || echo "(no signature - good)"
  chronyc tracking | grep -E "Reference|Leap"
  free -h | head -2
'
```

Cần: đĩa thứ 2 (giả định `vdb`, 40G) **trống, không partition, không signature**. Nếu node
của bạn có 2 đĩa rời (`vdb` + `vdc`) thì sướng hơn — xem ghi chú cuối 1.2. Nếu `vdb` có dấu
vết cũ, wipe như bài Ceph Phase 4.5 mục 1.2 (`wipefs -af` + `sgdisk --zap-all`).

### 1.2. Chia vdb làm đôi: vdb1 cho LVM, vdb2 cho NFS

```bash
ssh cinder01
# Trong shell cinder01:
sudo parted -s /dev/vdb mklabel gpt
sudo parted -s /dev/vdb mkpart lvm 1MiB 50%
sudo parted -s /dev/vdb mkpart nfs 50% 100%
lsblk /dev/vdb
# mong đợi: vdb1 ~20G, vdb2 ~20G
```

> Nếu có 2 đĩa rời: bỏ qua parted, dùng nguyên `vdb` cho LVM và nguyên `vdc` cho NFS —
> thay `vdb1`→`vdb`, `vdb2`→`vdc` trong các lệnh dưới.

### 1.3. LVM: tạo PV + VG `cinder-volumes`

```bash
# vẫn trong shell cinder01
sudo apt update
sudo apt -y install lvm2 thin-provisioning-tools
sudo pvcreate /dev/vdb1
sudo vgcreate cinder-volumes /dev/vdb1
sudo vgs
# mong đợi: VG cinder-volumes, VFree ~20G
```

### 1.4. Giới hạn LVM scan đúng đĩa của Cinder

Mặc định LVM scan **mọi** block device. Khi Cinder tạo LV, bên trong LV đó VM có thể tự tạo
LVM của nó → host scan thấy → loạn metadata. Khoá filter chỉ nhận `vda` (OS) + `vdb1`:

```bash
sudo sed -i 's|# filter = \[ "a|filter = [ "a|' /etc/lvm/lvm.conf 2>/dev/null || true
```

Cách chắc chắn hơn: mở đúng section `devices {` và đặt filter bằng tay. Dùng tee ghi 1 file
conf riêng KHÔNG ăn được với lvm.conf, nên đây là chỗ hiếm hoi phải sửa trực tiếp — nhưng
file này là ASCII thuần nên an toàn:

```bash
sudo python3 - <<'PYEOF'
import re
p = '/etc/lvm/lvm.conf'
s = open(p).read()
line = '        filter = [ "a|^/dev/vda|", "a|^/dev/vdb1$|", "r|.*|" ]\n'
if 'filter = [ "a|^/dev/vda' not in s:
    s = re.sub(r'(devices\s*\{)', r'\1\n' + line, s, count=1)
    open(p, 'w').write(s)
    print('filter inserted')
else:
    print('filter already present')
PYEOF

sudo grep -n 'filter = \[ "a' /etc/lvm/lvm.conf
sudo vgs    # vẫn phải thấy cinder-volumes (nếu mất = filter sai, sửa lại)
```

> Đổi `vda` nếu đĩa OS của bạn tên khác (`sda` → sửa pattern tương ứng). Quy tắc:
> **accept đĩa OS + đĩa PV của Cinder, reject tất cả còn lại** (`r|.*|`).

### 1.5. NFS server trên chính cinder01

```bash
# vẫn trong shell cinder01
sudo apt -y install nfs-kernel-server
sudo mkfs.ext4 /dev/vdb2
sudo mkdir -p /srv/nfs/cinder

# mount persistent qua fstab (dùng UUID cho chắc)
UUID=$(sudo blkid -s UUID -o value /dev/vdb2)
echo "UUID=$UUID /srv/nfs/cinder ext4 defaults 0 2" | sudo tee -a /etc/fstab
sudo systemctl daemon-reload
sudo mount -a
df -h /srv/nfs/cinder        # ~20G

# export cho subnet cluster
echo '/srv/nfs/cinder 192.168.70.0/24(rw,sync,no_subtree_check,no_root_squash)' | sudo tee -a /etc/exports
sudo exportfs -ra
sudo exportfs -v             # thấy /srv/nfs/cinder với các flag trên
exit
```

> `no_root_squash` bắt buộc: cinder-volume (root) và qemu trên compute cần full quyền trên
> file volume. Lab nội bộ + security group siết rồi nên chấp nhận.

### 1.6. Phân phối ceph.conf + cinder keyring sang cinder01

Phase 6 mới phát keyring cho controller + compute. cinder01 **chưa có** — RBDDriver sẽ chết
ngay khi start nếu thiếu:

```bash
# trên deployer
ssh cinder01 'sudo mkdir -p /etc/ceph'
sudo scp /etc/ceph/ceph.conf root@cinder01:/etc/ceph/ceph.conf

# tái xuất keyring client.cinder từ cụm Ceph (deployer có admin keyring)
sudo ceph auth get client.cinder -o /tmp/ceph.client.cinder.keyring
sudo scp /tmp/ceph.client.cinder.keyring root@cinder01:/etc/ceph/ceph.client.cinder.keyring
sudo rm /tmp/ceph.client.cinder.keyring
```

> `ceph auth get` chỉ **đọc lại** key sẵn có — không tạo key mới, không làm lệch key đã nằm
> trong libvirt secret trên compute. TUYỆT ĐỐI không dùng `get-or-create` với caps khác đi —
> caps lệch là Ceph báo lỗi, còn `auth caps` ghi đè là hỏng luôn quyền của Nova.

Bài học cũ áp dụng nguyên xi: **tee + heredoc, ASCII only, bind IP node** (không wildcard
để khỏi chiếm cổng 8776 của VIP):

```bash
ssh controller01 "sudo cp /etc/cinder/cinder.conf /etc/cinder/cinder.conf.bak"

ssh controller01 "sudo tee /etc/cinder/cinder.conf > /dev/null <<'EOF'
[DEFAULT]
my_ip = 192.168.70.122
osapi_volume_listen = 192.168.70.122
osapi_volume_listen_port = 8776
auth_strategy = keystone
transport_url = rabbit://openstack:RABBIT_PASS@controller01:5672,openstack:RABBIT_PASS@controller02:5672,openstack:RABBIT_PASS@controller03:5672
default_volume_type = ceph
state_path = /var/lib/cinder

[database]
connection = mysql+pymysql://cinder:CINDER_DBPASS@192.168.70.120/cinder
max_retries = -1

[keystone_authtoken]
www_authenticate_uri = http://192.168.70.120:5000
auth_url = http://192.168.70.120:5000
memcached_servers = 192.168.70.122:11211,192.168.70.127:11211,192.168.70.119:11211
auth_type = password
project_domain_name = Default
user_domain_name = Default
project_name = service
username = cinder
password = CINDER_PASS

[oslo_concurrency]
lock_path = /var/lib/cinder/tmp
EOF"

ssh controller01 "sudo chown root:cinder /etc/cinder/cinder.conf && sudo chmod 0640 /etc/cinder/cinder.conf"
```

**Lặp cho controller02** (`my_ip` + `osapi_volume_listen` = `.127`) **và controller03**
(`.119`). 2 chỗ đổi IP mỗi node. Thay `RABBIT_PASS`, `CINDER_DBPASS`, `CINDER_PASS` thật.

> `default_volume_type = ceph` — request không khai `--type` sẽ rơi vào ceph (backend
> "xịn" nhất). Type `ceph` tạo ở mục 7; trước đó tạo volume không khai type sẽ lỗi
> "default type not found" — đúng thứ tự là được.

> Ubuntu UCA 2025.1 ship `cinder-api` là **systemd service standalone** (giống nova-api,
> không qua Apache) → bind bằng `osapi_volume_listen` trong config là đủ, KHÔNG cần vhost
> Apache. Verify sau khi start: `systemctl status cinder-api` thấy `active (running)` và
> `ss -lntp | grep 8776` ra đúng IP node. Nếu bản UCA của bạn lại đẩy qua Apache
> (`systemctl status` không thấy service) thì làm vhost override Listen IP:8776 y hệt
> pattern Placement Phase 6 mục 3.4.

- `target_ip_address = .117` — IP cinder01 in vào portal iSCSI; compute sẽ login về đây.
- `nas_secure_file_*= false` — chế độ "secure" đòi mapping uid/gid phức tạp giữa NFS server
  và client; lab dùng `no_root_squash` nên tắt cho khỏi vướng `Permission denied` vu vơ.
- `nfs_sparsed_volumes = true` — file volume tạo sparse (truncate), 20G NFS chứa được nhiều
  volume "20G" miễn chưa ghi đầy. Thin provisioning kiểu nghèo.
- `rbd_flatten_volume_from_snapshot = false` + `rbd_max_clone_depth = 5` — giữ CoW chain khi
  clone volume từ snapshot, flatten tự động khi chain sâu quá 5.

## 9. TROUBLESHOOTING PHASE 8

### a. `cinder01@ceph` down ngay sau start
`journalctl -u cinder-volume | grep -i rbd`. 90% là keyring: quyền không phải 0640
root:cinder, hoặc thiếu `python3-rbd`. Test chữa cháy: lệnh `sudo -u cinder rbd ...` ở 5.2.

### b. Tạo volume lvm báo `No valid backend`
- `vgs` trên cinder01 còn thấy `cinder-volumes`? Filter lvm.conf (1.4) reject nhầm vdb1?
- Scheduler log: `ssh controller01 'sudo journalctl -u cinder-scheduler -n 30 --no-pager'`
  — tìm `filtered out`, thường do extra_spec type gõ sai hoa/thường (`LVM_iSCSI`).

### c. Attach LVM treo `attaching` rồi rollback
- tgt include (5.6) thiếu → `ssh cinder01 'sudo tgtadm --mode target --op show'` rỗng.
- compute thiếu open-iscsi / iscsid chết (mục 6).
- Security group cloud chặn 3260? `nc -zv cinder01 3260` từ compute phải succeeded.

### d. Attach NFS báo `Permission denied`
Export thiếu `no_root_squash`, hoặc quên `nas_secure_file_* = false`. Sửa rồi restart
cinder-volume + detach/attach lại.

### e. Volume ceph attach báo `Secret not found` từ libvirt
`rbd_secret_uuid` trong cinder.conf không trùng `virsh secret-list` trên compute. Sửa
cinder.conf (5.5), restart cinder-volume, attach lại.

### f. `openstack volume list` treo lâu rồi timeout
HAProxy backend cinder_api có UP không (4.5)? cinder-api bind nhầm IP (4.2)?

            '192.168.70.122:11211',

## 8. TROUBLESHOOTING PHASE 9

### a. Login xong bị đá ngược về trang login (loop)
Bẫy 1 hoặc 2: Memcached LOCATION sai IP, hoặc SECRET_KEY 3 node khác nhau. Verify:
```bash
for h in controller01 controller02 controller03; do
  ssh $h 'sudo grep -E "^SECRET_KEY" /etc/openstack-dashboard/local_settings.py | tail -1'
done
# 3 dong phai GIONG HET nhau (assignment cuoi file la cai an)
```

### b. `CSRF verification failed`
Thiếu origin trong `CSRF_TRUSTED_ORIGINS` (đặc biệt khi vào qua tunnel `localhost:8080`).
Thêm origin, reload apache2.

### c. Trang trắng / 500 Internal Server Error
`sudo tail -50 /var/log/apache2/error.log` — lỗi Python trong local_settings (block append
hỏng indent), hoặc thiếu `python3-pymemcache` (gói dashboard kéo theo, nhưng nếu thiếu:
`sudo apt -y install python3-pymemcache`).

### d. Login báo `Unable to authenticate` dù pass đúng
`OPENSTACK_KEYSTONE_URL` gõ sai, hoặc controller đang phục vụ không gọi được VIP:5000.
Test từ chính controller: `curl -s http://192.168.70.120:5000/v3 | head -1`.

### e. Tab Volumes báo lỗi nhưng tab khác bình thường
Horizon hỏi endpoint block-storage — kiểm tra `openstack catalog show block-storage` và
HAProxy cinder_api UP. Horizon chỉ phản chiếu sức khoẻ API phía sau.

### f. Console VNC màn hình đen / disconnect
novncproxy qua VIP:6080 — HAProxy phải `balance source` (đã set Phase 6). Check
`openstack console url show vm1` ra URL chứa `192.168.70.120:6080`.
