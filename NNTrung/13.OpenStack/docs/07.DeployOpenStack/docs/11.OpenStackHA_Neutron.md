# GIAI ĐOẠN 7 — NEUTRON OVN HA + SELF-SERVICE NETWORKING

> **Tiền đề:** Phase 0–6 xanh. VIP `.120` + Keystone + Glance (Ceph) + Placement + Nova
> đã chạy. `nova-compute` đã lên ở compute01/02/03 và báo `up` trong `openstack compute service list`.
> RabbitMQ + Galera + HAProxy/Keepalived sống.
> **Phạm vi:** Neutron với OVN mechanism driver — dựng OVN central (RAFT 3-node trên controller),
> ovn-controller trên mọi chassis (controller + compute), 1 provider network (flat trên `192.168.70.0/24`),
> tenant overlay (Geneve), router East-West, floating IP. Metadata qua `neutron-ovn-metadata-agent`.
> **Mục tiêu:** boot 1 VM cirros trong tenant network → gán floating IP `.20x` → ping được từ deployer,
> SSH vào VM được. Tắt 1 controller, traffic vẫn chạy.

## 0. OVN LÀ GÌ + TẠI SAO CHỌN OVN (đọc 5 phút trước khi gõ)

**Neutron** = "router + switch ảo cho VM". Cung cấp API `network`, `subnet`, `port`, `router`,
`floating IP`, `security group`. Neutron chỉ là tầng API + control plane — phải kết nối tới một
**mechanism driver** để biến API call thành cấu hình data plane thật. Các lựa chọn lịch sử:

| Driver | Data plane | Agent | Trạng thái |
|---|---|---|---|
| Linuxbridge | bridge Linux + iptables | linuxbridge-agent | Deprecated 2024 |
| ML2/OVS | OpenvSwitch + iptables | ovs-agent + l3-agent + dhcp-agent + metadata-agent | Vẫn maintain, được nhiều người dùng |
| **OVN** | OpenvSwitch + OpenFlow (do OVN gen) | **ovn-controller** (1 binary) + metadata-agent | **Default từ 2025.1** |

**OVN khác ML2/OVS ở chỗ nào (đọc kỹ phần này):**

- ML2/OVS: mỗi loại traffic 1 agent Python (`neutron-l3-agent`, `neutron-dhcp-agent`,
  `neutron-openvswitch-agent`, `neutron-metadata-agent`). L3-agent chạy trên "network node"
  → SPOF nếu không setup HA L3-agent + DVR riêng. DHCP cũng vậy.
- OVN: 1 binary `ovn-controller` chạy trên mọi node, dịch trực tiếp logical flow → OpenFlow.
  **Không có L3-agent, không có DHCP-agent.** Router + DHCP là logical entity trong OVN
  Northbound DB → ovn-controller render thành flow trên OVS local. → distributed natively,
  không SPOF.
- ML2/OVS lưu logical state trong DB Neutron (MariaDB) → mỗi lần boot VM phải đẩy config qua
  RPC sang agent. OVN tách logical state ra **2 DB OVSDB**: NB (Northbound) cho user-facing
  intent, SB (Southbound) cho per-chassis flow. Neutron chỉ ghi vào NB; ovn-northd dịch NB→SB;
  ovn-controller mỗi chassis đọc SB → render flow. RPC bus của Neutron gần như không dùng cho
  data plane → ít load lên RabbitMQ.

**3 daemon OVN central cần biết:**

- **`ovsdb-server` (NB)** — port 6641 (client), 6643 (RAFT). Lưu logical: `Logical_Switch`,
  `Logical_Router`, `Logical_Switch_Port`, NAT rule, ACL.
- **`ovsdb-server` (SB)** — port 6642 (client), 6644 (RAFT). Lưu per-chassis: chassis nào
  bind port nào, encap IP, flow logic dạng OVN match-action.
- **`ovn-northd`** — daemon dịch NB → SB. Idempotent. Trên 1 cụm 3-node, mỗi controller chạy
  1 northd, **chỉ 1 active** (active/passive qua RAFT lock).

**ovn-controller** — chạy trên **mọi chassis** (controller + compute). Đọc SB → render flow lên
OVS local. Đồng thời báo cáo chassis lên SB (system-id, encap IP, bridge mappings).

**Geneve overlay** — tenant traffic giữa các chassis đi qua Geneve tunnel (UDP 6081). OVN
auto-tạo tunnel khi ovn-controller báo chassis online. KHÔNG cần VXLAN/GRE manual setup.

**Gateway chassis** — chassis được đánh dấu `enable-chassis-as-gw` sẽ phụ trách bridge giữa
tenant overlay và provider network (làm North-South gateway, NAT cho floating IP). Trong lab
này: 3 controller làm gateway → HA gateway tự động qua BFD probe + priority.

**Tóm gọn: tại sao OVN cho lab 2025.1:**
1. Đúng default upstream → docs OpenStack 2025.1 ưu tiên OVN, ML2/OVS lùi về "secondary".
2. Ít daemon hơn → ít service phải nhớ restart.
3. HA L3/DHCP miễn phí — không cần config gì thêm.
4. Đánh đổi: debug khó hơn (phải biết `ovn-nbctl`, `ovn-sbctl`, `ovs-ofctl dump-flows`). Đây là
   lý do người ta vẫn chọn ML2/OVS cho production cũ.

## 1. CHUẨN BỊ NỀN + QUYẾT ĐỊNH NETWORK DESIGN

### 1.1. Sơ đồ logic

```
              +-------------------------+
              | External 192.168.70.0/24|  <- provider network (flat)
              | Floating IP .200-.250   |
              +------------+------------+
                           |
                  +--------+--------+
                  | br-provider     |   <- OVS bridge mỗi node
                  +--------+--------+
                           |
              gateway chassis = controller01/02/03
                           |
              +------------+-------------+
              |     Geneve overlay        |  <- tenant traffic
              |  192.168.100.0/24 (vd)    |
              +---------------------------+
                  ↑          ↑          ↑
              compute01  compute02  compute03
              (ovn-controller, ovs, metadata-agent)
```

### 1.2. Quyết định NIC cho br-provider — bắt buộc chốt trước

OVN gateway cần 1 OVS bridge (`br-provider`) gắn vào 1 NIC vật lý để traffic ra ngoài. Có 2 lựa chọn:

**Lựa chọn A — có sẵn `eth1` riêng (KHUYẾN NGHỊ NẾU CÓ).**
- Bạn add NIC thứ 2 vào controller01/02/03 (cùng VLAN với mgmt 192.168.70.0/24, không cấp IP).
- `eth1` lên `state UP` nhưng `inet (none)`. OVS attach trực tiếp vào br-provider, không lo mất SSH.
- An toàn cho lab: nếu cấu hình OVS sai, mgmt eth0 vẫn nguyên.

**Lựa chọn B — chỉ có `eth0`, đẩy nó lên br-provider.**
- OVS take-over eth0. IP mgmt `.122/.127/.119` phải chuyển từ eth0 sang **internal port** của
  br-provider (cùng MAC, cùng IP, OS không phân biệt).
- Rủi ro: 1 lệnh sai → mất SSH ngay lập tức, phải vào console cloud để cứu.
- Phải làm gọn trong 1 transaction (ovs-vsctl atomic). Có script sample ở Mục 5.

> **Lab này giả định Lựa chọn A** (eth1 riêng) ở mọi lệnh phía dưới. Nếu bạn chọn B, đọc kỹ
> Mục 5.2 trước khi gõ — có warning ASCII đỏ riêng.

### 1.3. Verify pre-flight từ deployer

```bash
for h in controller01 controller02 controller03 compute01 compute02 compute03; do
  echo "========== $h =========="
  ssh $h "
    echo '--- NIC ---'
    ip -br link | grep -E 'eth0|eth1'
    echo '--- IP ---'
    ip -br addr | grep -E 'eth0|eth1'
    echo '--- OVS pre-existing? ---'
    dpkg -l | grep -E 'openvswitch|ovn' || echo '(clean)'
    echo '--- kernel module ---'
    lsmod | grep -E 'openvswitch|geneve' || echo '(not loaded yet)'
    echo '--- chrony ---'
    chronyc tracking | grep -E 'Reference|Leap'
  "

```

Cần:
- `eth1` UP, không có IP, ở mọi controller (gateway chassis).
- Compute không bắt buộc có eth1 (compute chỉ chạy tenant overlay qua eth0 trên cùng subnet),
  nhưng nếu có thì cũng tốt — gateway sẽ failover được sang compute.
- Chưa cài OVS/OVN ở đâu cả (clean slate).
- Chrony lệch < 50ms — OVN RAFT lock siết clock skew khá chặt.

## 4. CẤU HÌNH `ovn-controller` TRÊN MỌI CHASSIS

Mỗi chassis (controller + compute) chạy 1 `ovn-controller` daemon. Daemon đọc config từ OVS DB
local (key `external_ids` trên row `Open_vSwitch`).

### 4.1. Set ovn-remote, encap, system-id trên mọi chassis

Lưu ý mỗi node `ovn-encap-ip` = IP mgmt của chính nó:

```bash
# Mapping: hostname -> mgmt IP
declare -A IPMAP=(
  [controller01]=192.168.70.122
  [controller02]=192.168.70.127
  [controller03]=192.168.70.119
  [compute01]=192.168.70.123
  [compute02]=192.168.70.124
  [compute03]=192.168.70.128
)

for h in "${!IPMAP[@]}"; do
  ip=${IPMAP[$h]}
  echo "=== $h ($ip) ==="
  ssh $h "
    sudo ovs-vsctl set open . external_ids:ovn-remote='tcp:192.168.70.122:6642,tcp:192.168.70.127:6642,tcp:192.168.70.119:6642'
    sudo ovs-vsctl set open . external_ids:ovn-encap-type=geneve
    sudo ovs-vsctl set open . external_ids:ovn-encap-ip=$ip
    sudo ovs-vsctl set open . external_ids:system-id=$h
  "
done
```

> `system-id` để OVN nhận diện chassis. Mặc định OVS sinh UUID random — đặt = hostname cho dễ đọc.

### 4.2. Đánh dấu 3 controller là gateway chassis

```bash
for h in controller01 controller02 controller03; do
  ssh $h 'sudo ovs-vsctl set open . external_ids:ovn-cms-options=enable-chassis-as-gw'
done
```

> Compute KHÔNG đánh dấu — chỉ controller làm North-South gateway. Đảm bảo tách traffic
> tenant-to-internet ra khỏi compute (cũng có thể đánh dấu compute nếu muốn DVR-style, nhưng
> phức tạp hơn — bỏ qua trong lab).

### 4.3. Start `ovn-controller` trên mọi chassis

```bash
for h in controller01 controller02 controller03 compute01 compute02 compute03; do
  ssh $h '
    sudo systemctl restart ovn-controller
    sudo systemctl enable ovn-controller
    sudo systemctl is-active ovn-controller
  '
done
```

## 5. TẠO `br-provider` + ATTACH NIC VẬT LÝ

### 5.1. Lựa chọn A (eth1 riêng) — KHUYẾN NGHỊ

Trên 3 controller (gateway chassis):

```bash
for h in controller01 controller02 controller03; do
  ssh $h '
    sudo ovs-vsctl --may-exist add-br br-provider
    sudo ovs-vsctl --may-exist add-port br-provider eth1
    sudo ip link set eth1 up
    sudo ovs-vsctl set open . external_ids:ovn-bridge-mappings=physnet1:br-provider
  '
done

# verify
for h in controller01 controller02 controller03; do
  echo "=== $h ==="
  ssh $h 'sudo ovs-vsctl show | grep -E "Bridge|Port"'
done
```

Mong đợi mỗi node có:
```
Bridge br-int       (auto, OVN se tao)
Bridge br-provider
    Port eth1
    Port br-provider
```

> `br-int` (integration bridge) sẽ được ovn-controller tự tạo lần đầu connect SB. Nếu chưa
> thấy br-int, đợi 30s rồi check lại — hoặc restart ovn-controller.

### 5.2. Lựa chọn B (chỉ có eth0) — RỦI RO, ĐỌC KỸ

> **CẢNH BÁO ASCII:** Nếu bạn gõ sai 1 dòng, mgmt eth0 mất IP, SSH rớt ngay. Đảm bảo có console
> truy cập (vSphere/cloud web console) trước khi gõ. Không gõ qua SSH session duy nhất.

Pattern: dùng 1 lệnh `ovs-vsctl` đa-step để atomic. Sau đó đẩy IP từ eth0 sang br-provider
internal port. Trên TỪNG controller, gõ qua console:

```bash
# THAY .122 BANG IP CUA NODE DANG GO
NODE_IP=192.168.70.122
GW=192.168.70.1   # gateway mac dinh cua subnet

sudo ovs-vsctl --may-exist add-br br-provider \
  -- --may-exist add-port br-provider eth0 \
  -- set open . external_ids:ovn-bridge-mappings=physnet1:br-provider

# Day IP tu eth0 sang br-provider internal port
sudo ip addr flush dev eth0
sudo ip addr add ${NODE_IP}/24 dev br-provider
sudo ip link set br-provider up
sudo ip route replace default via ${GW} dev br-provider
```

Sửa `/etc/netplan/*.yaml` để persistent (ngoài scope phase này — viết riêng nếu cần).

### 5.3. Verify connectivity sau khi setup br-provider


## 14. SỰ CỐ HAY GẶP (cập nhật khi gặp thật)

### a. `ovn-central` start fail: `cluster join failed`
DB cũ còn dấu vết. Stop service, xoá `/var/lib/ovn/*.db`, start lại theo đúng thứ tự bootstrap → 02 → 03 ở Mục 3.6.

### b. `openstack network agent list` không thấy chassis
- ovn-controller ở chassis đó chưa connect: kiểm tra `ovs-vsctl get open . external_ids:ovn-remote`.
- SB DB không reach được: telnet `192.168.70.122 6642` từ chassis đó.
- system-id trùng: 2 chassis cùng `system-id` → 1 cái bị OVN ghi đè. Đảm bảo set khác nhau ở Mục 4.1.

### c. VM boot xong không lấy được IP từ DHCP
- OVN trả DHCP qua logical flow, không qua dhcp-agent. Check `ovn-nbctl --db=$OVN_NB_CONN list Logical_Switch_Port` → port của VM phải có `dhcpv4_options` trỏ tới row Logical_Switch_Port DHCP_Options.
- Subnet phải có `--dhcp` enabled (selfservice subnet mặc định bật, provider ta chủ ý `--no-dhcp`).

### d. VM lấy IP nhưng không ra internet
- Router chưa add subnet: `openstack router show demo-router` → check `interfaces_info`.
- Router chưa có external gateway: `--external-gateway provider` (Mục 12.3).
- Gateway chassis không có route ra ngoài: trên controller chạy `ip route` — phải có default qua eth0 hoặc qua br-provider. Nếu Lựa chọn B, default phải qua br-provider.
- Floating IP chưa associate: `openstack floating ip list` xem port_id.
- `iptables`/security group: `openstack security group rule list default` phải có icmp + tcp/22.

### e. `metadata` request fail trong VM (`curl 169.254.169.254` timeout)
- `neutron-ovn-metadata-agent` chết ở compute. Check `systemctl status`.
- `metadata_proxy_shared_secret` không khớp giữa neutron + nova. Sửa cả 2 + restart.
- OVN namespace `ovnmeta-<network-id>` chưa tạo trên compute: `ip netns list` ở compute → phải thấy ovnmeta-*. Không có → metadata agent chưa schedule. Restart `ovn-controller` + `neutron-ovn-metadata-agent`.

### f. `Cluster ID mismatch` khi join node mới
Bootstrap node đã từng init trước với cluster ID khác. Xoá DB ở 02/03, để chúng pull fresh từ 01.

### g. Floating IP ping được trong subnet nhưng outside không reach
Outside (router subnet 192.168.70.1) không biết route về `.200-.250` qua đâu — vì IP này không thật, do OVN gateway chassis "claim". Bình thường ARP probe OVN sẽ trả lời. Nếu vẫn fail:
- Kiểm tra `ovn-sbctl --db=$OVN_SB_CONN list Port_Binding` → tìm floating IP port, xem `chassis` field — phải bind vào 1 gateway chassis.
- ARP cache outside cũ: `arp -d 192.168.70.<floating>` trên máy ngoài rồi ping lại.
