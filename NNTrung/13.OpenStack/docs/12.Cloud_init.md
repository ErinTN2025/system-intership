# Tìm hiểu về Cloud-init
## 1. Khái niệm
Cloud-init là một công cụ mã nguồn mở tiêu chuẩn công nghiệp được sử dụng để khởi tạo các máy ảo trên môi trường Cloud (cloud instances) trong quá trình khởi động lần đầu. Nó được coi là "phương pháp thiết lập đa nền tảng" (multi-distribution) phổ biến nhất hiện nay cho các hệ điều hành Linux và đôi khi là Windows.

- Thiết lập đa nền tảng nghĩa là:
  - Một ngôn ngữ chung: Bạn chỉ cần viết một file cấu hình duy nhất bằng định dạng YAML (như ví dụ `#cloud-config`).
  - Bộ dịch thông minh: Cloud-init đóng vai trò là "thông dịch viên". Nó sẽ đọc file YAML của bạn, sau đó tự nhận diện: "À, máy này là Ubuntu, mình sẽ dùng lệnh apt" hoặc "Máy này là CentOS, mình sẽ dùng yum".
- Tóm lại: Multi-distribution giúp bạn "Viết một lần, chạy mọi nơi". Bạn không cần quan tâm máy ảo bên dưới chạy hệ điều hành gì, Cloud-init sẽ lo phần "chuyển ngữ" các thiết lập đó vào hệ thống.

![altimage](../Images/Screenshot_6.png)

Tưởng tưởng dễ hiểu:
- Bạn có 1 file ISO Ubuntu sạch
- Nếu KHÔNG có cloud-init:
- Boot lên -> màn hình login, tự tạo user, cài ssh, config network.
- Khi có Cloud init:
```bash
VM boot
   ↓
cloud-init chạy
   ↓
- tạo user trung
- add ssh key
- cài nginx
- start nginx
   ↓
VM READY luôn
```
Bạn chưa login lần nào mà máy đã xong hết.

Flow trong Cloud
```bash
Bạn (user-data YAML)
        ↓
OpenStack / AWS
        ↓
Metadata service (169.254...)
        ↓
VM boot
        ↓
cloud-init đọc metadata
        ↓
chạy config
```
## 2. Chức năng chính
Khi bạn tạo VM (instance) bằng Nova, bản thân image chỉ là một OS “thô”. Cloud-init sẽ:
- **Thiết lập người dùng**: Tạo User mới và thêm các khóa SSH công khai (SSH Keys) để truy cập từ xa.
- **Cấu hình mạng**: Thiết lập Hostname, địa chỉ IP tĩnh hoặc DHCP.
- **Quản lý gói phần mềm**: Cập nhật hệ điều hành và cài đặt các gói phần mềm cần thiết ngay khi boot máy.
- **Chạy script khởi tạo (user-data)**: cài nginx, docker, pull code… ngay khi VM vừa lên.
- **Customize instance**: gần như mọi thứ bạn muốn “auto setup”.

Cloud-init hoạt động dựa trên việc đọc dữ liệu từ hai nguồn chính được cung cấp bởi nền tảng đám mây:
- **Metadata**: Chứa các thông tin về môi trường hạ tầng (như Instance ID, Hostname, Network config).
- **User-data**: Là dữ liệu do người dùng truyền vào (thường dưới định dạng YAML hoặc Shell Script) để chỉ định những việc cần làm khi máy ảo khởi chạy.

Khi máy ảo khởi động, dịch vụ Cloud-init sẽ tìm nạp dữ liệu này từ các dịch vụ Metadata (ví dụ: qua URL `http://169.254.169.254`) hoặc từ một ổ đĩa cấu hình đi kèm, sau đó áp dụng chúng vào hệ thống.

## 3. Cấu trúc cấu hình 
Cloud-init sử dụng định dạng `#cloud-config` (dựa trên YAML) để người dùng dễ dàng đọc và viết.

![altimage](../Images/Screenshot%202026-04-29%20144705.png)

Ví dụ một đoạn cấu hình đơn giản:

```YAML
#cloud-config
users:
  - name: admin
    ssh_authorized_keys:
      - ssh-rsa AAAAB3Nza...
    sudo: ['ALL=(ALL) NOPASSWD:ALL']
    groups: sudo
    shell: /bin/bash

packages:
  - git
  - curl
  - nginx

runcmd:
  - systemctl start nginx
```
## 4. Tầm quan trọng của Cloud-init
- **Tính nhất quán**: Bạn có thể dùng cùng một đoạn mã cấu hình cho máy ảo trên OpenStack, AWS, Google Cloud hay Azure mà không cần thay đổi quá nhiều.
- **Tự động hóa (Automation)**: Là thành phần không thể thiếu trong quy trình DevOps và CI/CD, giúp triển khai hàng loạt máy chủ chỉ trong vài giây mà không cần can thiệp thủ công.
- **Tách biệt Image và Cấu hình**: Bạn chỉ cần một Image hệ điều hành chuẩn (như Ubuntu Cloud Image hoặc CentOS Generic Cloud). Mọi tùy chỉnh đặc thù sẽ được đẩy qua Cloud-init thay vì phải tạo hàng chục Image tùy chỉnh khác nhau.

Ví dụ:
- Bạn clone nhiều VM từ 1 image
- Muốn mỗi VM tự:
  - đổi hostname
  - set IP
  - add SSH key
  - chạy script setup

Lưu ý:

Trong OpenStack:
- Cloud-init lấy config từ metadata service (169.254.169.254) → tự động

Trong VMware:
- không có metadata service mặc định
- Bạn phải cấp dữ liệu bằng cách khác:
  - ISO (NoCloud datasource)
  - hoặc dùng VMware vSphere + customization spec

## 5. Metadata
### 5.1 Khái niệm
Trong môi trường điện toán đám mây, Metadata (Siêu dữ liệu) đóng vai trò là "Giấy khai sinh" và "Thẻ căn cước" của một thực thể (Instance). Nếu như User-data là những gì bạn muốn máy ảo làm, thì Metadata là những gì máy ảo cần biết về chính nó và môi trường xung quanh để có thể tồn tại.

**Bản chất của Metadata**: là tập hợp các thông tin mô tả về đặc điểm kỹ thuật và vị trí logic của máy ảo trong hạ tầng của nhà cung cấp (như OpenStack, AWS, hay VNPT Cloud).

VM cần metadata để biết thông tin định danh và cấu hình nâng cao, không chỉ riêng network

### 5.2 Các thành phần phổ biến trong Metadata
Thông tin Metadata thường được tổ chức theo cấu trúc cây thư mục hoặc JSON. Các dữ liệu quan trọng nhất bao gồm:

| Nhóm thông tin| Ví dụ cụ thể |
|---------------|--------------|
| **Định danh (Identity)**| Instance ID, Tên máy (Hostname), UUID của máy ảo.|
| **Mạng (Networking)**| Địa chỉ IP nội bộ (Fixed IP), IP công cộng (Floating IP), Netmask, Gateway, địa chỉ MAC.|
| **Bảo mật (Security)**| Public SSH Keys (để bạn có thể login vào máy).|
| **Vị trí (Placement)**| Availability Zone (Khu vực vật lý), Region.|
| **Cấu hình phần cứng**| Flavor/Size (Số lượng vCPU, dung lượng RAM), ID của các ổ đĩa đang gắn kèm|

**Lưu ý**
- Security group  ko nằm trực tiếp trong metadata, nó được áp dụng ở tầng mạng.
### 5.3 Cách thức vận chuyển Metadata vào máy ảo
Làm sao máy ảo lấy được dữ liệu này khi nó chưa có cấu hình gì? Có hai phương thức chính:'

#### A. Qua dịch vụ Metadata (Http Service)
Đây là cách phổ biến nhất. Nhà cung cấp dịch vụ đám mây duy trì một server đặc biệt tại địa chỉ IP link-local: `169.254.169.254`.
- Địa chỉ này không thể định tuyến ra Internet
- Nó chỉ tồn tại trong mạng nội bộ giữa Host (máy chủ vật lý) và Guest (máy ảo).
- **Cloud-init** sẽ gửi một yêu cầu GET đến địa chỉ này ngay khi mạng vừa khởi động để lấy dữ liệu.

VM không cần route internet vẫn gọi được
#### B. Qua ổ đĩa cấu hình (Config Drive)
Trong một số trường hợp mạng chưa sẵn sàng hoặc vì lý do bảo mật, nhà cung cấp sẽ tạo một "ổ đĩa ảo" siêu nhỏ (thường là định dạng ISO hoặc VVFAT) chứa các file metadata và gắn (attach) trực tiếp vào máy ảo như một ổ CD-ROM.
- Cloud-init sẽ quét các thiết bị lưu trữ, tìm ổ đĩa có nhãn `config-2`.
- Nó mount ổ đĩa này và đọc dữ liệu trực tiếp từ đó.

### 5.4 Cấu trúc dữ liệu Metadata OpenStack 
```JSON
{
    "uuid": "d8e3920a-6db2-48ec-b830-1b7bc0e9a58b",
    "name": "web-server-01",
    "availability_zone": "nova",
    "hostname": "web-server-01.localdomain",
    "launch_index": 0,
    "public_keys": {
        "my-key": "ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQ..."
    },
    "project_id": "89392233f443423a85a067e2a900388a"
}
```

### 5.5 Cloud-init làm việc với Metadata
- **Phát hiện(Detection)**: Cloud-init xác định xem nó đang chạy trên nền tảng nào (KVM, OpenStack, AWS, Azure,...) để biết cách tìm Metadata tương ứng.
- **Truy xuất (Fetching)**: Nó kết nối tới `169.254.169.254` hoặc mount Config Drive.
- **Áp dụng (Applying)**:
  - Nó lấy `hostname` từ metadata và ghi vào `/etc/hostname`.
  - Nó lấy `public_keys` và ghi vào file `/home/ubuntu/.ssh/authorized_keys`.
  - Nó thiết lập cấu hình mạng dựa trên thông tin IP được cung cấp.

### 5.6 Tính cần thiết của Metadata 
- **Bảo mật**: Đây là cơ chế chính để đấy SSH Key vào máy ảo. Nếu dịch vụ metadata lỗi, bạn sẽ bị khóa bên ngoài máy ảo (vì không có mật khẩu mặc định).
- **Tự động hóa**: Các script bạn viết (User-data) có thể truy vấn ngược lại Metadata để biết mình đang ở đâu.
- **Vendor Data**: Ngoài Metadata thông thường, nhà cung cấp (VNPT, AWS...) có thể đẩy vào Vendor-data để thực thi các chính sách chung cho toàn bộ công ty (như cài sẵn phần mềm giám sát, agent bảo mật).

**Lưu ý**:

Vì Metadata Service (`169.254.169.254`) không yêu cầu mật khẩu để truy cập từ bên trong máy ảo, nên nếu một hacker chiếm được quyền điều khiển máy ảo của bạn, họ có thể lấy được rất nhiều thông tin về hạ tầng của bạn thông qua địa chỉ này. Do đó, việc quản lý quyền hạn (IAM Roles) gắn với Metadata là rất quan trọng.

## 6. Userdata
Nếu Metadata là "Giấy khai sinh" giúp máy ảo định danh bản thân, thì User-data chính là "Bản kế hoạch hành động" mà bạn giao cho máy ảo ngay khi nó vừa chào đời.
### 1. Khái niệm
User-data là dữ liệu do người dùng truyền vào thực thể đám mây tại thời điểm khởi tạo (provisioning). Cloud-init sẽ đọc dữ liệu này để thực hiện các tùy chỉnh đặc thù mà một Image hệ điều hành trắng không có sẵn.
- **Metadata**: Do hệ thống (Cloud Provider) quyết định
- **Userdata**: Do người dùng (User) quyết định
### 2. Các định dạng User-data phổ biến
#### 2.1 Shell Scripts (Dạng thô)
Đây là cách đơn giản nhất. Bạn viết một đoạn mã Bash script như bình thường.
- **Dấu hiệu nhận biết**: Phải bắt đầu bằng dòng Shebang: `#!/bin/bash` hoặc `#!/bin/sh`.
- **Ưu điểm**: Linh hoạt, tận dụng được kiến thức Linux có sẵn.
- **Cách chạy**: Cloud-init sẽ lưu script này thành một file tạm và thực thi nó với quyền **root**.

#### 2.2 Cloud-config (Định dạng YAML)
Đây là "ngôn ngữ" riêng của Cloud-init, sử dụng định dạng YAML để khai báo cấu hình.
- **Dấu hiệu nhận biết**: bắt đầu bằng dòng `#cloud-config`
- **Ưu điểm**: Rất dễ đọc, có tính trừu tượng cao (không cần quan tâm máy chạy Ubuntu hay CentOS, Cloud-init sẽ tự chọn lệnh cài đặt tương ứng).
  - Ví dụ:
```YAML
#cloud-config
package_update: true
packages:
  - docker.io
  - git
write_files:
  - path: /var/www/html/index.html
    content: |
      <h1>Hello from Cloud-init!</h1>
```
### Vòng đời thực thi (lifecycle)
**Lần chạy:** Theo mặc định, User-data chỉ chạy Duy nhất một lần vào lần boot đầu tiên của máy ảo (First Boot). Nếu bạn khởi động lại máy (Reboot), Cloud-init sẽ thấy máy đã được cấu hình và bỏ qua bước này.

**Thời điểm:** Nó chạy ở giai đoạn cuối cùng của quá trình khởi động hệ thống (sau khi mạng và các dịch vụ cơ bản đã sẵn sàng).

**Lưu ý:** Nếu bạn muốn script chạy lại mỗi khi reboot, bạn phải cấu hình thêm các module cụ thể (như `scripts-per-boot`), nhưng thông thường User-data chỉ dùng để "Setup" ban đầu.

### 3. Các định dạng User-data nâng cao

#### 3.1 MIME Multi-part (Kết hợp nhiều định dạng)
Khi bạn muốn vừa chạy shell script vừa áp dụng cloud-config trong cùng một lần boot, bạn dùng MIME multi-part.

```
Content-Type: multipart/mixed; boundary="===============1234567890=="
MIME-Version: 1.0

--===============1234567890==
Content-Type: text/cloud-config; charset="us-ascii"
MIME-Version: 1.0

#cloud-config
package_update: true
packages:
  - nginx

--===============1234567890==
Content-Type: text/x-shellscript; charset="us-ascii"
MIME-Version: 1.0

#!/bin/bash
echo "Hello from shell script" > /tmp/hello.txt

--===============1234567890==--
```

#### 3.2 Include File (Tải user-data từ URL ngoài)
Bắt đầu bằng `#include`, Cloud-init sẽ tải nội dung từ các URL liệt kê và xử lý như user-data bình thường.

```
#include
https://example.com/setup-webserver.sh
https://example.com/base-config.yaml
```

#### 3.3 Cloud Boothook
Bắt đầu bằng `#cloud-boothook`. Đây là loại chạy **sớm nhất**, trước cả giai đoạn khởi tạo cloud-init thông thường. Dùng cho các tác vụ cấp thấp (như cấu hình kernel module).

```bash
#!/bin/bash
modprobe overlay
```

### 4. Bảng so sánh các định dạng User-data

| Định dạng | Dấu hiệu nhận biết | Khi nào dùng |
|---|---|---|
| Shell Script | `#!/bin/bash` | Khi cần logic phức tạp, điều kiện, vòng lặp |
| Cloud-config | `#cloud-config` | Khi khai báo cấu hình rõ ràng, dễ đọc |
| MIME Multi-part | `Content-Type: multipart/mixed` | Khi cần kết hợp nhiều loại |
| Include | `#include` | Khi user-data quá lớn hoặc dùng lại từ URL |
| Cloud Boothook | `#cloud-boothook` | Khi cần chạy trước mọi thứ |

## 7. Các giai đoạn Boot của Cloud-init (Boot Stages)

Cloud-init không chạy tất cả mọi thứ trong một lần. Nó chia quá trình khởi tạo thành **5 giai đoạn (stages)** tuần tự để đảm bảo tính đúng đắn và thứ tự phụ thuộc.

```
Kernel Boot
    ↓
[Stage 1] Generator     ← Quyết định cloud-init có chạy không
    ↓
[Stage 2] Local         ← Cấu hình mạng cục bộ, tìm datasource
    ↓
[Stage 3] Network       ← Kết nối metadata service, áp dụng config
    ↓
[Stage 4] Config        ← Chạy các module cấu hình (user, package...)
    ↓
[Stage 5] Final         ← Chạy user-data scripts, runcmd
    ↓
VM READY
```

### 7.1 Chi tiết từng giai đoạn

| Giai đoạn | Service systemd | Việc làm chính |
|---|---|---|
| **Generator** | `cloud-init-generator` | Kiểm tra file `/etc/cloud/cloud-init.disabled`. Nếu tồn tại thì dừng toàn bộ. |
| **Local** | `cloud-init-local.service` | Gắn Config Drive, tìm datasource local, cấu hình network tạm thời. |
| **Network** | `cloud-init.service` | Kết nối `169.254.169.254`, tải metadata & user-data, áp dụng hostname. |
| **Config** | `cloud-config.service` | Chạy các module: tạo user, cài package, ghi file (`write_files`). |
| **Final** | `cloud-final.service` | Thực thi `runcmd`, shell scripts trong user-data. Đây là bước cuối cùng. |

### 7.2 Ví dụ thực tế

Khi bạn viết cloud-config như sau:

```yaml
#cloud-config
hostname: my-webserver       # → Áp dụng ở stage Network
users:
  - name: deploy             # → Áp dụng ở stage Config
packages:
  - nginx                    # → Áp dụng ở stage Config
runcmd:
  - systemctl enable nginx   # → Áp dụng ở stage Final
```

## 8. Datasource

### 8.1 Khái niệm
Datasource là "nguồn cấp dữ liệu" — tức là nơi Cloud-init tìm kiếm Metadata và User-data. Mỗi nền tảng cloud khác nhau sẽ có cách cung cấp dữ liệu khác nhau.

Cloud-init hỗ trợ nhiều datasource, nó sẽ **tự động dò tìm** datasource phù hợp khi khởi động.

### 8.2 Các datasource phổ biến

| Datasource | Nền tảng | Cách lấy dữ liệu |
|---|---|---|
| **OpenStack** | OpenStack | HTTP `169.254.169.254` hoặc Config Drive |
| **EC2** | AWS | HTTP `169.254.169.254/latest/meta-data/` |
| **NoCloud** | VMware, KVM local, test | File trên ổ đĩa ISO hoặc thư mục local |
| **ConfigDrive** | OpenStack (tùy chọn) | Ổ đĩa ISO gắn kèm với label `config-2` |
| **Azure** | Microsoft Azure | HTTP endpoint riêng của Azure |
| **GCE** | Google Cloud | HTTP `metadata.google.internal` |

### 8.3 Datasource NoCloud (quan trọng khi test)
Khi bạn muốn test Cloud-init trên máy tính cá nhân (không có cloud), bạn dùng **NoCloud**. Tạo hai file:

**meta-data** (YAML, không có extension):
```yaml
instance-id: test-instance-001
local-hostname: my-test-vm
```

**user-data**:
```yaml
#cloud-config
packages:
  - curl
runcmd:
  - echo "NoCloud test success" > /tmp/result.txt
```

Sau đó tạo ISO và gắn vào VM:
```bash
genisoimage -output seed.iso -volid cidata -joliet -rock user-data meta-data
```

## 9. Các module Cloud-config quan trọng

Cloud-config có rất nhiều module. Dưới đây là các module thường dùng nhất kèm ví dụ chi tiết.

### 9.1 users — Quản lý người dùng

```yaml
#cloud-config
users:
  - name: deploy
    gecos: "Deploy User"
    shell: /bin/bash
    sudo: ALL=(ALL) NOPASSWD:ALL
    groups: [sudo, docker]
    ssh_authorized_keys:
      - ssh-rsa AAAAB3NzaC1yc2EAAA... user@laptop
  - name: monitor
    shell: /bin/bash
    lock_passwd: true          # Không cho login bằng password
```

> **Lưu ý**: Nếu bạn khai báo `users:`, Cloud-init sẽ **thay thế** user mặc định của image (vd: `ubuntu`). Để giữ user mặc định, thêm `- default` vào đầu danh sách.

```yaml
users:
  - default                   # Giữ lại user ubuntu/centos mặc định
  - name: deploy
    ...
```

### 9.2 packages — Cài đặt phần mềm

```yaml
#cloud-config
package_update: true          # apt update / yum update
package_upgrade: true         # apt upgrade / yum upgrade
package_reboot_if_required: true

packages:
  - nginx
  - git
  - python3-pip
  - [libpq-dev, 1.0.0-1]     # Chỉ định version cụ thể
```

### 9.3 write_files — Ghi file vào hệ thống

```yaml
#cloud-config
write_files:
  - path: /etc/nginx/conf.d/app.conf
    owner: root:root
    permissions: '0644'
    content: |
      server {
          listen 80;
          server_name example.com;
          root /var/www/html;
      }

  - path: /opt/app/config.json
    encoding: b64             # Nội dung được encode base64
    content: eyJkYiI6ICJwb3N0Z3JlcyJ9
```

### 9.4 runcmd — Chạy lệnh sau khi cài đặt

```yaml
#cloud-config
runcmd:
  - systemctl enable nginx
  - systemctl start nginx
  - [curl, -o, /tmp/setup.sh, https://example.com/setup.sh]
  - bash /tmp/setup.sh
```

> **Thứ tự quan trọng**: `runcmd` chạy ở **Stage Final** — sau khi `packages` và `write_files` đã hoàn thành. Đây là nơi bạn khởi động service, chạy script cài đặt ứng dụng.

### 9.5 bootcmd — Chạy lệnh rất sớm (mỗi lần boot)

```yaml
#cloud-config
bootcmd:
  - echo 1 > /proc/sys/net/ipv4/ip_forward
  - cloud-init-per once my-format mkfs.ext4 /dev/vdb
```

> Khác với `runcmd`, `bootcmd` chạy ở **Stage Network** và chạy **mỗi lần reboot** (không chỉ lần đầu).

### 9.6 disk_setup và fs_setup — Phân vùng và định dạng ổ đĩa

```yaml
#cloud-config
disk_setup:
  /dev/vdb:
    table_type: gpt
    layout: true
    overwrite: false

fs_setup:
  - label: data-disk
    filesystem: ext4
    device: /dev/vdb1
    overwrite: false

mounts:
  - [/dev/vdb1, /data, ext4, "defaults,nofail", "0", "2"]
```

## 10. Logging và Debug Cloud-init

### 10.1 Các file log quan trọng

| File | Nội dung |
|---|---|
| `/var/log/cloud-init.log` | Log chi tiết toàn bộ quá trình chạy, bao gồm từng module |
| `/var/log/cloud-init-output.log` | Output của user-data scripts (stdout + stderr) |
| `/run/cloud-init/` | Trạng thái runtime, datasource đã phát hiện |
| `/var/lib/cloud/instance/` | Dữ liệu của instance hiện tại (user-data đã xử lý) |
| `/var/lib/cloud/instances/` | Lịch sử các lần chạy trước |

### 10.2 Xem log thực tế

```bash
# Xem toàn bộ log cloud-init
sudo cat /var/log/cloud-init.log

# Xem output của user-data script
sudo cat /var/log/cloud-init-output.log

# Xem trạng thái cloud-init
cloud-init status

# Xem trạng thái chi tiết
cloud-init status --long

# Phân tích các giai đoạn đã chạy
cloud-init analyze show

# Xem user-data đã được nhận
sudo cat /var/lib/cloud/instance/user-data.txt
```

### 10.3 cloud-init status output

```
# Đang chạy
status: running

# Thành công
status: done

# Lỗi
status: error
errors:
  - ...
```

### 10.4 Debug khi gặp lỗi

**Bước 1**: Kiểm tra trạng thái tổng quan
```bash
cloud-init status --long
```

**Bước 2**: Tìm lỗi trong log
```bash
grep -i "error\|failed\|traceback" /var/log/cloud-init.log
```

**Bước 3**: Xem output của script
```bash
tail -100 /var/log/cloud-init-output.log
```

**Bước 4**: Kiểm tra datasource đã nhận đúng chưa
```bash
cat /run/cloud-init/ds-identify.result
```

### 10.5 Chạy lại Cloud-init (dùng khi test)

> **Cảnh báo**: Chỉ làm trên môi trường test, không làm trên production vì sẽ reset toàn bộ cấu hình.

```bash
# Xóa trạng thái đã chạy
sudo cloud-init clean --logs

# Chạy lại toàn bộ
sudo cloud-init init
sudo cloud-init modules --mode=config
sudo cloud-init modules --mode=final

# Hoặc reboot để chạy lại từ đầu
sudo reboot
```

## 11. Cloud-init trong OpenStack — Luồng hoàn chỉnh

### 11.1 Khi bạn tạo VM trên OpenStack

```
Người dùng (Horizon / Nova API)
    |
    | Truyền user-data (YAML/Script)
    ↓
Nova API Server
    |
    | Lưu user-data vào Nova DB
    ↓
Nova Compute (trên hypervisor host)
    |
    | Khởi động VM bằng KVM/QEMU
    ↓
Neutron + iptables
    |
    | Cấu hình NAT: VM → 169.254.169.254 → Nova Metadata API
    ↓
VM Boot
    |
    ↓
cloud-init (chạy trong VM)
    |
    | GET http://169.254.169.254/openstack/latest/meta_data.json
    | GET http://169.254.169.254/openstack/latest/user_data
    ↓
Áp dụng cấu hình → VM READY
```

### 11.2 API Endpoint Metadata trong OpenStack

Khi đứng trong VM, bạn có thể truy vấn thủ công:

```bash
# Lấy toàn bộ metadata
curl http://169.254.169.254/openstack/latest/meta_data.json

# Lấy user-data (nếu có)
curl http://169.254.169.254/openstack/latest/user_data

# Lấy thông tin network
curl http://169.254.169.254/openstack/latest/network_data.json

# EC2-compatible endpoint (cũng hoạt động trên OpenStack)
curl http://169.254.169.254/latest/meta-data/
curl http://169.254.169.254/latest/meta-data/instance-id
curl http://169.254.169.254/latest/meta-data/local-ipv4
```

### 11.3 Ví dụ thực tế: Triển khai web server tự động

Khi tạo VM trên OpenStack, truyền user-data sau:

```yaml
#cloud-config
# Cập nhật và cài đặt Nginx
package_update: true
packages:
  - nginx
  - curl

# Tạo user deploy
users:
  - default
  - name: deploy
    sudo: ALL=(ALL) NOPASSWD:ALL
    shell: /bin/bash
    ssh_authorized_keys:
      - ssh-rsa AAAAB3NzaC1yc2EAAA... admin@company.com

# Ghi file cấu hình Nginx
write_files:
  - path: /var/www/html/index.html
    content: |
      <html>
        <body>
          <h1>Deployed by Cloud-init on OpenStack!</h1>
        </body>
      </html>

# Khởi động service
runcmd:
  - systemctl enable nginx
  - systemctl start nginx
  - echo "Setup complete at $(date)" >> /var/log/setup.log
```

Kết quả: VM vừa boot xong là có Nginx chạy, user `deploy` có SSH key, trang web đã có nội dung — **không cần đăng nhập lần nào**.

## 12. Best Practices (Thực hành tốt nhất)

### 12.1 Nên làm

| Nguyên tắc | Lý do |
|---|---|
| Dùng `#cloud-config` thay vì shell script khi có thể | Dễ đọc, cross-distro, ít lỗi hơn |
| Luôn có `- default` trong `users:` | Tránh mất user mặc định của image |
| Kiểm tra log sau khi VM lên | Phát hiện lỗi sớm qua `/var/log/cloud-init-output.log` |
| Dùng `package_update: true` | Đảm bảo package list mới nhất trước khi cài |
| Đặt `runcmd` ở cuối | Đảm bảo các module khác đã chạy xong |
| Test với NoCloud datasource trước | Không cần cloud thật để validate cú pháp |

### 12.2 Không nên làm

| Sai lầm phổ biến | Hậu quả | Cách đúng |
|---|---|---|
| Hardcode mật khẩu trong user-data | Lộ thông tin bảo mật | Dùng SSH key thay vì password |
| Quên `- default` khi khai báo `users:` | Mất user ubuntu/centos | Luôn thêm `- default` đầu danh sách |
| Dùng `apt-get install` trong `runcmd` khi chưa `package_update` | Cài được version cũ hoặc lỗi | Dùng module `packages:` với `package_update: true` |
| Bỏ qua lỗi cloud-init | VM không hoạt động như mong đợi | Luôn kiểm tra `cloud-init status` sau boot |
| Chạy `cloud-init clean` trên production | Mất toàn bộ cấu hình | Chỉ dùng khi test trên môi trường riêng |

### 12.3 Validate cú pháp trước khi dùng

```bash
# Kiểm tra cú pháp YAML của cloud-config
cloud-init schema --config-file my-user-data.yaml

# Output nếu OK:
# Valid cloud-config: my-user-data.yaml
```

## 13. Tổng kết

| Thành phần | Vai trò | Ai cung cấp |
|---|---|---|
| **Cloud-init** | Công cụ đọc và thực thi cấu hình khi VM boot | Cài sẵn trong cloud image |
| **Metadata** | Thông tin định danh VM (ID, hostname, IP, SSH key) | Cloud Provider (OpenStack, AWS...) |
| **User-data** | Kịch bản cấu hình tùy chỉnh (cài phần mềm, tạo user...) | Người dùng (bạn) |
| **Datasource** | Nguồn cấp dữ liệu metadata/userdata | Cloud Provider |
| **169.254.169.254** | Địa chỉ link-local để VM lấy metadata | Cloud Provider (qua NAT/proxy) |

**Luồng tóm tắt:**
```
Bạn viết user-data YAML
        ↓
Truyền vào lúc tạo VM (Nova / Horizon)
        ↓
VM boot → cloud-init chạy
        ↓
cloud-init lấy metadata từ 169.254.169.254
        ↓
cloud-init thực thi user-data (tạo user, cài gói, chạy script)
        ↓
VM sẵn sàng — tự động, không cần can thiệp thủ công
```

Cloud-init là nền tảng của mọi quy trình **Infrastructure as Code (IaC)** và **DevOps** hiện đại. Hiểu rõ Cloud-init giúp bạn xây dựng hạ tầng tự động, nhất quán và có thể tái tạo hoàn toàn chỉ bằng code.
