# Tìm hiểu về Cloud Computing
## 1. Khái niệm Cloud Computing
**Cloud Computing** là việc cung cấp các dịch vụ điện toán qua internet bao gồm máy chủ, lưu trữ, cơ sở dữ liệu, mạng, phần mềm, phân tích và trí tuệ nhân tạo (AI) theo nhu cầu một cách thuận tiện và nhanh chóng; đồng thời cho phép kết thúc sử dụng dịch vụ, giải phóng tài nguyên dễ dàng, giảm thiểu các giao tiếp với nhà cung cấp.

Thay vì bạn phải tự mua sắm, lắp đặt và bảo trì các máy chủ vật lý (server) đắt đỏ tại nhà hoặc công ty, bạn chỉ cần thuê các tài nguyên này từ một nhà cung cấp dịch vụ đám mây (như AWS, Google Cloud, Microsoft Azure) và trả tiền theo mức độ sử dụng thực tế (Pay-as-you-go).

**Cloud có thể là private và public**
  - Public cloud bán dịch vụ cho bất kỳ ai trên Internet
  - Private cloud là mạng độc quyền hoặc trung tâm dữ liệu cung cấp dịch vụ được lưu trữ cho một số người giới hạn, với các quyền truy cập và quyền nhất định.

Dù là public hay private, mục tiêu của cloud computing là cung cấp quyền truy cập dễ dàng, có thể mở rộng vào các tài nguyên điện toán và dịch vụ IT.

## 2. 3 Mô hình dịch vụ chính (Service Models)
| Mô hình | Người dùng tự quản lý | Nhà cung cấp quản lý | Ví dụ |
|---------|-----------------------|----------------------|-----| 
| Infrastructure as a Service(IaaS) | OS, App, Data | Virtualization, Servers, Storage | AWS EC2, Azure VM, GCP Compute |
| Platform as a Service(PaaS) | App, Data | Runtime, OS,  Infrastructure | AWS Elastic Beanstalk, Google App Engine |
| Software as a Service(SaaS) | Chỉ sử dụng(không quản lý gì) | All | Gmail, Slack, Office 365 |

Càng lên cao (IaaS → PaaS → SaaS), bạn càng ít phải quản lý hạ tầng.
- IaaS: Cơ sở hạ tầng như một dịch vụ
- PaaS: Nền tảng như một dịch vụ
- SaaS: Phần mềm như một dịch vụ

## 3. Đặc điểm thiết yếu (5 đặc điểm)
**On-demand self-service(Tự phục vụ theo nhu cầu)**: Người dùng có thể tự động cấp phát tài nguyên máy tính (CPU, RAM, storage...) mà không cần tương tác thủ công với nhà cung cấp dịch vụ.
Ví dụ:
```bash
# Bạn cần 1 server mới? Chỉ cần 1 lệnh hoặc click chuột:
aws ec2 run-instances \
  --image-id ami-0c55b159cbfafe1f0 \
  --instance-type t2.micro \
  --count 1

# Hoặc dùng Terraform:
terraform apply  # → 5 phút sau có server chạy
```
  - Giảm thời gian chờ đợi từ vài ngày(mua hardware) xuống vài phút
  - Dễ dàng test, phát triển, scale mà không cần xin phép

**Broad network access(Truy cập rộng qua mạng)**: Dịch vụ cloud có thể truy cập qua mạng (Internet/intranet) từ nhiều loại thiết bị khác nhau: laptop, điện thoại, tablet, IoT...
```bash
 Mobile App → API Gateway → Lambda → DynamoDB
 Developer laptop → SSH → EC2 instance
 CI/CD server → Deploy → Kubernetes cluster
 Customer browser → CDN → S3 static website
```
  - Thiết kế hệ thống phải "network-aware": API RESTful, HTTPS, CORS.
  - Bảo mật phải đa lớp: firewall, IAM, encryption in-transit
  - Test trên nhiều thiết bị và điều kiện mạng (4G, WiFi, latency cao).

**Resource pooling(Gộp tài nguyên dùng chung)**: Nhà cung cấp dùng mô hình multi-tenant(nhiều người thuê) để phục vụ nhiều khách hàng bằng cùng một hạ tầng vật lý, nhưng cách ly logic để đảm bảo an toàn và riêng tư.
```bash
Physical Server (128 CPU, 512GB RAM)
├─ VM Customer A (4 CPU, 8GB) → chạy app thương mại điện tử
├─ VM Customer B (2 CPU, 4GB) → chạy blog cá nhân  
├─ Container Cluster C → chạy microservices
└─ ... và hàng trăm tenant khác

→ Hypervisor (VMware, KVM) hoặc Container Runtime (Docker) đảm bảo cách ly
```

**Rapid elasticity(Co giãn nhanh chóng)**: Tài nguyên có thể tự động mở rộng hoặc giải phóng một cách linh hoạt, nhanh chóng theo nhu cầu thực tế. Đối với người dùng, các khả năng cung cấp thường xuất hiện là không giới hạn và có thể được sử dụng ở bất kỳ số lượng nào và bất kỳ lúc nào.
```bash
# Auto-scaling group trên AWS (ví dụ đơn giản)
- Khi CPU > 70% trong 5 phút → thêm 2 instance
- Khi CPU < 30% trong 10 phút → bớt 1 instance
- Giới hạn: tối thiểu 2, tối đa 20 instance

# Serverless: AWS Lambda
→ Tự động chạy 1 hay 10.000 request cùng lúc mà không cần cấu hình
```
  - Thiết kế app phải "stateless" để dễ scale ngang
  - Dùng load balancer + health check để phân phối traffic
  - Test khả năng chịu tải(load testing) trước khi ra production

**Measured service(Dịch vụ được đo lường)**: Hệ thống cloud tự động theo dõi, đo lường và báo cáo mức độ sử dụng tài nguyên, làm cơ sở tính phí minh bạch theo mô hình pay-per-use(ví dụ: storage, processing, bandwidth, và active user accounts).
```bash
# AWS Cost Explorer cho thấy:
- EC2: $45.20 (t2.micro × 720 giờ)
- S3: $12.50 (500GB storage + 1TB data transfer out)
- Lambda: $3.10 (2M requests + 400GB-seconds compute)
- RDS: $68.00 (db.t3.medium × 720 giờ + backup storage)

# CloudWatch Metrics:
- CPUUtilization: 45% average
- NetworkIn/Out: 2.3 GB/day
- RequestCount: 1.2M/day
```
  - Tối ưu chi phí: tắt resource không dùng, chọn instance phù hợp
  - Phát hiện bất thường: traffic đột biến -> có thể là attack hoặc bug
  - Budget alert: đặt ngưỡng cảnh báo trước khi hóa đơn vượt trần
  - FinOps: kết hợp kỹ thuật + tài chính để quản lý cloud hiệu quả

## 4. 4 Mô hình triển khai Cloud Computing
### 4.1 Public Cloud
- Hạ tầng cloud được sở hữu và vận hành bởi nhà cung cấp thứ 3, được cung cấp để sử dụng bởi cộng đồng (multi-tenant), truy cập qua Internet
- Ví dụ nhà cung cấp: AWS, Microsoft Azure, Google Cloud Platform(GCP), Alibaba Cloud,...
- Ưu điểm: 
  - Chi phí ban đầu thấp
  - Triển khai nhanh
  - Scale dễ dàng
  - Global reach
  - Nhà cung cấp tự lo bảo trì
- Nhược điểm:
  - Kiểm soát hạn chế
  - Chi phí dài hạn
  - Compliance phức tạp
  - Phụ thuộc vào API/tooling của một nhà cung cấp
- Phù hợp:
  - Ứng dụng web/mobile có traffic biến động
  - Workload không chứa dữ liệu nhạy cảm cực cao
  - Dự án test/PoC, development environment

### 4.2 Private Cloud
- Hạ tầng cloud được cung cấp để sử dụng độc quyền bởi một tổ chức nhiều người dùng (ví dụ: các đơn vị kinh doanh), có thể đặt on-premises(trong data center riêng) hoặc được host bởi bên thứ 3 nhưng cách ly hoàn toàn.
- Ví dụ triển khai:
  - On-premises: VMware vSphere, OpenStack, Nutanix trong data center công ty.
  - Hosted private: AWS Outposts, Azure Stack, Google Anthos tại colocation facility.
- Ưu điểm:
  - Bảo mật và kiểm soát tối đa
  - Đáp ứng GDPR, PCI-DSS, HIPAA... khi data không rời premises
  - Tối ưu hiệu năng
  - Tích hợp legacy
- Nhược điểm:
  - Chi phí đầu tư lớn
  - Triển khai chậm
  - Khó scale nhanh
  - Đòi hỏi expertise
- Phù hợp:
  - Hệ thống lõi (core systems) xử lý dữ liệu nhạy cảm
  - Workload ổn định, dự đoán được, chạy 24/7

### 4.3 Hybrid Cloud
- Kết hợp Public Cloud + Private Cloud với khả năng orchestration(Điều phối: là quy trình tự động hóa, quản lý và kết hợp nhiều hệ thống, ứng dụng, tác vụ máy tính hoặc tác nhân AI lại với nhau để thực hiện một quy trình công việc phức tạp, liền mạch) và di chuyển workload giữa hai môi trường, tạo thành một hệ thống thống nhất.
- Thành phần :
```bash
┌─────────────────┐     ┌─────────────────┐
│   Private Cloud │◄───►│   Public Cloud  │
│   (On-premises) │ VPN │   (AWS/Azure)   │
└────────┬────────┘     └────────┬────────┘
         │                       │
         ▼                       ▼
   ┌─────────────────────────────────┐
   │   Unified Management Layer      │
   │   • Identity (IAM sync)         │
   │   • Networking (VPC peering)    │
   │   • Orchestration (K8s federation)│
   │   • Monitoring (centralized)    │
   └─────────────────────────────────┘
```
Ví dụ:
```yaml
🛒 E-commerce platform:
├─ Private Cloud (on-prem):
│  ├─ Database chính chứa thông tin khách hàng, đơn hàng
│  ├─ Hệ thống thanh toán lõi (PCI-compliant)
│  └─ API gateway nội bộ
│
├─ Public Cloud (AWS):
│  ├─ Web frontend + CDN cho traffic toàn cầu
│  ├─ Search service dùng Elasticsearch managed
│  └─ Auto-scaling group xử lý traffic spike
│
└─ Kết nối:
   ├─ Site-to-site VPN hoặc Direct Connect
   ├─ Data replication từ private DB → public read replica
   └─ Khi Black Friday: traffic web → public cloud "burst", 
      transaction quan trọng → vẫn xử lý ở private
```
- Ưu điểm: 
  - Linh hoạt tối ưu
  - Cân bằng chi phí
  - Di chuyển linh hoạt
  - Vừa đáp ứng quy định, vừa tận dụng dịch vụ AI/ML mới của public cloud
- Nhược điểm:
  - Phức tạp kiến trúc
  - Integration cost
  - Monitoring khó
### 4.4 Community Cloud
- Sử dụng hai hay nhiều nhà cung cấp Public Cloud cùng lúc (ví dụ: AWS + Azure + GCP), có thể kết hợp với private cloud, để tránh phụ thuộc vào một vendor duy nhất.
- Ví dụ:
```yaml
 Global SaaS platform:
├─ AWS (us-east-1): 
│  ├─ Chạy primary workload cho khách hàng Bắc Mỹ
│  └─ Dùng SageMaker cho ML model training
│
├─ Azure (West Europe):
│  ├─ Phục vụ khách hàng EU để đáp ứng GDPR data residency
│  └─ Tích hợp với Microsoft 365 của enterprise customers
│
├─ GCP (asia-southeast1):
│  ├─ Dùng BigQuery cho analytics toàn cầu
│  └─ Tận dụng network backbone của Google cho latency thấp ở châu Á
│
└─ Quản lý thống nhất:
   ├─ Terraform để provision infra trên cả 3 cloud (multi-provider)
   ├─ Kubernetes federation để deploy app đồng nhất
   ├─ Service mesh (Istio) để quản lý traffic giữa các cloud
   └─ Centralized logging/monitoring với ELK hoặc Datadog
```
- Ưu điểm: 
  - Không bị "giam" bởi API/pricing của một nhà cung cấp
  - Tận dụng điểm mạnh từng vendor: AWS cho compute, GCP cho AI/ML, Azure cho enterprise integration
  - Nếu một cloud region/provider down, có thể failover sang provider khác
  - Có thể so sánh và chọn pricing tốt nhất cho từng workload
- Nhược điểm: 
  - Độ phức tạp cao
  - Chi phí quản lý tăng
  - Chuyển data giữa các cloud provider thường tốn phí và latency cao
  - Policy, compliance, auditing phải đồng bộ trên nhiều môi trường -> Secure fragmentation
