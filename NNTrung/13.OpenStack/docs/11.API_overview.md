# Tổng quan về API và API trong OpenStack
## 1. Khái niệm
API (Application Programming Interface) là một tập hợp các quy tắc, giao diện và định nghĩa cho phép các thành phần phần mềm hoặc các hệ thống khác biệt giao tiếp với nhau.

API đóng vai trò là một tầng trừu tượng hóa (Abstraction Layer), cung cấp khả năng truy cập tài nguyên hoặc dịch vụ mà không yêu cầu bên sử dụng phải can thiệp vào logic xử lý nội bộ hoặc mã nguồn của hệ thống cung cấp.

**Note**: Khi bạn dùng API, bạn chỉ cần biết "Cần đưa cái gì vào" và "Sẽ nhận lại cái gì", bạn không cần quan tâm bên trong nó xử lý thế nào.

2. **Các thành phần vận hành hệ thống**
- **Tính đóng gói (Encapsulation)**: Tính đóng gói (Encapsulation): API ẩn các chi tiết triển khai nội bộ và chỉ cung cấp các giao diện cần thiết cho bên ngoài.
- **Giao thức (Protocols)**: API sử dụng các giao thức như HTTP/HTTPS hoặc WebSocket, hoặc các framework như gRPC để truyền dữ liệu.
- **Điểm vào (Entry Point)**: Mọi API đều có nơi tiếp nhận yêu cầu. Trong Web là URL (Endpoint), trong lập trình hệ thống là tên hàm (Function Name) hoặc số hiệu ngắt (Interrupt vector).
- **Yêu cầu (Request) và Phản hồi (Response)**: Quy trình trao đổi dữ liệu hai chiều, trong đó bên yêu cầu gửi các tham số cần thiết và bên cung cấp trả về kết quả dưới dạng cấu trúc chuẩn (thường là JSON hoặc XML).
- **Định dạng chuẩn (Standardized Format)**: Dữ liệu truyền đi và trả về phải tuân thủ cấu trúc nhất định (như JSON trong Web, hoặc các kiểu dữ liệu `int`, `string`, `struct` trong ngôn ngữ lập trình).
- **Tính ổn định (Backward Compatibility)**: Một khi API đã được công bố, việc thay đổi nó sẽ làm hỏng mọi hệ thống đang kết nối. Do đó, API thường có các phiên bản (v1, v2).


3. Phân loại theo mục đích sử dụng

| Loại API| Môi trường| Ví dụ thực tế|
|----------|--------------------|-------------------|
| API Hệ điều hành| Giao tiếp giữa ứng dụng và Nhân (Kernel).| Ứng dụng yêu cầu Linux/Windows cấp quyền mở Camera hoặc đọc file từ ổ cứng.|
| API Thư viện| Giữa các module mã nguồn trong cùng một máy.| Gọi hàm `move_group.plan()` trong ROS 2 để lập trình cánh tay robot mà không cần tự viết thuật toán động học ngược.|
| API Phần cứng| Giữa phần mềm và thiết bị vật lý.| Lệnh điều khiển xung PWM để điều tốc động cơ thông qua Driver của vi điều khiển.|
| Web API| Giữa hai máy tính qua mạng Internet.| Ứng dụng điện thoại lấy dữ liệu thời tiết từ máy chủ của trung tâm khí tượng.|

4. Quy trình vận hành chung (Generic Workflow)
- Dù là API cấp thấp hay cấp cao, quy trình luôn là:
  - `Request (Yêu cầu)`: Bên sử dụng gửi một thông điệp kèm tham số (ví dụ: get_status(robot_id=1)).
  - `Processing (Xử lý)`: Hệ thống cung cấp tiếp nhận, kiểm tra quyền truy cập và thực hiện tính toán ngầm.
  - `Response (Phản hồi)`: Hệ thống trả về kết quả hoặc thông báo lỗi (ví dụ: status: "ready").

## 2. REST API
REST API (Representational State Transfer) là một phong cách kiến trúc (architectural style) dành cho việc thiết kế các hệ thống phân tán, chủ yếu là Web API.

Nó không phải là một giao thức (protocol) như HTTP, mà là một tập hợp các ràng buộc để đảm bảo hệ thống có khả năng mở rộng, linh hoạt và hiệu suất cao.

Trong REST, mọi thứ đều được coi là Tài nguyên (Resource). Mỗi tài nguyên được định danh duy nhất bằng một URI (Uniform Resource Identifier).

- Ví dụ về URI: `/users`, `/orders/123`, `/products/laptop-dell`.
- Thay vì gọi một hàm (function), bạn tương tác với các tài nguyên thông qua các danh từ.

Để một API được gọi là RESTful, nó phải tuân thủ các quy tắc sau:
- **A. Client-Server Architecture (Kiến trúc Khách - Chủ)**
  - Tách biệt hoàn toàn giữa giao diện người dùng (Client) và nơi lưu trữ dữ liệu (Server). Điều này cho phép bạn nâng cấp Server mà không cần thay đổi code ở Client và ngược lại.
- **B. Stateless (Không lưu trạng thái)**
  - Đây là đặc điểm kỹ thuật quan trọng nhất. Mỗi Request từ Client gửi lên Server phải chứa đầy đủ thông tin cần thiết để Server hiểu và xử lý.
  - Server không lưu bất kỳ thông tin nào về "phiên làm việc" (session) trước đó của Client. Điều này giúp hệ thống cực kỳ dễ mở rộng (Scale-out).
- **C. Cacheable (Có thể lưu bộ nhớ đệm)**
  - Phản hồi từ Server phải tự xác định xem dữ liệu đó có được phép lưu lại ở phía Client hay không. Điều này giúp giảm tải cho Server và tăng tốc độ phản hồi cho người dùng.
- **D. Layered System (Hệ thống phân tầng)**
  - Client không cần biết mình đang kết nối trực tiếp với Server thật hay đang qua một trung gian (Load Balancer, Proxy, hoặc Gateway). Tính chất này giúp bảo mật và cân bằng tải tốt hơn.
- **E. Uniform Interface (Giao diện thống nhất)**
  - Đây là "bản khế ước" để Client và Server hiểu nhau, bao gồm việc sử dụng các phương thức HTTP chuẩn và định dạng dữ liệu chuẩn (thường là JSON).

### 2.2 Các phương thức HTTP (HTTP Methods) trong REST

| Phương thức| Hành động| 
|---------------|-----------------|
| GET| Đọc/Lấy tài nguyên.|
| POST| Tạo mới một tài nguyên.| 
| PUT| Ghi đè/Cập nhật toàn bộ tài nguyên.|
| PATCH| Cập nhật một phần tài nguyên.|
| DELETE| Xóa tài nguyên.|

### 2.3 Cấu trúc của một REST Request/ Response
Một Request chuẩn bao gồm:
- Endpoint (URL): Ví dụ `https://api.example.com/v1/users/10`.
- Method: `GET`, `POST`, `PUT`, `DELETE`..
- Headers: Chứa thông tin về kiểu dữ liệu (`Content-Type: application/json`) hoặc xác thực (`Authorization: Bearer <token>`).
- Body: Chứa dữ liệu (thường dùng cho POST, PUT, PATCH).

Một Response chuẩn bao gồm:

- HTTP Status Code: * `2xx`: Thành công (200 OK, 201 Created).
- `4xx`: Lỗi phía Client (400 Bad Request, 401 Unauthorized, 404 Not Found).
- `5xx`: Lỗi phía Server (500 Internal Server Error).
- `Body (Payload)`: Dữ liệu trả về, thường là định dạng JSON.

REST API không phải là lựa chọn duy nhất. Với các hệ thống cần tốc độ cực cao và độ trễ thấp (như truyền thông tin giữa các vi điều khiển hoặc Robot), người ta có thể chọn gRPC. Với các truy vấn dữ liệu phức tạp, người ta chọn GraphQL. Tuy nhiên, REST vẫn là "ngôn ngữ chung" của thế giới Web hiện nay.

## 3. Các API trong OpenStack
Để thao tác với API Openstack, ta có các thao tác cơ bản:
- `culr`:
  - Công cụ dòng lệnh cho phép tương tác với các giao thức FTP, FTPS, HTTP, HTTPS, IMAP, SMTP, v.v.
  - Không có giao diện, chạy trên dòng lệnh, nhanh gọn.
- `REST client` là các công cụ như **Postman/Insomnia** giúp test API thông qua giao diện đồ họa.
- `OpenStack Python SDK`: Vai trò: wrapper cho REST API,
dùng cho automation, scripting.

### 3.1 Đặc điểm kỹ thuật của OpenStack API
- **Kiến trúc**: Tuân thủ chặt chẽ nguyên tắc **RESTful**.
- **Định dạng dữ liệu**: Sử dụng định dạng **JSON** cho cả Request và Response.
- **Xác thực (Authentication)**: Hoàn toàn **Stateless**. Mọi API request đều phải đính kèm một xác thực Token (được cấp phát bởi dịch vụ Identity) trong HTTP Header (`X-Auth-Token`).
- **Tính phiên bản (Versioning)**: Các API đều được đánh version rõ ràng trên URL (ví dụ: `/v2.1/servers`, `/v3/auth/tokens`) để đảm bảo tính tương thích ngược (Backward Compatibility).

### 3.2 Các API cốt lõi trong hệ sinh thái OpenStack
Mỗi service có API riêng:
- Nova → compute
- Neutron → network
- Cinder → block storage

#### A. Keystone API (Identity Service)
Đây là "người gác cổng" của toàn bộ hệ thống. Bạn không thể gọi bất kỳ API nào khác nếu chưa đi qua Keystone. Keystone Service cung cấp các API endpoint để xác thực và ủy quyền. Khi nhận được yêu cầu qua API, Keystone sẽ truy vấn cơ sở dữ liệu (Backend) để xác minh danh tính và khởi tạo một Token (thường dùng Fernet tokens) dựa trên các thông tin đó.
- **Chức năng**: Cấp phát Token, quản lý người dùng (Users), dự án (Projects/Tenants), và phân quyền (Roles).
- **Endpoint tiêu biểu**: `POST /v3/auth/tokens` (Gửi username/password để đổi lấy Token).

#### B. Nova API (Compute Service)
Trái tim của OpenStack, chịu trách nhiệm quản lý vòng đời của máy ảo (Instance). Nova Service cung cấp các API endpoint để người dùng gửi yêu cầu điều phối máy ảo. Khi nhận yêu cầu, hệ thống sẽ thông qua lớp API để xác thực, sau đó lớp Logic (Scheduler/Conductor) sẽ truy vấn Database để lập kế hoạch và ra lệnh cho các node xử lý phía sau (Compute Node) thực hiện việc ảo hóa phần cứng.
- **Chức năng**: Cấp phát tài nguyên tính toán. Khi nhận được request qua API, Nova không tự mình tạo máy ảo mà sẽ điều phối lệnh xuống các compute node để giao tiếp với các hypervisor bên dưới (phổ biến nhất là thao tác với KVM thông qua Libvirt).
  - Trong OpenStack, các máy ảo (Compute nodes) thường không được phép truy cập trực tiếp vào Database. Chúng phải "nhờ" Nova-Conductor làm hộ qua API nội bộ.
- **Endpoint tiêu biểu**: `POST /v2.1/servers` (Tạo máy ảo mới).
  - `POST /v2.1/servers/{server_id}/action` (Thực hiện hành động như Start, Stop, Reboot).

#### C. Neutron API (Networking Service)
Quản lý toàn bộ hạ tầng mạng ảo hóa (SDN - Software Defined Networking).
- **Chức năng**: Tạo mạng ảo (Virtual Networks), Subnets, Router. Neutron API cũng là nơi tiếp nhận các cấu hình mạng nâng cao, bao gồm việc thiết lập các mô hình High Availability, cấp phát Floating IP, hoặc cấu hình Load Balancer (LBaaS) để phân tải traffic vào các node bên trong.
- **Endpoint tiêu biểu**: `POST /v2.0/networks` (Tạo mạng L2).

#### D. Glance API (Image Service)
Quản lý các bản sao hệ điều hành (Image).
- **Chức năng**: Lưu trữ, truy vấn và cung cấp các file Image (như file `.qcow2` của Ubuntu, Rocky Linux...) để Nova sử dụng làm template khởi tạo máy ảo.

- **Endpoint tiêu biểu**: `GET /v2/images` (Lấy danh sách các Image có sẵn).

#### Cinder API (Block Storage Service)
Quản lý các ổ cứng ảo (Volume).
- **Chức năng**: Tạo ra các Volume (giống như USB hoặc ổ cứng rời) để gắn (attach) vào các máy ảo của Nova.
- **Endpoint tiêu biểu**: `POST /v3/{project_id}/volumes` (Tạo một ổ cứng mới).
